# PRD v2.2 - 설명 가능한 회의 결정 지원

> 보존 상태: 이 문서는 2026-07-11의 전체 초안이다. 현재 구현 기준은 `prd-v2.2.1.md`이며, 네이밍·마감 이후 응답·새 시간 범위·요청 대상 선택·P0 입력 실험의 충돌을 v2.2.1에서 교정했다.

- 제품명: 확정보드
- 상태: 구현 기준안
- 기준일: 2026-07-11
- 대상: 토스 프로덕트 디자인 채용 과제용 반응형 프로토타입
- 구현 조건: 백엔드 없이 핵심 흐름을 검증한다.
- 판단: ALIGN, SPECIFY, SLICE 후 GO

## 0. 문서 목적

v2.2는 제품 방향을 다시 바꾸는 문서가 아니다. v2.1이 정한 다음 문제를 사용자 입력, 판정 규칙, 결과 문장, 검증 범위까지 같은 의미로 연결한다.

> 지금 회의를 정할 수 있는가?
> 아직 정할 수 없다면 어떤 응답 조건이 더 필요한가?

v2.1에는 다음 불일치가 남아 있었다.

- `일정 조정하면 가능`이 확약인지 미결 응답인지 불분명했다.
- 대체 가능한 일반 미응답자 중 한 명을 제품이 임의로 지목할 수 있었다.
- 작성 중 빈칸과 명시적인 `어려움`을 제출 과정에서 구분하지 못했다.
- 30분 단위 응답을 1시간 후보로 합치는 계약이 부족했다.
- 필수 참석자와 최소 참석 인원 조건이 중복 계산될 수 있었다.
- 파생 후보가 많아질 때 무엇을 묶고 어떻게 비교할지 정하지 못했다.
- 기능 범위와 검증 목표가 넓어 핵심 가치가 흐려졌다.

v2.2의 목적은 기능을 늘리는 것이 아니라 다음 한 줄을 끝까지 증명하는 것이다.

> 사람별 응답을 모아 보여주는 것보다, 지금 정할 수 있는지와 어떤 긍정 응답 조건이 더 필요한지를 설명하는 결과가 주최자의 안전한 결정을 돕는가?

## 1. 제품 정의

### 1.1 제품 한 문장

> 확정보드는 필수 참석자와 진행 기준이 있는 업무 회의에서 지금 정할 수 있는 시간을 보여주고, 아직 정할 수 없다면 반드시 필요한 응답자와 대체 가능한 미응답자 집합, 필요한 가능 응답 수를 설명하는 회의 결정 지원 도구다.

### 1.2 사용자에게 전달할 약속

> 지금 정할 수 있는 시간과, 더 필요한 답만 알려드려요.

### 1.3 프로토타입 대표 상황

> 같은 회사 동료 6명이 다음 주 안에 1시간 업무 회의를 잡는다.

6명과 1시간은 과제 검증을 위한 대표 데이터다. 제품의 인원수를 6명으로 고정하는 요구사항은 아니다.

### 1.4 집중하는 회의

- 반드시 참석해야 하는 사람이 있다.
- 일부 참석자가 빠져도 진행할 수 있다.
- 단순한 빈 시간보다 참석자 구성과 진행 기준이 중요하다.
- 주최자가 최종 결정을 내려야 한다.
- 정기 회의가 아닌 비정기 업무 회의다.

### 1.5 우선 다루지 않는 회의

- 1:1 예약
- 대규모 행사
- 반복 회의
- 회의실 예약이 중심인 상황
- 조직 캘린더만으로 자동 배정할 수 있는 상황
- 주최자가 참석하지 않는 대리 조율

## 2. 문제 정의

업무 회의에서 가장 많은 사람이 가능한 시간과 실제로 확정할 수 있는 시간은 다를 수 있다.

- 다섯 명이 가능해도 필수 참석자 한 명이 어렵다면 회의를 정할 수 없다.
- 일부가 응답하지 않았더라도 필수 참석자와 진행 인원을 충족했다면 더 기다릴 필요가 없다.
- 현재 일정 그대로 가능한 상태와 다른 일정을 옮겨 참석하겠다는 상태는 성립 가능성은 같아도 부담이 다르다.
- 특정 필수 참석자의 응답은 반드시 필요할 수 있다.
- 일반 미응답자는 여러 명 중 일부의 가능 응답만으로 충분할 수 있다.

시간 정보가 모인 뒤에도 주최자는 다음을 직접 해석한다.

1. 지금 회의를 정해도 되는가?
2. 누구의 응답이 반드시 필요한가?
3. 대체 가능한 사람들 중 몇 명의 가능 응답이 필요한가?
4. 어떤 후보는 더 기다릴 필요가 없는가?
5. 어떤 후보는 현재 조건상 성립할 수 없는가?

핵심 문제는 시간 수집이 아니다.

> 성격이 다른 응답을 회의 성립 여부와 남은 결정 조건으로 바꾸는 해석 비용이 핵심 문제다.

이 문제의 발생 빈도, 전환 의향, 시장 규모는 아직 검증되지 않았다. v2.2는 시장 우위를 주장하지 않고 경험 가설을 검증한다.

## 3. 중심 사용자와 핵심 순간

### 3.1 중심 사용자

중심 사용자는 회의 주최자다. 참석자의 응답 부담도 중요하지만 여러 응답을 종합해 확정, 대기, 추가 요청을 결정하는 책임은 주최자에게 집중된다.

### 3.2 핵심 순간

응답을 받은 주최자가 결과 화면을 처음 열고 다음 질문에 답하는 순간이다.

> 지금 정할 수 있는가?
> 아니라면 어떤 가능 응답 조건이 더 필요한가?

회의 생성과 참석자 응답은 이 판단을 신뢰할 수 있게 만드는 입력 과정이다. 프로토타입의 대표 품질 기준은 결과 화면에 둔다.

## 4. 제품 목표

### 목표 G1. 설정한 기준과 다른 확정 판정을 막는다

주최자가 설정한 필수 참석자와 진행 기준이 충족되지 않은 후보를 `지금 정할 수 있음`으로 보여주지 않는다. 제품은 주최자가 입력한 기준 자체가 실제 회의에 적절한지까지 보증하지 않는다.

달성 방법:

- 필수 참석자 집합과 최소 참석 인원을 명시적으로 정규화한다.
- 후보별 참석 확약, 미응답, 어려움 상태를 같은 규칙으로 계산한다.
- 조건이 충족되지 않은 후보는 `응답 필요` 또는 `다른 시간 필요`로 구분한다.
- 최종 결정은 주최자에게 남기고 상세 근거를 제공한다.

성공 기준:

- 자동화된 도메인 계약 테스트에서 기준과 다른 확정 판정이 발생하지 않는다.
- 사용성 테스트 참가자 5명 중 최소 4명이 확정 가능 후보를 정확히 구분한다.

### 목표 G2. 남은 결정 조건을 정확하게 설명한다

제품이 근거 없이 한 사람을 추천하지 않고, 반드시 필요한 특정인과 대체 가능한 대상군을 구분한다.

달성 방법:

- 후보마다 `requiredPending`, `optionalPendingPool`, `positiveResponsesNeededAfterRequiredYes`를 계산한다.
- `응답 한 개가 필요하다`가 아니라 `누가 반드시 가능해야 하고 어느 집합에서 몇 명이 가능해야 하는지`를 조건부로 설명한다.
- 연락 순서와 연락할 사람 수는 제품이 임의로 정하지 않는다.

성공 기준:

- 참가자 5명 중 최소 4명이 `특정인의 응답이 반드시 필요함`과 `대상군 중 일부의 가능 응답이 필요함`을 구분한다.
- 후보 상태에 영향을 줄 수 없는 사람을 다음 요청 대상으로 고르지 않는다.

### 목표 G3. 참석자의 입력 의미를 판정까지 보존한다

작성 중 빈칸, 제출 완료 후 어려움, 현재 가능, 일정 조정 확약을 서로 다른 의미로 유지한다.

달성 방법:

- 사람 단위 작성 상태와 시간 칸 단위 가용 상태를 분리한다.
- 제출 전 `UNSET`을 판정에 사용하지 않는다.
- 제출 확인 뒤에만 남은 `UNSET`을 `UNAVAILABLE`로 저장한다.
- `정해지면 일정을 조정해 참석할게요`를 추가 확인이 필요 없는 조건부 확약으로 정의한다.
- 여러 시간 칸을 후보 하나로 합치는 보수적 집계 규칙을 사용한다.

성공 기준:

- 참가자 5명 중 최소 4명이 미선택 시간과 조정 확약의 의미를 자신의 말로 설명한다.
- 입력한 시간 칸과 파생된 1시간 후보 상태가 기대한 규칙과 일치한다.

### 목표 G4. 핵심 흐름 하나를 완결한다

백엔드 없는 프로토타입에서 생성, 응답, 판정, 응답 추가 후 재판정까지 한 줄기로 작동하게 한다.

달성 방법:

- 과제 대표 시나리오는 1시간 회의와 30분 단위로 제한한다.
- 결과 판정에 직접 필요하지 않은 기능을 P0에서 제외한다.
- 이어서 조율은 한 사람의 응답이 추가되고 같은 화면이 다시 판정되는 장면까지만 완성한다.
- PC와 모바일에서 핵심 화면과 조작을 동일한 정보 모델로 제공한다.

성공 기준:

- 새로고침 없는 한 세션에서 생성부터 재판정까지 시연할 수 있다.
- 모바일과 PC에서 핵심 CTA와 시간 입력이 가려지거나 겹치지 않는다.

## 5. 비목표

v2.2는 다음을 목표로 하지 않는다.

- 가장 좋은 시간을 자동으로 결정한다.
- 조직 안에서 누구에게 먼저 연락할지 판단한다.
- 참석자 관계, 직급, 연락 비용을 계산한다.
- 사용자가 다른 일정 조율 도구에서 전환할 것임을 증명한다.
- 실제 메시지를 발송하거나 캘린더에 일정을 생성한다.
- 모든 업무 회의 유형을 지원한다.

## 6. 제품 가설

### H1. 결정 상태 해석

필수 참석자와 진행 기준을 반영한 결과 요약은 사람별 응답표만 보여주는 방식보다 주최자의 안전한 결정을 빠르게 도울 것이다.

### H2. 미응답 영향 구분

필수 응답자와 대체 가능한 미응답자 집합을 구분하면 불필요한 전원 대기와 관련 없는 사람에 대한 요청을 줄일 수 있을 것이다.

### H3. 조건부 확약

`정해지면 일정을 조정해 참석할게요`라는 문구를 참석자와 주최자가 모두 추가 확인이 필요 없는 확약으로 이해할 것이다.

### H4. 연속 시간 입력

정확한 후보에 반복 투표하는 대신 연속된 가능 시간을 입력하면 주최자가 미리 고르지 않은 유효 후보도 발견할 수 있을 것이다.

### H5. 설명 가능한 판정

결론과 이유를 함께 보여주면 주최자는 빠르게 판단하면서도 필요할 때 상세 응답으로 판정을 검토할 수 있을 것이다.

## 7. 제품 원칙

1. **사용자 의도를 기능보다 우선한다.** 후속 행동을 만들기 위해 확약을 미결 응답으로 바꾸지 않는다.
2. **결정에 영향을 주는 정보만 받는다.** 회의 성립이나 후보 비교에 쓰이지 않는 정보는 받지 않는다.
3. **미입력을 부정 응답으로 몰래 바꾸지 않는다.** 변환은 제출 확인 뒤에만 일어난다.
4. **제품은 사람을 임의로 고르지 않는다.** 특정인과 대체 가능한 대상군을 구분한다.
5. **조건을 설명하되 연락 전략은 정하지 않는다.** 순차 연락과 동시 연락은 주최자가 선택한다.
6. **결론을 먼저, 근거는 검토 가능하게 보여준다.** 첫 화면과 상세의 역할을 나눈다.
7. **제품은 자동 확정하지 않는다.** 최종 결정은 주최자가 내린다.
8. **재조율은 같은 맥락에서 이어간다.** 기존 응답을 유지하고 새 정보만 다시 계산한다.

## 8. 핵심 사용자 흐름

### 8.1 회의 안내 계약

회의 생성의 첫 단계는 긴 설명을 작성하는 화면이 아니라, 참석자가 응답의 업무 맥락을 판단할 최소 정보를 만드는 과정이다.

주최자가 입력하거나 확인할 정보:

- 회의 이름: 필수
- 주최자 또는 팀: 프로토타입 기본값을 제공하고 수정 가능
- 한 문장 목적: `이 회의에서 무엇을 확인하거나 결정하나요?`
- 관련 자료: 선택, 문서명이나 링크만 제공

참석자가 응답 전에 볼 정보:

- 회의 이름
- 주최자 또는 초대 맥락
- 한 문장 목적
- 다음 주 조율 범위와 1시간 확보 조건
- 응답 마감
- 제출 뒤에도 마감 전 수정할 수 있다는 안내
- 응답을 모은 뒤 주최자가 최종 시간을 정한다는 사실

참석자에게 보이지 않는 정보:

- 필수 참석자 여부
- 최소 참석 인원
- 다른 참석자의 응답
- 후보 판정과 내부 계산 규칙

관련 자료는 응답 전 과제처럼 보이지 않게 낮은 위계로 둔다. 참석자가 링크를 열고 5초 안에 어떤 회의이고 누가 보낸 요청인지 이해할 수 있어야 한다.

### A. 주최자 생성

```text
회의 안내
→ 참석자 선택
→ 회의 진행 기준
→ 다음 주의 조율 가능 시간대
→ 응답 마감
→ 검토와 링크 생성
```

과제 시나리오에 맞춰 새 회의의 확보 시간은 1시간을 기본값으로 제공한다. 다만 이는 제품 제약이 아니며, 주최자는 30분·1시간·1시간 30분·2시간 중 선택하거나 30분 단위로 직접 입력할 수 있다. 참석자는 주최자가 정한 확보 시간을 회의 맥락으로 확인한다.

### B. 참석자 응답

```text
회의 맥락 확인
→ 현재 일정 그대로 가능한 시간 선택
→ 정해지면 일정을 조정해 참석할 수 있는 시간 선택
→ 가능하면 피하고 싶은 시간 선택(선택 사항)
→ 미선택 시간의 의미 확인
→ 제출
```

### C. 주최자 결정

```text
전체 결론
→ 상태별 후보군
→ 후보의 이유와 남은 조건
→ 확정 또는 응답 요청 대상 선택
→ 필요할 때 상세 근거
```

### D. 얇은 이어서 조율

```text
한 사람의 응답 추가 또는 수정
→ 기존 시간 응답 유지
→ 모든 파생 후보 재계산
→ 같은 결과 화면 갱신
```

## 9. 회의 성립 기준 계약

### 9.1 주최자 규칙

- 주최자는 자동으로 참석자에 포함한다.
- 주최자는 자동으로 필수 참석자다.
- 주최자는 최소 참석 인원에 포함한다.
- 후보는 주최자가 열어둔 시간 안에서만 생성한다.
- 따라서 주최자는 모든 파생 후보에서 `AVAILABLE`로 처리한다.

### 9.2 사용자 입력 흐름

#### 질문 1

> 이 회의는 모두 참석해야 하나요?

- 모두 참석해야 해요
- 일부가 빠져도 진행할 수 있어요

#### 일부가 빠져도 진행할 수 있는 경우

> 꼭 참석해야 하는 사람을 골라주세요.

주최자는 이미 필수 참석자에 포함돼 있다.

다음 질문을 이어서 묻는다.

> 필수 참석자만 오면 진행할 수 있나요?

- 네, 필수 참석자만 오면 돼요
- 아니요, 전체 참석 인원도 중요해요

두 번째를 선택한 경우에만 총 최소 참석 인원을 받는다.

### 9.3 내부 정규화

모든 회의는 판정 전에 필수 참석자 집합 `R`과 최소 참석 인원 `Q`를 갖는다.

```text
전원 참석:
R = 전체 참석자
Q = 전체 참석자 수

일부 불참 가능, 필수 참석자만 필요:
R = 주최자 + 선택한 필수 참석자
Q = |R|

일부 불참 가능, 전체 인원 기준도 필요:
R = 주최자 + 선택한 필수 참석자
Q = 주최자가 입력한 수
```

유효성 조건:

```text
|R| <= Q <= 전체 참석자 수
```

`아직 모르겠어요`는 P0에서 제공하지 않는다. 진행 기준이 없으면 제품도 확정 가능 여부를 판정할 수 없기 때문이다.

## 10. 참석자 응답 계약

### 10.1 가용 상태

#### 가능해요

> 현재 일정을 바꾸지 않고 참석할 수 있어요.

내부 값: `AVAILABLE`

#### 정해지면 일정을 조정해 참석할게요

> 이 시간으로 회의가 정해지면 다른 일정을 옮겨 참석하겠다는 의미예요.

내부 값: `ADJUSTMENT_COMMIT`

이 응답은 참석 확약으로 계산한다. 주최자가 다시 허락을 받을 필요는 없다. 확정 뒤에는 해당 참석자에게 조정이 필요한 시간으로 확정됐다는 사실을 전달한다.

#### 어려워요

> 이 시간에는 참석하기 어려워요.

내부 값: `UNAVAILABLE`

### 10.2 P0에서 제공하지 않는 상태

다음 의사는 표현하지 않는다.

> 일정을 조정할 수 있을지 아직 모르겠어요.

이는 확약도 불가도 아닌 별도의 미결 상태다. 필요성이 검증되기 전까지 P0에서 제외한다.

### 10.3 선호 정보

P0에서는 다음 하나만 선택적으로 제공한다.

> 가능하면 피하고 싶어요.

내부 값: `AVOID_PREFERRED`

선호는 후보 비교에만 사용하며 회의 성립을 막지 않는다. 점심 직후, 앞뒤 일정, 이동 시간, 자유 입력 사유는 P0에서 제외한다.

## 11. 작성 상태와 제출 계약

### 11.1 사람 단위 상태

```text
NOT_STARTED
DRAFT
SUBMITTED
```

- `NOT_STARTED`: 응답을 시작하지 않았다.
- `DRAFT`: 작성 중이지만 제출하지 않았다.
- `SUBMITTED`: 응답 의미를 확인하고 제출했다.

### 11.2 시간 칸 상태

```text
UNSET
AVAILABLE
ADJUSTMENT_COMMIT
UNAVAILABLE
```

`UNSET`은 작성 중이거나 새 시간 범위가 추가됐을 때만 존재한다.

### 11.3 판정에 사용하는 데이터

- `NOT_STARTED`와 `DRAFT`인 참석자는 모든 후보에서 `UNKNOWN`으로 처리한다.
- `SUBMITTED` 응답만 시간 칸별 판정에 사용한다.
- 부분 제출은 P0에서 허용하지 않는다.

### 11.4 제출 과정

제출 직전에 다음 의미를 명시적으로 확인한다.

> 선택하지 않은 시간은 `참석하기 어려워요`로 저장돼요. 조율 범위를 모두 확인하셨나요?

사용자가 확인한 뒤에만 기존 조율 범위의 `UNSET`을 `UNAVAILABLE`로 저장한다.

### 11.5 제출 후 수정과 범위 확장

- 제출한 응답을 다시 열면 모든 기존 칸은 명시적인 상태로 보인다.
- 참석자는 기존 상태를 수정하고 다시 제출할 수 있다.
- 수정 또는 추가 제출 뒤 모든 후보를 다시 계산한다.
- 공유 이후 주최자가 새 시간 범위를 여는 기능은 P0에서 지원하지 않는다.
- 후속 범위 확장에서는 응답 범위 버전과 `NEEDS_UPDATE` 상태가 필요하므로 별도 계약으로 다룬다.

### 11.6 응답 마감 계약

- 응답 마감은 운영 안내와 후속 행동을 위한 값이다.
- 마감이 지나도 미응답은 `UNKNOWN`으로 유지한다.
- 마감이 지났다는 이유만으로 후보를 `IMPOSSIBLE`로 바꾸지 않는다.
- 마감 이후에도 후보는 `PENDING`일 수 있다.
- 결과 화면은 `응답 마감이 지났어요`를 보조 상태로 표시하고, 주최자가 재요청 또는 다른 후보 검토를 선택하게 한다.

### 11.6 입력 UI 가설

P0은 다음 순차 범위 입력을 우선 구현한다.

1. 현재 일정 그대로 가능한 시간을 선택한다.
2. 그 밖에 조정해서 참석할 수 있는 시간을 선택한다.
3. 선택한 시간 중 가능하면 피하고 싶은 시간을 표시한다.

이는 확정된 정답이 아니다. 상태 도구를 전환해 한 그리드에 칠하는 방식과 완료 시간, 오류 수, 수정 횟수, 의미 이해도를 비교한다.

## 12. 파생 후보 계약

### 12.1 P0 시간 모델

- 회의 확보 시간: 1시간
- 시간 최소 단위: 30분
- 후보 시작 간격: 30분
- 날짜 범위: 다음 주
- 후보는 하루를 넘어가지 않는다.
- 기준 시간대: `Asia/Seoul`
- 타임존 선택과 변환 UI는 P0에서 제공하지 않는다.

### 12.2 후보 생성

주최자가 열어둔 연속 시간 범위 안에서 1시간 후보를 계산한다.

예를 들어 오전 10시부터 정오까지 열었다면 다음 후보가 생성된다.

```text
10:00-11:00
10:30-11:30
11:00-12:00
```

참석자는 후보에 직접 투표하지 않는다. 30분 단위의 연속 가용 상태를 입력하고 제품이 정확한 후보를 파생한다.

### 12.3 참석자별 후보 상태 집계

1시간 후보를 구성하는 두 칸을 다음 우선순위로 합친다.

```text
UNAVAILABLE
→ UNKNOWN
→ ADJUSTMENT_COMMIT
→ AVAILABLE
```

| 후보 안의 시간 칸 | 참석자의 후보 상태 |
| --- | --- |
| 한 칸이라도 `UNAVAILABLE` | `UNAVAILABLE` |
| 어려움은 없지만 한 칸이라도 `UNKNOWN` | `UNKNOWN` |
| 어려움과 미응답은 없고 한 칸이라도 `ADJUSTMENT_COMMIT` | `ADJUSTMENT_COMMIT` |
| 모든 칸이 `AVAILABLE` | `AVAILABLE` |

선호는 별도로 계산한다. 후보 구간에 `AVOID_PREFERRED` 칸이 하나라도 있으면 해당 참석자에게 선호 부담이 있는 후보로 기록한다.

## 13. 후보별 판정 계약

각 후보마다 다음 집합을 계산한다.

```text
R = 필수 참석자
C = 참석 확약자
U = 미응답자
N = 참석 어려움
A = 조정 확약자
P = 낮은 선호가 있는 참석자
Q = 정규화된 최소 참석 인원
```

`C`에는 `AVAILABLE`과 `ADJUSTMENT_COMMIT`이 포함된다.

### 13.1 IMPOSSIBLE

사용자 표현:

> 다른 시간을 찾아야 해요.

조건:

```text
R ∩ N이 비어 있지 않음
또는
|C| + |U| < Q
```

### 13.2 READY

사용자 표현:

> 지금 정할 수 있어요.

조건:

```text
R이 모두 C에 포함
그리고
|C| >= Q
```

### 13.3 PENDING

사용자 표현:

> 응답이 더 필요해요.

조건:

```text
READY도 아니고 IMPOSSIBLE도 아님
```

### 13.4 판정 순서

```text
1. 필수 참석자 중 어려운 사람이 있는가?
2. 모든 미응답자가 가능해도 최소 기준을 채울 수 없는가?
3. 지금 필수 참석자와 최소 기준을 충족했는가?
4. 아니라면 어떤 긍정 응답 조건이 남았는가?
```

---

## 14. PENDING 설명 계약

후보마다 다음 값을 계산한다.

```text
requiredPending = R ∩ U
optionalPendingPool = U - R

positiveResponsesNeededAfterRequiredYes =
max(0, Q - (|C| + |requiredPending|))
```

이 계산은 모든 필수 미응답자가 가능하다고 답했을 때 일반 미응답자 집합에서 추가로 필요한 가능 응답 수를 뜻한다. P0의 필수 참석자와 최소 인원 규칙 안에서 수행하는 제한된 조건부 계산이다.

### 14.1 특정인의 응답이 반드시 필요한 경우

> 필수 참석자인 민지 님의 가능 응답이 필요해요.

필수 참석자는 이름으로 지목할 수 있다.

### 14.2 대체 가능한 대상군이 있는 경우

> 민수, 지수, 현우 님 중 한 명 이상이 가능해야 이 시간을 정할 수 있어요.

제품은 대상군 중 한 명을 자동 추천하지 않는다.

### 14.3 두 조건이 함께 남은 경우

> 민지 님의 가능 응답이 반드시 필요해요.
> 일반 미응답자 세 명 중 한 명 이상도 가능해야 해요.

`먼저`, `그다음`처럼 연락 순서를 암시하지 않는다. 주최자는 필요에 따라 한 명 또는 여러 명에게 동시에 요청할 수 있다.

### 14.4 부정 응답이 들어온 경우

미응답자가 `UNAVAILABLE`로 바뀌면 모든 후보 값을 다시 계산한다.

제품은 다음처럼 말하지 않는다.

> 응답 하나만 더 받으면 결정할 수 있어요.

한 명의 부정 응답으로는 결론이 나지 않을 수 있기 때문이다. 필요한 것은 단순 응답 수가 아니라 조건을 충족하는 가능 응답이다.

## 15. 후보 묶음과 비교 계약

### 15.1 상태 묶음

결과 화면은 후보를 다음 순서로 나눈다.

1. 지금 정할 수 있는 시간
2. 응답이 더 필요한 시간
3. 다른 시간을 찾아야 하는 시간

상태는 색만으로 구분하지 않고 문구와 이유를 함께 제공한다.

### 15.2 중첩 후보 묶음

연속 시간 입력은 30분 간격의 중첩 후보를 많이 만들 수 있다. 인접 후보의 다음 정보가 모두 같을 때만 하나의 후보군으로 묶는다.

- 결정 상태
- 참석자별 후보 상태
- 필수 미응답자 집합
- 대체 가능한 미응답자 집합과 필요한 가능 응답 수
- 조정 확약자 집합
- 낮은 선호가 있는 참석자 집합

인원수만 같고 사람이 다르면 묶지 않는다.

예:

> 오전 10시 또는 10시 30분에 시작할 수 있어요.

각 실제 시작 시각은 후보군 안에서 선택할 수 있어야 한다.

전체 요약의 시간 개수는 후보군 수가 아니라 실제 시작 시각 수를 센다. 결과 첫 화면에는 상태별 상위 3개 후보군을 보여주고 나머지는 펼쳐본다. 3개 제한은 초기 UI 계약이며 사용성 검증으로 조정할 수 있다.

### 15.3 후보 정렬

숨겨진 종합 점수를 사용하지 않는다.

READY 후보군:

1. 조정 확약자가 적음
2. 낮은 선호가 적음
3. 일반 미응답자가 적음
4. 참석 확약자가 많음
5. 조건이 같으면 시간순으로 표시

PENDING 후보군:

1. 필수 미응답자가 적음
2. 필수 응답이 모두 긍정일 때 추가로 필요한 가능 응답 수가 적음
3. 조정 확약자가 적음
4. 낮은 선호가 적음
5. 조건이 같으면 시간순으로 표시

시간순은 표시 순서를 안정시키는 규칙이지 더 좋은 후보라는 주장으로 사용하지 않는다. 조건이 완전히 같은 후보군은 동등한 선택지로 보여준다.

## 16. 결과 화면 정보 구조

### 16.1 전체 요약

첫 시야에는 후보 전체의 현재 결론을 보여준다.

예:

> 지금 정할 수 있는 시간이 2개 있어요.

또는:

> 아직 정할 수 있는 시간이 없어요.
> 필수 참석자 1명의 응답이 필요해요.

### 16.2 후보 또는 후보군

```text
시간
→ 현재 결론
→ 가장 중요한 이유
→ 부담 또는 남은 조건
→ 가능한 행동
```

### 16.3 READY 예시

> **화요일 오후 3시**
> 지금 정할 수 있어요.
> 필수 참석자가 모두 참석할 수 있고 최소 4명을 충족해요.
> 1명은 다른 일정을 조정해 참석해요.
> `[이 시간으로 정하기]`

### 16.4 PENDING 예시

> **수요일 오전 11시**
> 민지 님의 가능 응답이 필요해요.
> 민지 님은 필수 참석자예요.
> `[응답 요청 대상 선택]`

또는:

> **목요일 오후 2시**
> 세 명 중 한 명 이상이 가능해야 정할 수 있어요.
> 민수, 지수, 현우 님은 이 조건에서 서로 대체 가능해요.
> `[응답 요청 대상 선택]`

CTA는 주최자가 한 명 또는 여러 명을 선택할 수 있게 한다. 제품은 연락 순서를 정하지 않는다.

### 16.5 IMPOSSIBLE 예시

> **금요일 오후 4시**
> 다른 시간을 찾아야 해요.
> 필수 참석자인 준호 님이 참석하기 어렵다고 답했어요.
> `[다른 시간 보기]`

### 16.6 상세 근거

후보를 펼치면 다음을 확인할 수 있다.

- 사람별 후보 상태
- 필수 참석자 여부
- 조정 확약 여부
- 미응답 여부
- 낮은 선호 여부
- 현재 확약 인원과 최소 참석 인원
- 판정에 적용한 규칙

요약에서는 결정하고 상세에서는 근거를 검토한다.

## 17. 확정과 재판정

### 17.1 확정

제품은 자동 확정하지 않는다. 주최자가 READY 후보의 실제 시작 시각을 선택한다.

확정 결과에는 다음만 전달한다.

- 회의 제목
- 날짜와 확보 시간
- 확정된 시작 및 종료 경계
- 필요한 경우 장소 또는 링크

조정 확약한 참석자에게는 다음 사실을 함께 전달한다.

> 조정해서 참석 가능하다고 표시한 시간으로 회의가 확정됐어요.

이는 재확인 요청이 아니라 이미 한 확약의 실행 시점을 알리는 안내다.

### 17.2 재판정

- 응답 추가 또는 수정 후 모든 후보를 다시 계산한다.
- 기존 응답은 후보 ID가 아니라 시간 칸 상태를 원본으로 유지한다.
- 새 시간 범위를 열면 기존 칸은 유지하고 새 칸만 `UNSET`으로 만든다.
- P0에서는 한 사람의 추가 응답으로 결과가 바뀌는 장면까지 완전히 작동하게 한다.

## 19. 필수 검증 시나리오

### S1. 다수결과 회의 성립이 다른 상황

가장 많은 사람이 가능하지만 필수 참석자가 어렵다.

기대 결과: `IMPOSSIBLE`

### S2. 미응답자가 있어도 확정 가능한 상황

일반 참석자가 미응답이지만 필수 참석자와 최소 인원을 이미 충족한다.

기대 결과: `READY`, 미응답자에게 요청하는 CTA 없음

### S3. 특정 필수 참석자의 답이 필요한 상황

필수 참석자 한 명이 미응답이다.

기대 결과: `PENDING`, 해당 사람을 반드시 필요한 응답자로 표시

### S4. 여러 사람 중 일부의 가능 응답이 필요한 상황

일반 미응답자 세 명 중 한 명 이상이 가능해야 최소 인원을 충족한다.

기대 결과: `PENDING`, 세 명을 대체 가능한 대상군으로 표시

### S5. 필수 응답과 일반 긍정 응답이 함께 필요한 상황

필수 미응답자의 가능 응답과 일반 미응답자 집합의 추가 가능 응답이 모두 필요하다.

기대 결과: 두 조건을 중복 없이 함께 표시하되 연락 순서를 지정하지 않음

### S6. 시간 칸 상태가 섞인 상황

1시간 후보의 한 칸은 가능이고 다른 칸은 조정 확약 또는 어려움이다.

기대 결과: 보수적 우선순위에 따라 조정 확약 또는 어려움으로 집계

### S7. 조정 확약으로 확정 가능한 상황

필수 참석자와 최소 인원을 충족하지만 한 명 이상이 조정 확약이다.

기대 결과: `READY`, 조정 부담을 보조 정보로 표시

### S8. 응답 추가 후 재판정

PENDING 후보의 미응답자 한 명이 가능 또는 어려움으로 답한다.

기대 결과: 같은 결과 화면에서 후보 상태와 남은 조건을 다시 계산

### S9. 인접한 동등 후보

10:00 시작과 10:30 시작 후보의 참석자별 상태와 남은 조건이 같다.

기대 결과: 하나의 후보군으로 묶되 두 시작 시각을 모두 선택 가능하게 표시

## 20. 성공 지표와 검증 방법

### 20.1 가장 중요한 지표

1. **기준과 다른 확정률**: 설정한 조건을 충족하지 않은 후보를 확정하려 한 비율
2. **판정 정확도**: READY, PENDING, IMPOSSIBLE을 정확히 이해한 비율
3. **결정 조건 이해율**: 특정 필수 응답자와 대체 가능한 대상군을 구분한 비율
4. **불필요한 전원 대기율**: 이미 READY인데 전원 응답을 기다린 비율
5. **첫 안전한 행동까지 걸린 시간**: 잘못된 확정을 피하고 유효한 행동에 도달한 시간
6. **판정 이유 설명 성공률**: 제품 결론을 자신의 말로 설명한 비율

### 20.2 입력 계약 지표

- 미선택 시간의 의미 이해율
- 조정 확약의 의미 이해율
- 30분 칸과 1시간 후보 관계 이해율
- 응답 완료 시간
- 상태 입력 오류와 수정 횟수
- 제출 후 자신의 응답에 대한 확신

### 20.3 검증 방법

- 대상: 업무 회의를 직접 잡아본 경험이 있는 5명 이상
- 비교안 A: 사람별 응답표 중심 결과
- 비교안 B: v2.2의 결론, 이유, 남은 조건 중심 결과
- 입력 비교: 상태 도구 전환 방식과 순차 범위 입력 방식
- 자동화 통과 기준: 모든 필수 시나리오와 경계 조건의 도메인 계약 테스트 100% 통과
- 사용성 통과 기준: 주요 이해 과제 5명 중 4명 이상 성공

작은 표본의 비율을 시장 성과로 일반화하지 않는다. 이 검증은 프로토타입 논리와 이해 가능성을 확인하기 위한 것이다.

### 20.4 출시 후에만 볼 수 있는 지표

- 충분한 응답 이후 확정까지 걸린 시간
- 확정 전 추가 연락 횟수
- 불필요한 전원 대기 비율
- 확정 뒤 재조율률
- 같은 조율 안에서 확정까지 이어진 비율
- 다른 일정 조율 도구에서의 전환과 반복 사용

P0에서는 기대 성과로만 기록하고 달성했다고 주장하지 않는다.

## 21. 수용 기준

v2.2 구현은 다음 조건을 모두 만족해야 한다.

- [ ] 회의 기준 입력이 항상 유효한 `R`과 `Q`로 정규화된다.
- [ ] 참석자가 응답 전에 회의 이름, 주최 맥락, 한 문장 목적, 확보 시간, 응답 마감을 확인할 수 있다.
- [ ] 참석자에게 필수 참석자, 최소 참석 인원, 다른 사람의 응답이 노출되지 않는다.
- [ ] 주최자가 참석자와 최소 인원에 정확히 한 번 포함된다.
- [ ] 제출 전 빈칸과 제출 후 어려움이 내부적으로 구분된다.
- [ ] 제출 후 응답 수정과 새 범위의 `UNSET` 상태를 구분한다.
- [ ] 조정 확약이 READY 판정에 포함되고 추가 확인 상태를 만들지 않는다.
- [ ] 여러 시간 칸의 후보 상태가 정의된 우선순위로 집계된다.
- [ ] 필수 응답자와 대체 가능한 대상군을 후보별로 계산한다.
- [ ] 필요한 가능 응답 수가 필수 참석자의 긍정 응답을 중복 계산하지 않는다.
- [ ] 제품이 일반 참석자 한 명이나 연락 순서를 임의로 추천하지 않는다.
- [ ] 조건이 같은 인접 후보만 후보군으로 묶는다.
- [ ] 시간순 정렬을 추천 근거처럼 표현하지 않는다.
- [ ] 결과 첫 시야에서 현재 확정 가능 여부를 알 수 있다.
- [ ] 상세에서 사람별 상태와 판정 근거를 검토할 수 있다.
- [ ] 응답 수정 후 같은 규칙으로 후보가 재판정된다.
- [ ] 마감 경과가 미응답을 불가로 바꾸거나 후보를 자동 제외하지 않는다.
- [ ] 모든 날짜 계산은 `Asia/Seoul` 기준으로 일관된다.
- [ ] PC와 모바일에서 핵심 흐름을 완료할 수 있다.

## 25. 최종 중심 문장

### 문제 정의

> 일정 조율을 가장 많이 겹치는 시간을 찾는 일이 아니라, 서로 다른 응답을 회의 성립 여부와 남은 결정 조건으로 바꾸는 일로 정의한다.

### 제품 정의

> 확정보드는 지금 정할 수 있는 시간을 보여주고, 정할 수 없다면 반드시 필요한 응답자와 대체 가능한 대상군, 필요한 가능 응답 수를 설명한다.

### 설계 판단

> 더 많은 상태와 기능을 추가하기보다 사용자 응답의 의미를 왜곡하지 않고, 한 칸의 입력이 후보 판정과 화면 문장으로 일관되게 이어지도록 설계한다.

### 이번 과제에서 증명할 것

> 사람별 응답을 모아 보여주는 것보다, 지금 회의를 정할 수 있는지와 어떤 가능 응답 조건이 더 필요한지를 설명하는 결과가 주최자의 안전한 결정을 돕는가?
