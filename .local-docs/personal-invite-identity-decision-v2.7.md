# 개인 초대 링크와 참여자 식별 결정 기록 v2.7

- 상태: 대체됨 / `account-based-request-delivery-decision-v2.8.md`가 현재 계약
- 결정일: 2026-07-12
- 적용 범위: MeetCue P0의 초대 링크 공유, 참여자 진입, 응답 수정
- 제품 명세: `prd-v2.2.1.md`
- 실행 명세: `../UX-ARCHITECTURE.md`

## 1. 문제

기존 프로토타입의 `참여자 선택` 화면은 하나의 공용 링크를 연 사람이 참석자 목록에서 자기 이름을 고르게 했다. 이 구조는 백엔드 없는 데모에서 여러 페르소나를 전환하기에는 편하지만, 제품 경험으로 해석하면 다음 책임을 참여자에게 넘긴다.

- 자신을 정확히 식별할 책임
- 다른 참석자의 이름과 응답 상태를 보게 되는 불필요한 정보 노출
- 잘못된 사람으로 응답하거나 기존 응답을 덮어쓸 위험
- 초대받은 사람이 바로 자기 일정에 답할 수 있다는 기대의 훼손

채용용 프로토타입이라고 해서 이 모순을 제품 화면에 남기면, 실제 서비스에서도 같은 식별 방식을 채택한 것으로 오해받을 수 있다.

## 2. 목표

> 초대받은 사람은 링크를 여는 순간 이미 식별되어 있어야 하며, 자신의 회의 맥락과 응답 과업만 보고 바로 시작해야 한다.

프로토타입의 페르소나 전환 편의와 실제 제품의 식별 계약을 분리한다.

## 3. 검토한 대안

| 대안                        | 장점                                                   | 문제                                                | 결정    |
| --------------------------- | ------------------------------------------------------ | --------------------------------------------------- | ------- |
| 공용 링크에서 이름 선택     | 링크 하나만 공유하면 되고 데모 전환이 쉽다             | 오응답, 사칭, 개인정보 노출, 책임 전가              | 기각    |
| 공용 링크 + 이름과 PIN 입력 | 최소한의 검증을 추가할 수 있다                         | 입력 마찰이 크고 P0에서 PIN 전달 문제를 새로 만든다 | 기각    |
| 계정 로그인 후 본인 식별    | 실제 제품에서는 강한 식별을 제공한다                   | 인증·조직 계정·권한이 필요한 별도 제품 범위다       | P0 제외 |
| 참석자별 개인 토큰 링크     | 바로 응답할 수 있고 사람별 상태와 수정 경로가 명확하다 | 링크 오발송과 전달 위험을 별도로 다뤄야 한다        | 채택    |

## 4. 결정

### 주최자

- 회의 생성 시 지정한 참석자마다 별도의 응답 토큰과 개인 초대 링크가 생긴다.
- 공유 화면은 참석자별 이름, 역할, 응답 상태, 링크 복사, 해당 페르소나 미리보기를 한 행에 묶는다.
- 공용 `응답 링크` 하나를 주 CTA로 제공하지 않는다.
- 실제 메시지 발송은 하지 않고 각 사람에게 보낼 링크 복사까지만 제공한다.

### 참여자

- `#/invite/:token`을 열면 토큰에 연결된 참여자의 신규 응답 화면이 바로 열린다.
- `#/invite/:token/edit`은 같은 참여자의 제출 응답 수정 화면이다.
- `#/invite/:token/done`은 같은 참여자의 제출 완료 화면이다.
- 이름이나 계정을 다시 고르지 않는다.
- 다른 참석자의 이름, 링크, 응답 상태를 보지 않는다.
- 토큰이 없거나 유효하지 않으면 참석자 목록 대신 잘못된 초대 링크 상태만 본다.

### 프로토타입

- 토큰은 보안 인증 수단을 구현한 것이 아니라 사람별 진입 계약을 시연하는 fixture다.
- 여러 참여자 페르소나와 신규·수정·완료 상태 전환은 제품 화면이 아닌 `DEV 화면 바로가기`가 소유한다.
- DEV 메뉴에는 개인 링크별 항목과 별도의 `잘못된 링크` 검증 항목을 둔다.
- 현재 항목 표시는 라우트뿐 아니라 참여자 토큰까지 일치할 때만 활성화한다.

## 5. 사용자 경험의 변화

### 주최자 관점

이전에는 링크 하나를 복사하고 참여자가 알아서 자신을 고르기를 기대했다. 이제는 누구에게 어떤 링크를 보내는지가 명시되고, 아직 응답하지 않은 사람에게 같은 개인 링크를 다시 보낼 수 있다.

### 참여자 관점

이전에는 링크를 연 뒤 자기 이름을 찾아야 했다. 이제는 링크를 열자마자 자신의 이름이 포함된 회의 안내와 일정 응답 과업을 보고 시작하며, 같은 링크에서 제출본을 다시 수정한다.

## 6. 알려진 트레이드오프와 후속 범위

- 개인 링크를 다른 사람에게 전달하면 그 사람의 응답 화면에 접근할 수 있다. 실제 제품에서는 만료, 재발급, 로그인 결합, 접근 로그 중 하나 이상이 필요하다.
- 프로토타입은 토큰의 서버 검증과 영속 저장을 구현하지 않는다.
- 링크를 잘못 보낸 경우의 회수와 재발급은 P0에서 제외한다.
- 이 결정은 참석자의 신원을 법적으로 인증한다는 뜻이 아니라, 제품 흐름에서 자기 선택을 없애는 계약이다.

## 7. 수용 기준

- 개인 링크를 열면 이름 선택 없이 해당 참여자의 응답 화면이 열린다.
- 신규, 수정, 완료 라우트가 같은 토큰의 참여자 맥락을 유지한다.
- 토큰 없는 초대 링크는 참석자 이름이나 응답 상태를 노출하지 않는다.
- 주최자는 공유 화면에서 사람별 링크를 복사하고 해당 화면을 미리 볼 수 있다.
- DEV 화면 바로가기로 각 참여자의 신규·수정·완료와 잘못된 링크 상태를 직접 열 수 있다.
- 참여자 화면에서 다른 참석자의 정보로 이동할 수 없다.

## 8. 검증 질문

- 참여자는 링크를 연 뒤 5초 안에 이 응답이 자신의 것임을 이해하는가?
- 참여자는 별도의 이름 선택 없이 바로 다음 행동을 설명할 수 있는가?
- 주최자는 특정 미응답자에게 어떤 링크를 다시 보내야 하는지 찾을 수 있는가?
- 사용자는 DEV 전환 도구를 실제 제품 기능으로 오해하지 않는가?
