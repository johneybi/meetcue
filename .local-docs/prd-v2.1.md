# PRD v2.1 - 회의 결정 지원

상태: 현재 제품 기준  
기준일: 2026-07-11  
대상: 백엔드 없는 채용 과제용 반응형 프로토타입

## 1. 제품 한 문장

확정보드는 여러 사람의 가용시간을 모으는 데서 멈추지 않고, 지금 회의를 정할 수 있는지와 다음에 누구에게 무엇을 확인해야 하는지를 알려주는 회의 결정 지원 도구다.

## 2. 문제 정의

업무 회의에서 가장 많은 사람이 가능한 시간과 실제로 확정하기 좋은 시간은 다를 수 있다.

- 필수 참석자 한 명이 불가능하면 다수가 가능해도 회의가 성립하지 않는다.
- 참석 인원이 충분해 보여도 누군가의 일정 변경을 전제로 할 수 있다.
- 미응답자가 있어도 현재 응답만으로 이미 회의가 성립할 수 있다.
- 반대로 미응답자 한 명의 답이 있어야만 회의 성립 여부가 결정될 수 있다.

기존 일정 조율 도구도 가용시간, 겹치는 구간, 응답 현황을 제공한다. 이 제품은 그것을 부정하지 않는다. 과제에서 집중할 미해결 지점은 수집된 사실을 다음 두 가지로 바꾸는 일이다.

1. 현재 후보가 회의 성립 기준을 충족하는가
2. 충족하지 않는다면 주최자의 가장 작은 다음 행동은 무엇인가

핵심 문제는 시간 수집이 아니라 결정에 필요한 해석 비용이다.

## 3. 중심 사용자와 핵심 순간

### 중심 사용자

주최자다. 참석자 응답의 부담도 중요하지만, 여러 응답을 종합해 확정·대기·추가 확인을 결정하는 책임과 비용은 주최자에게 집중된다.

### 핵심 순간

응답을 받은 주최자가 결과 화면을 처음 열었을 때 다음 질문에 빠르게 답하는 순간이다.

> 지금 정해도 되는가? 아니라면 누구에게 무엇을 확인해야 하는가?

생성 화면과 참석자 응답은 이 순간에 신뢰할 수 있는 판단을 만들기 위한 입력 과정이다. 대표 품질 기준은 결과 화면에 둔다.

## 4. 제품 가설

회의 성립 기준과 가용시간의 강도를 함께 해석하면, 단순 겹침표보다 주최자의 첫 결정과 후속 요청을 빠르고 정확하게 만들 수 있다.

이 가설은 아직 사용자 조사로 입증된 사실이 아니다. 프로토타입에서는 다음을 검증한다.

- 사용자가 네 가지 결정 상태를 보고 올바른 다음 행동을 고르는가
- 상태 이유를 자신의 말로 설명할 수 있는가
- 모든 미응답자가 아니라 결정에 영향을 주는 사람만 구별하는가
- 가장 많은 인원보다 필수 참석자와 확정 리스크를 우선하는가
- 필요한 정보만 다시 묻는 흐름을 새 요청보다 자연스럽게 이해하는가

## 5. 제품 원칙

1. **응답은 가볍게**: 참석자에게 결정에 필요한 최소 정보만 묻는다.
2. **해석은 제품이**: 주최자가 사람별 표를 직접 계산하지 않게 한다.
3. **근거는 검토 가능하게**: 결론을 먼저 보여주되 상세 응답을 숨기지 않는다.
4. **결정은 주최자가**: 제품은 자동 확정하지 않는다.
5. **다음 행동은 최소 단위로**: 회의 성립 여부를 바꿀 사람과 시간만 다시 확인한다.
6. **재조율은 이어가기**: 기존 링크와 응답을 유지한다.

## 6. 핵심 개념

### 회의 성립 기준

- 참석자
- 필수 참석자
- 최소 참석 인원
- 회의 가능 기간
- 일정 확보 시간
- 주최자의 조율 가능 시간대

전원 참석 회의는 모든 참석자를 필수로 보고 최소 참석 인원을 전체 인원으로 고정한다. 그 외 회의만 필수 참석자와 최소 참석 인원을 별도로 정한다.

### 참석자 가용 상태

- `가능해요`: 현재 일정 그대로 참석할 수 있다.
- `일정 조정하면 가능해요`: 다른 일정을 바꿀 수 있는지 최종 확인이 필요하다.
- `어려워요`: 해당 시간에는 참석하기 어렵다.
- 미응답: 아직 판단 근거가 없다.

선호는 가용 상태와 분리한다. `가능하면 피하고 싶어요`, `점심 직후예요`, `앞뒤 일정이 촉박해요`, `이동 시간이 필요해요`는 후보 비교에는 쓰지만 회의 성립 자체를 막지 않는다.

### 파생 후보

주최자가 연 조율 가능 시간대와 일정 확보 시간에서 30분 간격의 실제 후보를 계산한다. 참석자는 후보에 투표하지 않고 조율 범위의 가용 상태를 입력한다.

### 결정 상태

| 상태                            | 사용자가 이해해야 할 결론                                | 우선 행동                     |
| ------------------------------- | -------------------------------------------------------- | ----------------------------- |
| 바로 정할 수 있어요             | 현재 일정 그대로 필수 참석자와 최소 인원을 충족한다      | 이 시간으로 정하기            |
| 정하기 전에 한 번 확인해 주세요 | 일정 조정이 실제로 가능한지 확인하면 성립한다            | 필요한 사람에게 확인          |
| 응답이 더 필요해요              | 특정 미응답자의 답에 따라 성립 여부가 달라진다           | 필요한 사람에게 응답 요청     |
| 다른 시간을 찾아야 해요         | 현재 응답으로는 필수 조건이나 최소 인원을 충족할 수 없다 | 다른 후보 검토 또는 범위 확장 |

정확한 판정 규칙은 `decision-model-v2.1.md`를 따른다.

## 7. 핵심 사용자 흐름

### A. 주최자 생성

```text
회의 안내
→ 참석자 선택
→ 전원 참석 여부
→ 필요한 경우 필수 참석자와 최소 인원
→ 기간과 일정 확보 시간
→ 주최자 조율 가능 시간대
→ 응답 마감
→ 검토와 링크 생성
```

단계마다 현재 질문 하나만 보여준다. 결정 화면에서 쓰지 않는 정보는 받지 않는다.

### B. 참석자 응답

```text
회의 맥락 확인
→ 조율 범위 확인
→ 30분 칸별 3상태 입력
→ 필요한 경우 선호 이유 추가
→ 저장
```

다른 참석자의 응답과 제품 판정은 보이지 않는다.

### C. 주최자 결정

```text
전체 결론
→ 가장 행동 가능한 후보
→ 상태 이유와 영향 인원
→ 확정 또는 대상 지정 요청
→ 필요할 때 상세 근거 검토
```

후보 수가 많아도 첫 화면에서 모두 같은 강도로 나열하지 않는다. 행동 가능성이 높은 상태 순으로 묶고, 우선 후보 하나를 먼저 읽게 한다.

### D. 이어서 조율

```text
필요한 응답 요청
또는 일정 조정 확인
또는 주최자 시간대 확장
→ 기존 응답 유지
→ 새 정보만 수집
→ 같은 결정 화면에서 다시 판정
```

## 8. P0 기능 범위

### 포함

- 회의 안내와 참석 기준 설정
- 주최자 조율 가능 시간대 입력
- 참석자 30분 단위 3상태 응답
- 선호 태그 선택 입력
- 파생 후보 계산
- 후보별 결정 상태와 판정 이유
- 회의 성립에 영향을 주는 사람 식별
- 상태별 확정·확인·응답 요청·범위 확장 행동
- 기존 응답을 유지하는 이어서 조율
- 확정 문구 복사
- PC·모바일 반응형 경험

### 제외

- 실제 계정, 조직도, 권한, 서버 저장
- Google·Outlook 캘린더 연동
- 메시지 자동 발송과 자동 리마인드
- AI 후보 추천과 자동 확정
- 회의실 예약, 반복 회의, 타임존 고급 처리
- 주최자가 참석하지 않는 대리 조율

## 9. 차별화의 경계

차별점으로 주장할 수 있는 것은 단일 기능의 최초성이 아니다.

> 회의 기준, 가용 상태의 강도, 미응답의 영향도를 하나의 판정 모델로 묶고 그 결과를 주최자의 다음 행동까지 연결한 경험 구조

다음은 과장하지 않는다.

- 기존 제품은 가능 시간만 수집한다.
- 기존 제품은 확정이나 재조율을 전혀 돕지 않는다.
- 사용자가 재조율을 가장 큰 문제로 느낀다.
- 네 가지 상태만으로 실제 회의를 완전히 자동 판단할 수 있다.

## 10. 성공 기준

### 과제 시연 성공

- 결과 화면 첫 시야에서 현재 확정 가능 여부와 우선 행동이 보인다.
- 서로 다른 상태의 후보를 비교했을 때 사용자가 더 안전한 후보를 고른다.
- 미응답자가 있어도 기다릴 필요가 없는 경우를 구별한다.
- 확인이 필요한 사람과 단순 참고 대상을 구별한다.
- 막힌 상태에서 새 회의를 만들지 않고 다음 행동으로 이어진다.

### 사용성 검증 지표

- 첫 결정까지 걸린 시간
- 올바른 후보 선택률
- 불필요한 전원 대기 선택률
- 상태 이유 설명 성공률
- 올바른 후속 행동 선택률
- 상세 근거를 열지 않고 결정한 비율과 잘못된 결정률

## 11. 프로토타입 필수 시나리오

1. **바로 확정**: 일반 참석자 미응답이 있지만 필수 참석자와 최소 인원을 충족한다.
2. **조정 확인**: 필수 참석자 또는 최소 인원 충족에 필요한 사람이 `일정 조정하면 가능`이다.
3. **필요한 응답**: 특정 미응답자의 답에 따라 최소 인원 또는 필수 조건이 결정된다.
4. **다른 시간 필요**: 필수 참석자가 어렵거나 가능한 최대 인원이 최소 인원보다 적다.
5. **이어서 조율**: 필요한 사람에게만 요청하거나 시간대를 넓힌 뒤 기존 응답을 유지한다.

## 12. 오픈 리스크

- 세 단계 가용 상태를 참석자가 일관되게 해석하는가
- `일정 조정하면 가능`인 일반 참석자가 많을 때 위험 임계값을 어떻게 둘 것인가
- 미응답자가 후보 순위에 미치는 영향을 사용자가 납득하는가
- 많은 파생 후보를 상태별로 묶는 방식이 비교를 방해하지 않는가
- 제품 판정이 잘못됐을 때 주최자가 근거를 보고 안전하게 뒤집을 수 있는가

이 항목은 숨기지 않는다. 프로토타입은 완전 자동화가 아니라 설명 가능한 결정 지원의 타당성을 검증한다.
