# Decision Coherence Sprint v2.2.1

- 상태: 실행 중
- 기준일: 2026-07-11
- 제품 UI명: MeetCue
- 과거 기획명: 확정보드
- 최우선 범위 잠금: `p0-submission-scope-lock-v2.10.md`

## 1. 목적

이번 스프린트는 기능 개수를 늘리는 작업이 아니다. 다음 사용자 결과를 검증 가능한 제품 경험으로 만든다.

> 회의 주최자가 결과를 처음 본 뒤, 지금 확정할 수 있는지 판단하고, 확정할 수 없다면 반드시 필요한 특정인의 가능 응답과 대체 가능한 대상군에서 필요한 가능 응답 수를 정확히 설명할 수 있다.

제출용 완성 범위는 `회의 기준 설정 → 참석자 응답 → 주최자 판단 → 한 사람에게 재요청 → 응답 변경 후 재판정 → 확정`이다. 계정 홈, 여러 회의 목록, 받은 요청, 알림은 구현 코드가 있더라도 P0 완료 게이트에서 제외하고 DEV 전용 후속 제품 탐색으로 둔다.

## 2. 대표 평가 과제

사용자에게 사람별 응답을 직접 계산하라고 요구하지 않고 다음 질문에 답하게 한다.

1. 지금 정할 수 있는 시작 시각이 있는가?
2. 선택한 시각이 READY, PENDING, IMPOSSIBLE인 이유는 무엇인가?
3. PENDING이면 누구의 가능 응답이 반드시 필요한가?
4. 대체 가능한 미응답자 중 몇 명의 가능 응답이 필요한가?
5. 누구에게 응답을 요청할지는 어떻게 선택하는가?
6. 한 사람이 응답을 수정하면 결론이 어떻게 바뀌는가?

## 3. 성공 게이트

### 의미 무결성

- 제출 전 선택은 초안이며 판정과 주최자 상세에서 `미응답`이다.
- 제출 후에만 시간 칸을 후보 상태로 집계한다.
- 조정 확약은 READY를 막지 않지만 부담 정보로 남는다.
- 같은 입력은 같은 후보, 상태, 이유, 정렬을 만든다.

### 여정 완결성

- READY, PENDING, IMPOSSIBLE이 같은 결과 보드에서 비교된다.
- PENDING은 일반 대기 화면으로 축약되지 않는다.
- 주최자가 필수 특정인과 대체 가능한 대상군을 구분해 요청 대상을 직접 고른다.
- 모든 후보가 IMPOSSIBLE이면 기존 시간 응답을 유지한 채 회의 기준을 다시 검토한다.
- 한 사람의 제출 또는 수정 뒤 같은 결과 화면이 다시 계산된다.

### 범위 규율

- 공유 후 새 시간 범위 추가와 응답 범위 버전 관리는 구현하지 않는다.
- 입력 방식 A/B 비교는 이번 스프린트의 성공 조건이 아니다.
- 실제 발송, 계정, 캘린더 연동, 장소 입력은 구현하지 않는다.

### 검증 가능성

- 필수 PRD 시나리오는 도메인 테스트로 재현한다.
- 결과 화면에는 실제 후보 수, 필수 응답자, 대체 가능 대상군, 필요한 가능 응답 수가 원본 계산과 일치한다.
- 타입 검사, 린트, 프로덕션 빌드가 통과한다.
- 자동 브라우저 시각 검수는 별도 합의 전까지 수행하지 않는다.

## 5. 의도적으로 미루는 판단

- MeetCue를 최종 제출 UI명으로 사용한다.
- 순차 범위 입력과 상태 도구 입력 중 어느 쪽이 더 빠른가
- 마감 이후 실제 서비스에서 응답을 잠글 것인가
- 모든 후보가 어려울 때 새 시간 범위를 같은 링크에서 여는가

P0에서는 마감을 운영 안내로 사용하고 늦은 응답도 받는다. 이름과 후속 기능은 검증 결과 뒤 별도 결정한다.
