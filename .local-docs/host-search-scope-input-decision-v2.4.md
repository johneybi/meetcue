# 주최자 탐색 범위 입력 결정 기록 v2.4

- 상태: 채택 / 문서 반영 완료 / 구현 전
- 결정일: 2026-07-11
- 적용 범위: MeetCue P0의 주최자 시간 생성 단계
- 선행 결정: `availability-window-model-decision-v2.3.md`
- 제품 명세: `prd-v2.2.1.md`
- 실행 명세: `../UX-ARCHITECTURE.md`

> 참여자 입력 후속 결정: `participant-availability-input-decision-v2.5.md`가 참여자의 반복 3버튼 입력을 `일반 패턴 선택 + 3상태 예외 편집`으로 구체화한다. 주최자 입력을 줄인 비용을 참여자에게 이전하지 않는 짝 결정이다.

## 1. 결정 요약

주최자가 빈 시간표에서 가능한 시간을 모두 칠해야만 다음 단계로 갈 수 있는 구조를 폐기한다.

> 제품은 회의 가능 기간과 확보 시간을 바탕으로 기본 탐색 범위를 먼저 제안한다. 주최자는 모든 가능 시간을 새로 만들지 않고, 안 되는 시간을 제외하거나 필요한 시간을 추가한다. 별도 그리드 조작 없이 계속하면 화면에 보이는 남은 범위를 자신이 참석 가능한 조율 범위로 확인한 것으로 본다.

P0 대표 fixture의 기본 탐색 범위는 선택한 날짜 안의 평일 `09:00~18:00`이다. 이는 제품의 보편적 업무시간 규칙이 아니라 백엔드와 조직 설정이 없는 프로토타입 기본값이다. 실제 제품에서는 조직 업무시간, 개인 설정 또는 캘린더 free/busy로 대체할 수 있다.

정확한 후보를 가용시간과 확보 시간에서 파생하는 v2.3의 핵심 결정은 유지한다. 이번 결정은 주최자 입력의 초기 상태와 필수 조작만 대체한다.

## 2. 다시 검토한 계기

v2.3은 주최자가 정확한 후보 3~5개를 먼저 고를 때 생기는 후보 누락을 피하기 위해 넓은 조율 가능 시간대를 받도록 했다. 그러나 구현에서는 주최자가 다음 작업을 완료해야만 진행할 수 있었다.

1. 날짜 범위를 정한다.
2. 확보 시간을 정한다.
3. 빈 30분 그리드에 자신이 가능한 모든 범위를 직접 칠한다.
4. 선택한 범위를 다시 확인한다.

이 구조는 참석자가 검토할 시간을 줄여주지만 그 비용을 중심 사용자인 주최자에게 선불로 이전한다. 제품의 중심 가치가 주최자의 판단 부담을 줄이는 것이라면, 주최자에게 조율판 자체를 설계하도록 요구하는 것은 방향과 충돌한다.

## 3. 구분해야 하는 정보

### 회의 가능 기간

회의를 실제로 실시할 수 있는 날짜의 외곽 경계다. 개인의 빈 시간을 뜻하지 않는다.

### 일정 확보 시간

후보 하나가 성립하기 위해 연속으로 비워둘 시간이다. 실제 종료 시각을 보장하지 않는다.

### 기본 탐색 범위

제품이 참석자에게 물어볼 수 있도록 먼저 구성한 잠정 시간 범위다. 아직 주최자의 확정 가용시간으로 저장하지 않는다.

### 주최자 조율 가능 시간대

기본 탐색 범위에서 예외 편집을 마치고 계속한 뒤 남은 범위다. P0에서 주최자는 항상 참석하므로 이 범위 안에서는 주최자가 암묵적으로 참석 가능한 것으로 판정한다.

### 파생 후보

확인된 조율 가능 시간대 안에서 확보 시간이 온전히 들어오는 30분 간격의 시작 시각이다. 주최자가 직접 입력하는 값이 아니다.

## 4. 제거할 수 없는 입력과 제거할 수 있는 노동

제품이 임의로 추론하면 안 되는 정보는 다음과 같다.

- 회의를 실시해야 하는 날짜 범위
- 일정에서 확보할 시간
- 주최자가 최종적으로 참석 가능하다고 확인한 범위

반면 다음 노동은 제품이 줄일 수 있다.

- 빈 시간표에서 가능한 칸을 모두 새로 칠하기
- 같은 업무시간을 날짜마다 반복 입력하기
- 후보 시작 시각을 일일이 계산하기
- 선택 범위를 참석자용 조율 범위로 다시 구성하기

제품은 제약을 받되 조율판 제작을 사용자에게 맡기지 않는다.

## 6. 채택한 경험 계약

### 6.1 기본 흐름

```text
회의 가능 기간과 확보 시간 입력
→ 제품이 기본 탐색 범위 생성
→ 주최자가 요약 확인
→ 필요하면 제외·추가 편집
→ 계속하여 남은 범위를 주최자 조율 가능 시간대로 확정
```

- 시간표를 한 칸도 조작하지 않아도 계속할 수 있다.
- 계속 행동은 현재 보이는 범위에서 주최자가 회의에 참석할 수 있다는 명시적 확인을 겸한다.
- 기본 범위와 사용자가 편집한 범위를 시각적으로 구분할 필요는 없지만, 현재 적용 범위는 항상 문장과 요약으로 확인할 수 있어야 한다.
- 가능한 범위가 0이면 계속할 수 없다.

### 6.2 편집 방식

시간표는 후보 선택기가 아니라 범위 편집기다.

- `제외할 시간`: 현재 범위에서 연속 구간을 제거한다.
- `추가할 시간`: 제거했거나 기본값에 없던 연속 구간을 포함한다.
- 드래그와 터치 드래그를 지원하되 클릭·탭 대안을 제공한다.
- 이 화면은 30분 셀을 칠하고 지우는 페인트 입력만 사용한다. 선택 블록, 호버 리사이즈 핸들, 블록별 이동·삭제 목록을 함께 제공하지 않는다.
- 클릭·탭은 정확히 한 칸만 바꾸고, 드래그는 지나간 연속 칸을 같은 브러시 상태로 바꾼다.
- 확보할 시간은 브러시 크기가 아니다. 칠한 범위 안에서 유효한 후보를 계산할 때만 사용한다.
- 하루 전체 적용, 여러 날 반복, 되돌리기, 기본값 복원을 제공할 수 있다.
- 셀을 반복 클릭해 상태를 순환시키는 것보다 현재 편집 도구를 먼저 선택하는 브러시 모델을 우선 검증한다.
- 긴 범위를 첫 셀에서 시작하는 하나의 대형 오버레이로 그리지 않는다. 각 30분 칸의 선택 상태와 실제 클릭 영역을 일치시킨다.
- 선택된 칸은 동일한 파란 표면으로 표시하고, 그리드 위 범례에서 `참석자에게 물어볼 시간`임을 명시한다.
- 범위 개수는 중간 제외 때 증가해 의미가 없으므로 `선택된 날짜 수 · 총 시간`을 우선 요약한다.

`되는 시간 / 안 되는 시간` 패턴은 유효한 참고다. 다만 제품 언어는 개인 일정 기록보다 조율 범위 편집이라는 목적을 드러내기 위해 `추가할 시간 / 제외할 시간`을 우선 사용한다.

### 6.3 의미 무결성

- 기본 탐색 범위는 제안일 뿐이며 계속하기 전에는 판정 데이터가 아니다.
- 계속한 뒤 남은 모든 칸은 주최자 `AVAILABLE`로 취급한다.
- 제외된 칸에서는 후보를 만들지 않는다.
- 주최자에게 `ADJUSTMENT_COMMIT` 상태를 별도로 받지 않는다. 조정 가능한 시간을 열고 싶다면 주최자가 포함 여부를 결정한다.
- 참석자의 3상태 입력과 주최자의 2상태 범위 편집을 같은 컴포넌트 의미로 합치지 않는다.

## 7. 제품 원칙과의 정합성

기존 원칙 `응답은 가볍게, 해석은 제품이, 결정은 주최자가`를 생성에도 확장한다.

> 제약은 주최자가, 탐색 범위의 초안은 제품이, 예외는 주최자가, 후보 계산은 제품이 맡는다.

주최자는 결과에 책임을 지지만 탐색 공간을 처음부터 제작할 필요는 없다. 제품은 주최자의 확인 없이 참석 가능성을 확정하지 않으며, 동시에 반복적인 시간표 입력을 필수 과제로 만들지 않는다.

## 8. 알려진 위험과 검증 질문

### 기본값 무비판 승인

사용자가 기본 범위를 읽지 않고 계속하면 주최자가 실제로 불가능한 후보가 생길 수 있다. 다음을 검증한다.

- 사용자가 계속 행동을 자신의 가용 범위 확인으로 이해하는가?
- 현재 범위를 5초 안에 설명할 수 있는가?
- 안 되는 시간을 수정할 수 있다는 사실을 발견하는가?

### 기본 업무시간의 부적합

평일 09:00~18:00은 프로토타입 fixture일 뿐이다. 주말 근무, 교대 근무, 해외 협업에는 맞지 않는다. P0에서는 명시적인 요약과 편집을 제공하고, 실제 제품에서는 조직·개인 설정으로 대체한다.

### 부담의 참석자 재전가

주최자 입력을 줄였다고 참석자의 전체 그리드 입력이 자동으로 가벼워지는 것은 아니다. 참석자 입력은 별도 문제로 유지하며 다음을 비교한다.

- 상태 브러시로 연속 범위 입력
- 기본 상태를 정하고 예외만 수정
- 캘린더에서 free/busy 가져오기
- 제한된 후보 투표형

## 9. 구현 영향

- 빈 그리드와 필수 선택 검증을 기본 탐색 범위 초기화로 대체한다.
- 확보 시간은 1시간 기본값을 유지하되 30분 단위로 변경할 수 있다.
- `AvailabilityWindowPicker`에 `add`와 `exclude` 편집 모드를 둔다.
- 시간 단계 CTA는 선택 개수가 아니라 유효한 남은 범위와 주최자 확인 여부로 활성화한다.
- 파생 후보 계산, 참석자 응답, READY·PENDING·IMPOSSIBLE 판정은 변경하지 않는다.
- 캘린더 연동은 P0에서 구현하지 않는다.

## 10. 이번 결정이 대체하거나 유지하는 것

### 대체

- 주최자가 빈 30분 그리드에서 가능한 범위를 필수로 생성한다.
- 시간대를 하나도 칠하지 않으면 단순히 다음 단계로 갈 수 없다.
- 주최자 입력을 후보 축소를 위한 수동 사전 작업으로 표현한다.

### 유지

- 주최자는 항상 참석한다.
- 확인된 주최자 범위 밖에서는 후보를 만들지 않는다.
- 참석자는 열린 범위 안에서 3상태 가용시간을 제출한다.
- 후보는 30분 간격과 선택된 확보 시간으로 계산한다.
- 최종 결정은 주최자가 내린다.
