# 30분 시간 최소 단위 결정 기록 v2.2

- 상태: P0 프로토타입 채택
- 결정일: 2026-07-10
- 적용 범위: 확정보드의 후보 생성, 시간 블록, 향후 가용시간 그리드
- 선행 결정: `.local-docs/meeting-time-reservation-decision-v2.1.md`
- 실행 명세: `UX-ARCHITECTURE.md`
- 제품 명세: `.local-docs/prd-v2.0.md`

> 후속 결정: `.local-docs/availability-window-model-decision-v2.3.md`가 이 문서에서 열어둔 후보 응답형과 가용시간 수집형 중 가용시간 수집형을 채택했다. 30분 최소 단위는 유지된다.

## 1. 문제 제기

일정 확보 시간을 입력하는 UI를 검토하면서 30분, 45분, 60분, 90분 프리셋과 5분 단위 직접 입력을 함께 제공하는 방향을 세웠다. 그러나 후보와 참여자 가용시간을 시간표 형태로 표현하려면 입력값보다 먼저 그리드 한 칸의 최소 단위를 정해야 한다.

다음 값이 서로 다르면 같은 시간을 화면과 판정 로직에서 일관되게 표현할 수 없다.

- 그리드 한 칸의 시간
- 후보가 시작할 수 있는 시각 간격
- 확보 시간의 증감 단위
- 참여자가 선택하거나 응답하는 시간 블록
- 후보 충족 여부를 계산하는 최소 단위

따라서 칩이나 스테퍼를 먼저 선택하지 않고 제품 전체의 시간 최소 단위를 먼저 검토했다.

## 2. 구분해야 하는 개념

### 시간 최소 단위

제품이 서로 다른 시간을 구분하는 가장 작은 간격이다. 이 문서에서는 `time quantum`이라고 부른다.

### 기본 회의 시간

새 회의에 미리 채워두는 초기 확보 시간이다. 시간 최소 단위와 같은 값일 필요가 없다.

### 빠른 입력값

사용자가 자주 입력할 것으로 예상해 바로 선택할 수 있게 노출하는 값이다. 허용 가능한 전체 값의 목록이 아니다.

Google Workspace가 30분을 기본 일정 길이로 권장하는 것은 30분이 익숙한 초기값이라는 참고 신호다. 그 자체로 그리드 한 칸도 30분이어야 한다는 증거는 아니다.

## 3. 검토한 최소 단위

9시간 업무일을 한 열로 표시한다고 가정했다.

| 최소 단위 | 하루 칸 수 | 장점 | 문제 |
| --- | ---: | --- | --- |
| 5분 | 108칸 | 거의 모든 시간을 정확히 표현 | 화면 밀도와 드래그 정밀도가 과도하고 모바일 사용이 어렵다 |
| 15분 | 36칸 | 15분·45분 회의와 `:15`, `:45` 시작을 표현 | 30분 대비 행 수가 두 배이며 모바일 탐색이 길어진다 |
| 30분 | 18칸 | PC와 모바일에서 스캔·탭·드래그가 단순하다 | 15분·45분과 `:15`, `:45` 시작을 그대로 표현하지 못한다 |
| 상황별 가변 단위 | 화면마다 다름 | 정밀도와 단순성을 상황별로 조정 가능 | 입력, 표시, 판정 단위가 달라져 사용자가 결과를 예측하기 어렵다 |

## 4. 외부 자료를 해석한 방식

- Google Workspace는 조직의 기본 이벤트 길이로 30분을 권장한다. 이는 익숙한 기본값의 근거로만 사용한다. <https://knowledge.workspace.google.com/admin/calendar/set-the-default-duration-for-events-in-your-organization?hl=en>
- Microsoft Outlook은 일반적인 캘린더 표시 간격으로 15분과 30분을 언급한다. 두 단위 모두 통상적인 캘린더 표현이라는 참고 자료다. <https://support.microsoft.com/en-US/Outlook/calendar/change-the-calendar-time-scale-in-outlook>
- Supernormal의 2025년 데이터는 평균 회의 시간이 47분이고 60분 초과 회의가 37%라고 보고한다. 이는 몇 개의 프리셋이 모든 회의를 대표한다고 주장할 수 없다는 근거이지, 5분 단위가 필요하다는 근거는 아니다. <https://www.supernormal.com/state-of-meetings>

외부 자료는 제품 사용자의 실제 분포를 대신하지 않는다. 최종 선택은 P0의 복잡성 예산과 검증 목적을 함께 고려한 설계 결정이다.

## 5. 판단 기준

P0에서는 다음 우선순위를 적용한다.

1. 참여자가 시간표를 빠르게 이해하고 선택할 수 있어야 한다.
2. PC와 모바일이 같은 시간 모델을 사용해야 한다.
3. 후보 생성과 충족 판정이 단순해야 한다.
4. 실제 회의 소요 시간보다 충돌을 피하기 위한 안전한 일정 확보를 우선한다.
5. 낮은 빈도의 정밀한 시간 요구 때문에 기본 화면 전체를 복잡하게 만들지 않는다.

## 6. 결정

> P0의 시간 최소 단위는 30분으로 고정한다.

```ts
const TIME_QUANTUM_MINUTES = 30
```

- 후보 시작 시각은 정각 또는 30분이다.
- 일정 확보 시간은 30분의 양의 배수다.
- 후보 종료 경계도 정각 또는 30분이다.
- PC와 모바일은 같은 30분 단위를 사용한다.
- 5분·15분 단위 직접 입력은 P0에서 제공하지 않는다.
- 45분 프리셋은 제거한다.
- 30분, 60분, 90분, 120분은 모두 같은 모델에서 각각 1~4칸으로 표현한다.

## 7. 실제 예상 시간과 확보 시간의 관계

시간 최소 단위는 실제 회의가 반드시 30분 단위로 끝난다는 뜻이 아니다. 실제 예상 시간이 단위 사이에 있으면 다음 30분 경계로 올림해 일정을 확보한다.

| 실제 예상 | 일정 확보 |
| ---: | ---: |
| 15분 | 30분 |
| 30분 | 30분 |
| 45분 | 60분 |
| 70분 | 90분 |

이 정책은 실제 소요 시간을 왜곡해 표시하기 위한 것이 아니라 다른 일정과의 충돌을 피하기 위한 보수적인 예약 정책이다. UI에서는 실제 예상 시간이 아니라 확보할 시간만 받는다.

## 8. 입력 UI 영향

- 초기에는 확보 시간을 선택하지 않은 상태로 둔다.
- 빠른 선택으로 30분, 60분, 90분, 120분을 제공한다.
- `직접 입력`을 선택하면 빠른 선택 칩을 접고 30분 단위 스테퍼와 숫자 입력만 펼친다.
- 직접 입력 컨테이너 자체를 선택 영역처럼 강조하지 않는다. 현재 숫자 값과 증감 버튼을 주 조작 대상으로 보이게 한다.
- 스테퍼 위에는 낮은 위계의 `직접 입력` 라벨과 닫기 아이콘을 둔다.
- 닫기 아이콘은 직접 입력값을 취소하고 빠른 선택 칩을 다시 표시한다. `Esc`도 같은 동작을 수행한다.
- 닫은 뒤 포커스는 다시 나타난 `직접 입력` 버튼으로 이동한다.
- 직접 입력을 열 때 기존 선택값이 있으면 유지하고, 없으면 사용자가 직접 입력을 선택한 시점에 30분부터 시작한다.
- 직접 입력은 30~240분 범위의 30분 배수만 유효하다.
- 프리셋과 직접 입력을 모두 고르지 않으면 후보 선택으로 진행할 수 없다.

## 9. 그리드와 판정 영향

### 주최자 후보 그리드

- 한 칸은 30분이다.
- 확보 시간이 60분이면 연속 2칸을 차지한다.
- 후보를 선택할 때 전체 점유 블록이 보이게 한다.

### 현재 참여자 후보 응답

현재 P0 계약에서는 참여자가 자유 가용시간을 칠하지 않고 주최자가 보낸 정확한 후보에 응답한다. 이 경우 참여자는 `14:00~15:00`처럼 완성된 블록을 보고 `가능해요`, `일정 조정하면 가능해요`, `어려워요` 중 하나를 선택한다.

### 향후 참여자 가용시간 그리드

참여자가 직접 가능한 시간을 칠하는 모델을 채택하면 같은 30분 단위를 사용한다. 60분 후보가 가능하려면 연속 2칸이 가능해야 한다.

시간 최소 단위 결정만으로 후보 응답형과 가용시간 입력형 중 하나를 확정하지 않는다. 두 방식은 별도 IA 결정이 필요하다.

## 10. 포기하는 정밀도

P0에서는 다음 상황을 정확히 표현하지 못한다.

- 15분 회의만 잡고 싶은 경우
- 45분만 확보하고 싶은 경우
- `09:15` 또는 `09:45`에 시작해야 하는 경우
- 30분 미만의 빈틈을 활용해야 하는 경우

이 손실은 알려진 제약이다. 사용자가 실제로 일정을 잡지 못하거나 과도한 확보 시간 때문에 후보가 사라지는 문제가 반복될 때 15분 단위 확장을 검토한다.

## 11. 구현 계약

```ts
reservedDurationMinutes % TIME_QUANTUM_MINUTES === 0
startMinute % TIME_QUANTUM_MINUTES === 0
reservedDurationMinutes >= TIME_QUANTUM_MINUTES
```

기존 코드의 45분 프리셋, 5분 스테퍼, 5분 배수 검증은 이 결정과 불일치했으며 v2.2 구현에서 다음과 같이 수정한다.

- 프리셋은 30분, 60분, 90분, 120분으로 제한한다.
- 직접 입력 스테퍼와 숫자 검증은 30분 단위로 통일한다.
- 빠른 선택과 직접 입력을 동시에 펼치지 않되, 별도 모드 패널처럼 과도하게 강조하지 않는다.
- `빠른 선택`처럼 결과를 추론해야 하는 전환 문구 대신 익숙한 닫기 아이콘을 사용한다.
- 후보 시작 간격은 기존 30분 그리드를 유지한다.
- 후보 블록이 확보 시간만큼 차지한다는 시각적 표현은 후속 그리드 개선에서 반영한다.

## 12. 기존 결정과의 관계

이 문서는 `.local-docs/meeting-time-reservation-decision-v2.1.md`의 일정 확보 시간 의미를 유지하면서 입력 가능 단위만 구체화하고 대체한다.

유지되는 결정은 다음과 같다.

- 실제 회의 종료와 일정 확보 시간을 구분한다.
- 모든 후보는 같은 확보 시간을 사용한다.
- 종료 시각이 전혀 없는 후보는 P0에서 허용하지 않는다.

대체되는 결정은 다음과 같다.

- 15~240분 범위의 5분 단위 직접 입력
- 45분 프리셋
- 시간 단위와 무관한 임의의 확보 시간

## 13. 검증 가설

- 30분 단위 그리드는 별도 설명 없이도 빠르게 이해된다.
- 모바일에서 30분 칸을 탭하거나 드래그할 때 오입력이 적다.
- 15분 회의를 30분, 45분 회의를 60분 확보하는 정책이 일반 업무 회의에서 허용된다.
- 30분 단위 때문에 실제로 가능한 후보가 사라지는 사례가 드물다.
- 15분 단위보다 작업 완료 시간이 짧다.

## 14. 검증 질문

- 45분 정도 필요한 회의에 1시간을 비워두는 것이 자연스러운가?
- 15분 단위 시작 시각이 없어 회의를 만들지 못하는 경우가 있는가?
- 30분 한 칸을 선택했을 때 확보되는 범위를 정확히 이해하는가?
- 60분 후보가 연속 2칸이라는 관계를 이해하는가?
- 더 정밀한 시간 선택과 더 단순한 조작 중 무엇이 중요한가?

## 15. 아직 확정하지 않은 문제

- 참여자 응답을 후보 응답형으로 유지할지 가용시간 그리드로 바꿀지
- 15분 단위를 후속 고급 설정으로 제공할지
