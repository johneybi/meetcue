# 회의 결정 모델 v2.1

상태: 구현 계약  
기준 문서: `prd-v2.1.md`

## 1. 목적

가용시간 원본을 후보별 회의 성립 상태, 이유, 영향 인원, 다음 행동으로 변환한다. 이 문서는 UI 문구가 아니라 도메인 판정의 단일 기준이다.

## 2. 입력

- 주최자와 참석자 목록
- 필수·선택 참석자 역할
- 최소 참석 인원
- 주최자의 조율 가능 시간대
- 일정 확보 시간
- 참석자별 `available` / `adjustable` / `unavailable` / 미응답
- 선택형 선호 태그
- 응답 마감

주최자는 자신이 연 모든 후보에 `available`로 암묵 포함하며 응답 대상 수에는 포함하지 않는다.

## 3. 후보별 파생 값

- `firmAvailable`: 주최자 + `available` 응답자
- `adjustable`: `adjustable` 응답자
- `attendable`: `firmAvailable + adjustable`
- `missing`: 미응답자
- `requiredUnavailable`: 필수 참석자 중 `unavailable`
- `requiredAdjustable`: 필수 참석자 중 `adjustable`
- `missingRequired`: 필수 참석자 중 미응답
- `neededAdjustable`: 최소 인원 충족에 실제로 필요한 `adjustable` 응답자
- `neededMissing`: 답에 따라 최소 인원 충족 여부가 달라지는 미응답자

`neededAdjustable`과 `neededMissing`은 모든 대상을 나열하지 않는다. 필요한 인원 수와 후보 순위를 기준으로 회의 성립 여부를 바꿀 수 있는 사람만 행동 대상으로 표시한다.

## 4. 판정 우선순위

아래 규칙은 위에서 먼저 일치한 상태를 사용한다.

### 4.1 다른 시간을 찾아야 해요

- 후보 시간이 지났거나 일정 확보 시간과 다름
- 필수 참석자 중 `unavailable`이 있음
- `firmAvailable + adjustable + missing`이 최소 참석 인원보다 적음
- 마감 이후 `firmAvailable + adjustable`이 최소 참석 인원보다 적음

행동: 다른 후보 검토. 모든 후보가 같으면 조율 가능 기간·시간대·확보 시간을 재검토한다.

### 4.2 응답이 더 필요해요

- 필수 참석자 중 미응답이 있음
- 또는 `firmAvailable + adjustable`만으로 최소 인원을 채우지 못하지만 미응답자가 답하면 채울 수 있음

행동: `missingRequired`와 `neededMissing`에게만 응답을 요청한다.

### 4.3 정하기 전에 한 번 확인해 주세요

- 필수 참석자 중 `adjustable`이 있음
- 또는 `firmAvailable`만으로 최소 인원을 채우지 못하고 `adjustable` 응답자의 참여가 있어야 채울 수 있음

행동: `requiredAdjustable`과 `neededAdjustable`에게 일정 조정 가능 여부를 확인한다.

### 4.4 바로 정할 수 있어요

- 필수 참석자가 모두 `available`
- `firmAvailable`만으로 최소 참석 인원을 충족

일반 참석자의 미응답이나 선택적 `adjustable`은 확정을 막지 않는다. 보조 위험과 후보 순위에는 반영한다.

## 5. 후보 정렬

상태 우선순위:

1. 바로 정할 수 있어요
2. 정하기 전에 한 번 확인해 주세요
3. 응답이 더 필요해요
4. 다른 시간을 찾아야 해요

같은 상태에서는 다음 순서로 정렬한다.

1. 필수 참석자 조정 수가 적음
2. 최소 인원 충족에 필요한 조정 수가 적음
3. 전체 조정 수가 적음
4. 선호 충돌이 적음
5. 현재 일정 그대로 가능한 인원이 많음
6. 결정에 영향을 주는 미응답이 적음
7. 시작 시각이 빠름

인원수가 가장 많은 후보가 회의 성립 위험이 더 낮은 후보보다 앞서지 않는다.

## 6. 화면 전체 상태

- `HOST_DECISION_READY`: 바로 정할 수 있는 후보가 하나 이상
- `HOST_REVIEW_NEEDED`: 바로 정할 후보는 없고 확인 후 가능한 후보가 하나 이상
- `HOST_WAITING_PARTIAL`: 위 두 상태가 없고 응답이 필요한 후보가 하나 이상
- `HOST_RECOVERY_REQUIRED`: 모든 후보가 어렵거나 후보가 없음

화면 상단에는 가장 높은 상태의 결론과 행동 하나만 보여준다. 다른 상태의 후보는 비교 또는 대안으로 낮춘다.

## 7. 설명 계약

후보 상태는 다음 구조로 설명한다.

```text
결론
→ 가장 중요한 이유 한 문장
→ 영향을 주는 사람
→ 다음 행동
→ 필요할 때 상세 응답
```

내부 계산 용어인 `quorum`, `required_pending`, `rank`를 사용자에게 노출하지 않는다.

## 8. 재판정

응답 추가·수정, 주최자 시간대 확장, 마감 변경 후 모든 후보를 같은 규칙으로 다시 계산한다. 기존 응답은 후보 ID가 아니라 시간 구간의 가용 상태를 원본으로 유지한다.

## 9. 수용 예시

### 예시 A

- 최소 4명, 주최자 포함 `available` 4명, 일반 미응답 1명
- 결과: 바로 정할 수 있어요
- 미응답자는 보조 정보이며 응답 요청 CTA를 만들지 않는다.

### 예시 B

- 최소 4명, `available` 3명, `adjustable` 1명
- 결과: 정하기 전에 한 번 확인해 주세요
- 행동 대상: 해당 `adjustable` 참석자

### 예시 C

- 최소 4명, `available + adjustable` 3명, 일반 미응답 2명
- 결과: 응답이 더 필요해요
- 행동 대상: 최소 한 명의 응답이 필요하다는 사실과 영향 대상

### 예시 D

- 필수 참석자 1명이 `unavailable`
- 결과: 다른 시간을 찾아야 해요
- 다른 인원이 많이 가능해도 상태는 바뀌지 않는다.
