## 1. 전제 수정

P0에서 백엔드와 실제 외부 API는 의도적으로 제외한다. 그러나 이는 이상적인 연동 경험을 화면에서 제외한다는 뜻이 아니다. 채용 지원용 프로토타입은 더미 데이터로 캘린더 연결과 일정 반영을 시뮬레이션할 수 있다.

실제 계정 인증, 권한, 동기화, 실패 복구는 구현하지 않는다. 제품 화면에서는 연결된 캘린더와 반영 결과가 실제처럼 동작하며, 과제 문서와 README에서 시뮬레이션임을 밝힌다.

## 2. 결정 요약

참여자의 기본 흐름을 수동 기본 패턴 선택에서 캘린더 기반 자동 채움으로 바꾼다.

```text
회의 맥락 확인
→ 연결된 캘린더 일정 불러오기
→ 비어 있음과 일정 있음이 반영된 타임테이블 확인
→ 가능 / 일정 조정 / 어려움 브러시로 예외 수정
→ 응답 저장
```

- 비어 있는 시간은 `AVAILABLE` 초안으로 채운다.
- 일정이 있는 시간은 `UNAVAILABLE` 초안으로 채운다.
- 캘린더는 일정 이동 가능성을 판단하지 않는다. 참여자가 필요한 칸을 `ADJUSTMENT_COMMIT`으로 바꾼다.
- 자동 반영 결과는 제출 전 초안이며 주최자 판정에 들어가지 않는다.
- 일정 제목은 노출하거나 공유하지 않고 free/busy 의미만 사용한다.

## 3. 타임테이블 계약

후보 수가 적을 때 적합했던 긴 카드 목록은 넓은 조율 범위에서 사용하지 않는다.

- PC는 날짜를 가로축, 시간을 세로축으로 한 주간 타임테이블을 사용한다.
- 모바일은 날짜 스트립과 선택한 하루의 세로 타임테이블을 사용한다.
- 열린 조율 시간만 상태를 바꿀 수 있고 나머지 칸은 `조율 대상 아님`으로 표시한다.
- 상단 3상태 브러시를 먼저 고른 뒤 셀을 클릭·탭한다.
- PC는 마우스 드래그로 연속 칸을 칠할 수 있다.
- 모바일은 탭을 기본으로 하며 터치 드래그는 스크롤 충돌 검증 뒤 확장한다.
- 날짜 전체 적용과 저장 전 미선택 확인을 유지한다.
- `가능하면 피하기`는 핵심 상태와 분리된 보조 토글이다.

## 4. 직접 입력 fallback

`직접 입력하기`를 보조 경로로 유지한다.

- `대체로 가능해요`: 전체 열린 칸을 `AVAILABLE`로 채우고 예외를 수정한다.
- `가능한 시간이 적어요`: 전체 열린 칸을 `UNAVAILABLE`로 채우고 예외를 수정한다.

v2.5의 기본 패턴 모델은 폐기하지 않고 캘린더를 사용하지 않는 경우의 시작점으로 이동한다.

## 5. 역할 분리

주최자와 참여자는 시간 좌표와 페인트 인터랙션 기반을 공유할 수 있지만 의미는 분리한다.

- 주최자: 참석자에게 질문할 범위의 포함·제외 2상태
- 참여자: 자신의 참석 가능성에 대한 가능·조정·어려움 3상태

회의 확보 시간은 셀 크기나 브러시 크기가 아니다. 제품이 연속 칸을 후보 상태로 집계할 때 사용하는 길이 제약이다.

## 6. 검증 질문

- 참여자가 캘린더 일정 제목이 공유되지 않는다는 사실을 이해하는가?
- 자동으로 채워진 상태가 제출 전 초안임을 이해하는가?
- 일정이 있는 칸 중 이동 가능한 시간을 `일정 조정해요`로 바꿀 수 있는가?
- 5일 이상의 조율 범위에서도 10초 안에 예외 날짜와 연속 시간을 찾을 수 있는가?
- 모바일에서 날짜 전환, 상태 브러시, 저장 CTA를 놓치지 않는가?
