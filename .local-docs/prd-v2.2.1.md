# PRD v2.2.1 - MeetCue 결정 일관성 슬라이스

- 상태: 현재 구현 기준
- 기준일: 2026-07-12
- 제품 UI명: MeetCue
- 과거 기획명: 확정보드
- 구현 조건: 백엔드 없는 반응형 P0 프로토타입
- 최우선 범위 잠금: `p0-submission-scope-lock-v2.10.md`
- 상위 문서: `research-internalized-thinking-v1.md`
- 상세 판정 계약: `decision-model-v2.2.md`

## 1. 이번 버전에서 증명할 것

> 사람별 응답을 모아 보여주는 것보다, 지금 회의를 정할 수 있는지와 어떤 가능 응답 조건이 더 필요한지를 설명하는 결과가 주최자의 안전한 결정을 돕는가?

중심 사용자는 회의 주최자다. 생성과 참석자 응답은 결과를 신뢰할 수 있게 만드는 입력 과정이다.

완성 범위는 `회의 기준 설정 → 참석자 응답 → 주최자 판단 → 한 사람에게 재요청 → 응답 변경 후 재판정 → 확정`으로 제한한다.

제품이 보증하는 범위는 주최자가 설정한 필수 참석자와 최소 참석 인원 기준까지다. 회의 중요도나 입력한 기준의 업무 적절성은 보증하지 않는다.

## 2. P0 핵심 여정

```text
회의 맥락·참석자·진행 기준 설정
→ 주최자가 미래 날짜 범위와 확보 시간을 입력
→ 제품이 기본 탐색 범위를 만들고 주최자가 필요할 때만 예외를 편집
→ 참석자가 더미 캘린더의 free/busy 초안을 불러오고 30분 칸의 예외만 수정
→ 미선택 칸의 의미를 확인하고 제출
→ 제품이 선택된 확보 시간 길이의 후보를 계산
→ 주최자가 READY·PENDING·IMPOSSIBLE을 같은 결과 보드에서 비교
→ READY 확정 또는 PENDING 응답 대상 선택
→ 한 사람의 응답 추가·수정
→ 같은 결과 보드 재판정
```

대표 fixture는 같은 회사 동료 6명과 다음 주 1시간 회의다. 인원, 날짜, 확보 시간은 제품 제약이 아니다. 새 회의의 확보 시간은 과제 맥락에 맞춰 1시간을 기본값으로 제공하되, 주최자는 30분 단위로 변경할 수 있다.

## 3. 입력 계약

### 회의 생성

- 회의 이름, 주최자 또는 팀, 한 문장 목적은 필수다.
- 관련 자료는 선택이다.
- 참석자를 이름으로 선택한다.
- 전원 참석 또는 일부 불참 가능을 고른다.
- 일부 불참 가능이면 필수 참석자와 필요할 때 최소 참석 인원을 정한다.
- 주최자는 참석자·필수 참석자·최소 인원에 정확히 한 번 포함된다.
- 제품은 날짜 범위와 확보 시간을 바탕으로 기본 탐색 범위를 먼저 제안한다.
- P0 fixture의 기본 탐색 범위는 선택 날짜 안의 평일 09:00~18:00이며 제품의 보편 규칙이 아니다.
- 주최자는 빈 그리드를 필수로 칠하지 않고 제외할 시간과 추가할 시간만 편집한다.
- 별도 편집 없이 계속하면 현재 보이는 범위를 주최자가 참석 가능한 조율 범위로 확인한 것으로 본다.
- 확인된 주최자 조율 가능 시간 안에서만 후보를 만든다.
- 응답 마감은 운영 안내이며 후보 상태를 바꾸거나 입력을 잠그지 않는다.

### 참석자 응답

- 주최자가 지정한 참석자는 참여자별 fixture 라우트에서 자신의 응답 과업을 연다.
- 참여자는 fixture에서 이미 식별되며 이름을 다시 선택하지 않는다.
- 실제 계정·알림 전달은 구현하지 않고 DEV 페르소나 전환으로 응답 전후를 시연한다.
- 다른 참석자의 이름, 링크, 응답 상태는 참여자 화면에서 볼 수 없다.
- 편집기 상태는 `NOT_STARTED`, `DRAFT`, `SUBMITTED`다.
- 첫 시간 칸을 바꾸면 편집기가 `DRAFT`가 된다.
- 신규 응답의 `NOT_STARTED`와 `DRAFT`는 모든 후보에서 `UNKNOWN`이다.
- 제출본을 수정 중이면 마지막 `SUBMITTED` 응답은 새 초안을 저장할 때까지 판정에 유지한다.
- 시간 칸은 `UNSET`, `AVAILABLE`, `ADJUSTMENT_COMMIT`, `UNAVAILABLE`다.
- 신규 응답은 `대체로 가능해요` 또는 `가능한 시간이 적어요`를 먼저 선택한다.
- 제품은 선택한 패턴으로 열린 모든 칸을 채우고 참여자는 3상태 브러시로 예외만 수정한다.
- 날짜 전체 적용을 제공하며 각 칸에 세 버튼을 반복하지 않는다.
- 제출 확인 뒤에만 남은 `UNSET`을 `UNAVAILABLE`로 바꾼다.
- 제출 후 수정은 기존 범위 안에서만 지원한다.
- 늦은 응답도 저장하며 결과 화면에 마감 경과를 별도로 표시한다.
- 회의가 확정되면 응답 편집을 닫고 확정된 시간을 보여준다.
- 선호는 `가능하면 피하고 싶어요` 하나만 제공한다.

이번 P0은 더미 캘린더의 free/busy 자동 채움과 상태 브러시를 함께 사용해 의미 무결성과 입력 마찰을 검증한다. 실제 캘린더 API 연동은 제외하며, 캘린더를 쓰지 않는 경우에는 명시적 기본 패턴을 직접 입력 fallback으로 제공한다. 터치 드래그 연속 칠하기는 별도 입력 실험으로 분리한다.

## 4. 판정과 결과 계약

후보 판정은 `decision-model-v2.2.md`를 따른다.

- `READY`: 필수 참석자가 모두 확약했고 최소 인원을 충족한다.
- `PENDING`: 아직 가능 응답 조건이 남았지만 성립 가능성이 있다.
- `IMPOSSIBLE`: 필수 참석자가 어렵거나 모든 미응답자가 가능해도 최소 인원을 채울 수 없다.

모든 상태는 같은 결과 보드에서 다음 구조로 보인다.

```text
전체 결론
→ 상태별 실제 시작 시각 수
→ 상태별 후보군
→ 선택한 후보의 가장 중요한 이유
→ 필수 응답자와 대체 가능한 대상군
→ 사람별 판정 근거
→ 가능한 행동
```

상태별 상위 3개 후보군을 먼저 보여주고 나머지는 사용자가 펼칠 수 있어야 한다.

## 5. 행동 계약

### READY

- 주최자가 실제 시작 시각을 선택해 확정한다.
- 확정 안내에는 회의 제목, 날짜, 시작·종료 시각을 포함한다.
- 조정 확약자가 있으면 조정이 필요한 시간으로 확정됐음을 알린다.

### PENDING

- 필수 미응답자는 이름으로 구분한다.
- 일반 미응답자는 대체 가능한 대상군과 필요한 가능 응답 수로 구분한다.
- 주최자는 한 명 또는 여러 명을 직접 선택한다.
- 제품은 사람이나 연락 순서를 자동 추천하지 않는다.
- 선택한 계정에 응답 재요청을 보낸 것으로 처리한다.

### IMPOSSIBLE

- 다른 기존 후보를 검토한다.
- 모든 후보가 IMPOSSIBLE이면 현재 회의 기준을 다시 검토할 수 있다.
- P0에서는 새 시간 범위를 열지 않는다.

## 6. 참석자에게 보이는 정보

응답 전에 다음을 확인한다.

- 회의 이름
- 주최자 또는 팀
- 한 문장 목적
- 날짜 범위와 선택된 확보 시간
- 응답 마감
- 주최자가 응답을 모아 최종 시간을 정한다는 사실
- 제출 후에도 같은 링크에서 수정할 수 있다는 사실

필수 참석자 여부, 최소 참석 인원, 다른 사람의 응답, 후보 판정은 보이지 않는다.

## 7. 완전히 작동해야 하는 범위

- 회의 생성과 기준 정규화
- 기본 탐색 범위 생성, 주최자 예외 편집과 선택된 확보 시간 길이의 후보 생성
- 더미 캘린더 일정 반영, 참여자 타임테이블 검토와 30분 단위 예외 편집, 초안, 제출, 수정
- READY, PENDING, IMPOSSIBLE 판정과 후보군
- 필수 특정인과 대체 가능한 대상군 구분
- 필요한 가능 응답 수
- 사람별 상세 근거
- 응답 대상 직접 선택과 재요청 처리
- READY 확정과 안내 미리보기
- 한 사람의 응답 추가·수정 후 재판정
- DEV 페르소나 전환과 현재 회의·선택 후보를 보존한 재판정
- PC와 모바일 핵심 흐름

## 8. 제외 범위

- 최종 제품 네이밍 결정
- 공유 후 새 시간 범위 추가와 범위 버전 관리
- 순차 범위 입력과 상태 도구 입력의 A/B 실험
- 실제 알림·이메일 발송, 계정 인증, 권한, 서버 저장
- 계정 홈, 내 회의, 받은 요청, 알림과 여러 회의 재진입
- 실제 계정 인증과 Google·Outlook 캘린더 API 연동, 회의실, 장소, 화상회의 링크
- 마감에 따른 자동 잠금과 자동 리마인드
- 30분보다 작은 확보 시간 단위

## 9. 수용 기준

- [ ] READY가 없고 PENDING만 있어도 동일한 결과 보드를 본다.
- [ ] 초안 응답은 판정과 상세에서 모두 미응답이다.
- [ ] 제출 뒤에만 미선택 칸이 어려움으로 저장된다.
- [ ] 필수 응답자와 대체 가능한 대상군이 섞이지 않는다.
- [ ] 주최자가 요청 대상을 직접 선택한다.
- [ ] 숨겨진 후보군을 펼칠 수 있다.
- [ ] 마감 경과가 후보 상태를 바꾸지 않는다.
- [ ] 모든 후보가 IMPOSSIBLE이어도 새 시간 확장 흐름으로 보내지 않는다.
- [ ] 응답 수정 뒤 같은 결과 화면이 재판정된다.
- [ ] 모든 날짜 계산과 표시는 `Asia/Seoul` 기준이다.
- [ ] 주최자는 시간 칸을 직접 만들지 않고도 제안된 탐색 범위를 확인하고 계속할 수 있다.
- [ ] 제외·추가 편집 뒤 남은 범위에서만 후보가 생성된다.
- [ ] 신규 참여자는 기본 패턴 한 번으로 열린 모든 칸의 상태를 만들 수 있다.
- [ ] 신규 참여자는 더미 캘린더에서 free/busy 상태를 불러오고 제출 전에 검토할 수 있다.
- [ ] 참여자는 상태 브러시와 날짜 전체 적용으로 예외를 수정할 수 있다.
- [ ] 참여자는 DEV fixture에서 이름 선택 없이 자신의 응답 화면을 연다.
- [ ] 주최자 제품 화면에는 참여자별 토큰이나 개인 링크가 노출되지 않는다.
- [ ] PENDING 후보와 선택 후보를 유지한 채 참여자 응답 화면으로 전환할 수 있다.
- [ ] 참여자 제출 뒤 같은 결과 보드에서 선택 후보가 다시 판정된다.
- [ ] 평가용 기본 진입과 핵심 화면에 계정 홈·목록·알림 메뉴가 노출되지 않는다.
- [ ] 대표 도메인 시나리오와 경계 조건 테스트가 통과한다.

## 10. 검증 과제

업무 회의를 잡아본 사용자에게 결과 화면을 보여주고 다음을 자신의 말로 설명하게 한다.

1. 지금 확정할 수 있는가?
2. 왜 그런가?
3. 반드시 필요한 응답은 누구의 어떤 응답인가?
4. 대체 가능한 대상군에서는 몇 명이 가능해야 하는가?
5. 지금 누를 행동은 무엇인가?

5명 중 4명 이상이 다섯 질문에 정확히 답하면 이해 가능성 가설을 통과한 것으로 본다. 시장성이나 반복 사용성을 입증한 것으로 해석하지 않는다.
