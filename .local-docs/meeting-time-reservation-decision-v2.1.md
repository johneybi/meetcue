# 회의 종료와 일정 확보 시간 결정 기록 v2.1

- 상태: 채택
- 결정일: 2026-07-10
- 적용 범위: 확정보드 P0
- 선행 결정: `.local-docs/host-time-model-decision-v2.0.md`
- 실행 명세: `UX-ARCHITECTURE.md`
- 제품 명세: `.local-docs/prd-v2.0.md`

> 후속 결정: 이 문서의 5분 단위 직접 입력과 45분 프리셋은 `.local-docs/time-quantum-decision-v2.2.md`의 30분 시간 최소 단위 결정으로 대체되었다. 일정 확보 시간의 의미는 유지된다.

> 시간 입력 후속 결정: `.local-docs/availability-window-model-decision-v2.3.md`는 일정 확보 시간을 유지하되, 주최자가 정확한 시작 후보를 직접 고르는 대신 주최자와 참석자의 가용시간에서 후보를 계산한다.

## 1. 문제 제기

시간 단계에서 `회의는 얼마나 걸리나요?`라고 묻고 시작 시각과 종료 시각이 정해진 후보를 만드는 방향을 채택했다. 그러나 실제 업무 회의는 시작 시각은 정해도 안건이 끝나는 시점까지 정확히 예측하지 못하는 경우가 있다.

이 문제를 그대로 받아들이면 두 주장이 충돌한다.

- 실제 회의의 종료 시각은 불확실할 수 있다.
- 참석자가 후보 시간에 참여할 수 있는지 판단하려면 일정에서 얼마를 비워야 하는지 알아야 한다.

따라서 `회의가 실제로 끝나는 시각`과 `일정 조율을 위해 확보할 시간의 끝`을 같은 값으로 취급해도 되는지 검토했다.

## 2. 먼저 분리해야 하는 개념

### 실제 소요 시간

회의를 시작한 뒤 실제로 진행된 시간이다. 안건 수, 논의 속도, 추가 쟁점에 따라 달라지며 생성 시점에는 확정할 수 없다.

### 일정 확보 시간

참석자가 다른 일정을 잡지 않고 회의를 위해 비워두기로 요청받는 시간 블록이다. 후보의 참석 가능 여부와 일정 충돌을 판단하려면 생성 시점에 유한한 값이어야 한다.

### 실제 종료 시각

회의가 현실에서 끝난 시각이다. 일정 확보 시간보다 이르거나 늦을 수 있으며 P0의 조율 판정 대상이 아니다.

## 3. 핵심 반례

주최자가 `14:00 시작, 종료 미정`인 후보를 보냈고 참여자에게 15:00 일정이 있다고 가정한다.

- 참여자는 14:00에 비어 있지만 회의 전체에 참석 가능한지는 판단할 수 없다.
- `가능해요`라고 답하면 15:00 일정과 충돌할 수 있다.
- `어려워요`라고 답하면 실제로는 14:30에 끝날 수 있는 후보를 불필요하게 제외한다.
- 제품은 같은 응답을 참석 가능과 부분 참석 중 무엇으로 해석해야 할지 결정할 수 없다.

종료 시각을 전혀 받지 않는 방식은 현실의 불확실성을 정직하게 표현하지만, 이 제품이 해결하려는 후보 비교와 참석 가능 판정을 약화한다.

## 4. 검토한 대안

| 대안 | 장점 | 문제 |
| --- | --- | --- |
| 정확한 회의 길이를 입력 | 후보와 캘린더 블록이 명확하다 | 실제 종료를 보장하는 것처럼 보이며 거짓 정밀도를 만든다 |
| 종료 시각을 받지 않음 | 실제 회의의 불확실성을 그대로 표현한다 | 참여자가 충돌을 판단할 수 없고 후보 판정이 불가능해진다 |
| 최소~최대 길이 범위를 입력 | 불확실성을 일부 표현할 수 있다 | 참석 가능 판정에는 결국 최댓값이 필요하며 입력과 응답 의미가 복잡해진다 |
| 일정에서 확보할 시간을 입력 | 후보 충돌을 판단할 수 있고 실제 종료를 약속하지 않는다 | 확보 시간과 실제 소요 시간이 다를 수 있음을 언어로 명확히 해야 한다 |

## 5. 범위 입력을 채택하지 않은 이유

`30~60분`처럼 범위를 받더라도 참여자가 30분만 가능한 경우의 응답 의미가 모호하다.

- 최대 60분 전체에 참석해야 `가능해요`라면 판정에는 60분만 필요하다.
- 30분만 참석해도 `가능해요`라면 부분 참석이라는 새로운 응답 상태가 필요하다.
- 주최자가 실제 종료 시각을 나중에 정해야 한다면 확정 단계가 다시 복잡해진다.

P0은 부분 참석과 후보별 가변 길이를 다루지 않는다. 따라서 범위보다 하나의 유한한 확보 시간으로 단순화한다.

## 6. 결정

> P0에서 주최자가 정하는 값은 회의의 정확한 실제 소요 시간이 아니라, 참여자가 해당 회의를 위해 일정에서 확보해야 하는 시간이다.

- 시작 시각과 확보 시간으로 후보의 시간 블록을 만든다.
- 참여자는 전체 시간 블록을 기준으로 가능 여부를 답한다.
- 확보 시간이 60분이라고 해서 회의가 정확히 60분에 끝난다고 약속하지 않는다.
- 실제 회의는 일찍 끝나거나 길어질 수 있지만, 초과 가능성 자체는 P0의 후보 판정에 사용하지 않는다.
- 종료 시각이 전혀 없는 후보는 P0에서 허용하지 않는다.

## 7. 언어 계약

주최자 입력 질문은 다음처럼 쓴다.

```text
참석자들이 얼마 동안 시간을 비워두면 될까요?
```

필요할 때만 다음 보조 설명을 사용한다.

```text
실제 종료 시각이 아니라, 일정에서 확보할 시간을 정해요.
```

화면별 권장 용어는 다음과 같다.

| 위치 | 용어 |
| --- | --- |
| 주최자 입력 | 시간을 비워둘 기간 / 확보할 시간 |
| 접힌 조건 요약 | 1시간 확보 |
| 후보 목록 | 14:00~15:00 |
| 참여자 초대 요약 | 1시간을 비워주세요 |
| 최종 검토 | 확보할 시간 |
| 내부 문서와 코드 | `reservedDurationMinutes` |

`회의 길이`, `예상 소요 시간`, `종료 시각`은 실제 종료를 암시할 수 있으므로 핵심 입력 라벨로 사용하지 않는다.

## 8. 입력 계약

> 아래 입력 단위 중 5분 직접 입력과 45분 프리셋은 v2.2에서 대체되었다. 사고 이력을 위해 원래 결정을 보존한다.

- 기본값을 미리 선택하지 않는다.
- 30분, 45분, 60분, 90분 프리셋은 빠른 입력 수단이다.
- 15~240분 범위에서 5분 단위 직접 입력을 허용한다.
- 모든 후보는 같은 확보 시간을 사용한다.
- 확보 시간을 정하기 전에는 후보 선택을 열지 않는다.
- 확보 시간을 바꾸면 기존 후보의 종료 경계를 같은 시작 시각 기준으로 다시 계산한다.

## 9. 후보와 응답 계약

- 후보의 `startAt`은 제안할 시작 시각이다.
- 후보의 종료 경계는 `startAt + reservedDurationMinutes`로 계산한다.
- 참여자의 `가능해요`는 해당 시간 블록 전체를 비울 수 있다는 뜻이다.
- 일부 시간만 가능한 경우 P0에서는 `일정 조정하면 가능해요` 또는 `어려워요`로 답한다.
- 주최자가 후보를 골랐다는 것은 주최자도 같은 시간 블록을 확보할 수 있다는 뜻이다.

## 10. 데이터 모델 영향

현재 코드의 `durationMinutes`는 의미가 넓어 실제 회의 소요 시간으로 오해될 수 있다. 후속 구현에서 다음 이름으로 바꾼다.

```ts
reservedDurationMinutes: number | null
```

후보의 `endAt`은 실제 종료 예측값이 아니라 예약된 시간 블록의 끝이다. 백엔드나 캘린더 연동이 생기기 전까지 별도 실제 종료 값은 저장하지 않는다.

## 11. 종료 미정 회의의 후속 방향

장애 대응, 워크숍, 장시간 의사결정처럼 종료가 특히 유동적인 회의가 실제 검증에서 중요하게 나타나면 다음 계약을 별도 범위로 검토한다.

```text
14:00 시작
최소 1시간은 비워주세요
종료 시각은 유동적이에요
```

이 경우에도 최소 확보 시간은 필수다. 추가로 `flexibleEnd`, 부분 참석, 캘린더 저장 방식, 초과 충돌 경고를 함께 설계해야 하므로 P0에는 포함하지 않는다.

## 12. 기존 결정과의 관계

이 문서는 `.local-docs/host-time-model-decision-v2.0.md`의 `회의 길이` 의미만 구체화하고 대체한다.

유지되는 결정은 다음과 같다.

- 회의 가능 기간과 확보 시간을 먼저 정한 뒤 후보를 선택한다.
- 모든 후보에 같은 길이를 적용한다.
- 프리셋과 직접 입력을 함께 제공한다.
- 후보 선택 후 응답 마감을 정한다.

변경되는 결정은 다음과 같다.

- `회의 길이`를 정확한 실제 소요 시간이 아니라 `일정 확보 시간`으로 해석한다.
- UI의 핵심 질문을 `얼마나 진행하나요?`에서 `얼마 동안 시간을 비워두면 될까요?`로 바꾼다.
- 내부 모델 이름을 `reservedDurationMinutes`로 명확히 할 후속 구현 과제를 만든다.

## 13. 검증 가설

- 주최자는 실제 종료 시각을 확정하지 않아도 확보할 시간은 선택할 수 있다.
- 참여자는 후보의 시작과 끝을 자신이 비워야 할 시간 블록으로 이해한다.
- 사용자는 `1시간 확보`를 `정확히 1시간 뒤 종료`라는 약속으로 해석하지 않는다.
- 종료 미정 옵션이 없어도 일반적인 업무 회의 생성에 막힘이 없다.
- 종료가 유동적인 회의에서 별도 옵션 요구가 반복되는지 확인할 수 있다.

## 14. 검증 질문

- `1시간을 비워주세요`를 보면 무엇을 약속한 것으로 이해하는가?
- 실제 회의가 40분에 끝나거나 70분까지 이어질 수 있다고 생각하는가?
- 종료를 예측하기 어려운 회의에서도 확보할 시간을 고를 수 있는가?
- `종료 미정`과 `최소 1시간 확보` 중 어떤 정보가 응답하기 쉬운가?
- 15:00에 다른 일정이 있을 때 `14:00 시작, 1시간 확보` 후보에 자신 있게 답할 수 있는가?
