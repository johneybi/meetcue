업무 회의 가용시간 수집과 확정 판단 도구

- 상태: P0 실행 기준

- 대체 문서: `.local-docs/prd-v1.5.md`

- UX 실행 명세: `UX-ARCHITECTURE.md`

## 0. 한 줄 정의

확정보드는 업무 회의에 참여할 수 있는 시간대를 모아 실제 후보를 계산하고, 현재 응답만으로 회의를 확정해도 되는지와 다음 행동을 주최자에게 알려주는 회의 시간 결정 보드다.

## 1. 제품 방향

제품은 가장 많은 사람이 선택한 시간을 보여주는 후보 투표 도구가 아니다. 다음 조건을 함께 해석해 실제로 확정 가능한 시간을 찾는다.

- 꼭 참석해야 하는 사람이 가능한가
- 최소 참석 인원을 충족하는가
- 일정 조정이 필요한 사람이 적은가
- 미응답으로 인한 불확실성이 얼마나 남았는가
- 확정 후 다시 바뀔 가능성이 낮은가

사용자가 알고 있는 정보는 `화요일 오전은 가능하다` 같은 시간대다. 정확한 시작 시각을 반복해서 만드는 계산은 제품이 맡는다.

제품 원칙은 다음과 같다.

> 응답은 가볍게, 해석은 제품이, 결정은 주최자가.

## 2. 사용자와 역할

### 주최자

- P0에서는 항상 회의에 참석한다.
- 회의 목적, 참석자, 회의 가능 기간, 일정 확보 시간을 정한다.
- 자신이 회의를 진행할 수 있는 조율 가능 시간대를 연다.
- 제품이 계산한 후보와 판정 근거를 확인한다.
- 최종 회의 시간을 직접 확정하고 공유한다.

### 참석자

- 회의 맥락과 확보할 시간을 확인한다.
- 주최자가 연 범위 안에서 자신의 가용 상태를 표시한다.
- 마감 전까지 같은 링크에서 응답을 수정한다.
- 다른 참석자의 응답, 필수 여부, 내부 판정 기준은 보지 않는다.

대리 조율자와 주최자가 다른 역할은 P0에서 제외한다.

## 3. 핵심 시간 모델

시간 정보는 다음 순서로 결정한다.

```text
회의 가능 날짜 범위
→ 일정 확보 시간
→ 주최자 조율 가능 시간대
→ 참석자 가용시간
→ 파생 후보
→ 응답 마감과 최종 결정
```

- 시간 최소 단위는 30분이다.
- 일정 확보 시간은 30분의 양의 배수다.
- 주최자 시간대는 확보 시간 이상이어야 한다.
- 한 날짜에 여러 시간대를 열 수 있다.
- 맞닿고 상태가 같은 시간대는 하나의 구간으로 합친다.
- 후보는 조율 시간대 안에서 30분 간격으로 시작점을 이동해 계산한다.
- 후보끼리 겹칠 수 있으며 오류로 취급하지 않는다.
- 실제 회의 종료 시각과 참석자가 일정에서 확보할 시간은 구분한다.

예를 들어 09:00-12:00 시간대와 90분 확보 시간이 있으면 09:00, 09:30, 10:00, 10:30 시작 후보가 계산된다.

## 4. 참석자 응답 계약

각 30분 칸의 상태는 다음과 같다.

- `가능해요`: 별도 일정 조정 없이 비울 수 있다.
- `내 일정 조정`: 앞뒤 일정이나 개인 계획을 조정하면 비울 수 있다.
- `어려워요`: 해당 시간을 비울 수 없다.
- `미응답`: 아직 상태를 저장하지 않았거나 새로 열린 시간대다.

참석자가 저장하면 표시하지 않은 기존 칸은 `어려워요`로 확정한다. 이후 주최자가 시간대를 확장하면 새로 열린 칸만 `미응답`이 된다.

후보별 참석자 상태는 후보가 차지하는 모든 칸에서 계산한다.

```text
어려운 칸이 하나라도 있음       → 어려워요
어려운 칸은 없고 미응답이 있음   → 미응답
모든 칸이 가능함                 → 가능해요
조정 가능 칸이 하나 이상 있음    → 내 일정 조정
```

## 5. 주최자 판단 계약

후보는 다음 순서로 평가한다.

1. 회의 길이와 날짜가 유효한가
2. 필수 참석자 중 어려운 사람이 있는가
3. 필수 참석자의 미응답이 남았는가
4. 최소 참석 인원을 충족하는가
5. 조정 가능한 응답이 확정 리스크를 만드는가
6. 일반 참석자의 미응답과 마감 상태는 어떤가

후보 상태는 `바로 정할 수 있어요`, `정하기 전에 확인`, `필수 응답 대기`, `이 시간은 어려워요`로 표현한다. 제품은 자동 확정하지 않는다.

## 6. 조율 회복

확정 가능한 후보가 없더라도 새 요청을 만들게 하지 않는다.

- 기존 링크를 유지한다.
- 기존 참석자와 가용시간 응답을 유지한다.
- 필요한 사람에게 수정 또는 응답을 요청한다.
- 주최자의 조율 가능 시간대나 회의 날짜 범위를 넓힌다.
- 새로 열린 시간대만 미응답으로 요청한다.
- 필요하면 일정 확보 시간을 재검토한다.

조율 실패는 예외가 아니라 정상 흐름이다. 제품 밖의 메신저에서 조율을 처음부터 반복하지 않게 하는 것이 P0 Support의 목적이다.

## 7. P0 범위

### 포함

- 회의 맥락과 참석자 조건 입력
- 30분 단위 일정 확보 시간
- 주최자 조율 가능 시간대 생성, 수정, 삭제
- 참석자 3상태 가용시간 입력과 수정
- 가용시간 기반 정확한 후보 계산
- 필수 참석자, 최소 인원, 조정 필요, 미응답 판정
- 주최자 수동 확정
- 기존 링크와 응답을 유지하는 시간대 확장
- 확정 및 재요청 문구 복사

### 제외

- Google 또는 Outlook 캘린더 연동
- free/busy 자동 수집
- AI 기반 후보 추천이나 참석 가능성 예측
- 자동 응답과 자동 확정
- 자동 알림 발송
- 반복 회의와 회의실 예약
- 대리 조율자 역할
- 30분보다 작은 시간 단위

## 8. 데이터 원칙

- `availabilityWindows`가 주최자와 참석자의 시간 입력 원본이다.
- `candidates`는 주최자 시간대와 일정 확보 시간에서 파생한다.
- 후보별 `responses`는 참석자의 연속 가용시간에서 파생해 판정 함수에 제공한다.
- 후보 ID는 시작 시각과 확보 시간을 기준으로 안정적으로 만든다.
- 최종 확정 시각은 파생 후보 중 주최자가 선택한 결과로 저장한다.
- 시간대 확장 시 기존 구간과 응답을 삭제하지 않는다.

## 9. 성공 지표와 검증 가설

핵심 결과 지표는 기존 링크 안에서 회의가 확정된 비율이다.

보조 지표:

- 생성부터 확정까지 걸린 시간
- 참석자 응답 완료율
- 재시작 없이 확정까지 이어진 비율
- 확정 후 변경 요청률
- 일정 조정 응답이 포함된 후보의 실제 확정률

프로토타입에서 우선 검증할 가설:

- 주최자는 정확한 후보 여러 개보다 연속 시간대를 더 빠르게 입력한다.
- 참석자는 후보별 반복 응답보다 가용시간 한 번 입력을 더 가볍게 느낀다.
- 참석자는 가능한 모든 칸과 선호하는 칸을 구분한다.
- 미선택 칸이 저장 시 `어려워요`가 된다는 규칙을 이해한다.
- 파생 후보를 제품이 임의로 추천한 시간으로 오해하지 않는다.
- 시간대 확장 후 기존 응답이 유지된다는 사실을 이해한다.

## 10. 완료 기준

- 주최자가 정확한 후보 시작 시각을 직접 나열하지 않고 요청을 만들 수 있다.
- 참석자가 PC와 모바일에서 모든 시간 칸에 키보드, 클릭 또는 탭으로 응답할 수 있다.
- 파생 후보가 일정 확보 시간 전체의 가용 상태를 반영한다.
- 필수 참석자와 최소 인원 판정이 기존 의미를 유지한다.
- 시간대를 확장해도 기존 링크와 응답이 유지된다.
- 주최자가 판정 근거를 보고 수동으로 시간을 확정할 수 있다.
- `npm run check`, `npm run lint`, `npm run build`가 통과한다.
