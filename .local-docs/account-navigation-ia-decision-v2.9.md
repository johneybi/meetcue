# 계정 내비게이션과 재진입 IA 결정 v2.9

- 상태: 현재 구현 기준
- 기준일: 2026-07-12
- 상위 계약: `prd-v2.2.1.md`, `account-based-request-delivery-decision-v2.8.md`
- 적용 범위: 백엔드 없는 조직 계정형 프로토타입

## 1. 문제

v2.8에서 회의 요청은 개인 링크가 아니라 조직 계정의 과업이 됐다. 그러나 제품의 진입점이 단일 회의 생성에 머물면 사용자는 다음 질문에 답할 수 없다.

- 내가 응답해야 할 요청은 어디에 있는가?
- 내가 만든 진행 중 회의는 어디에서 다시 여는가?
- 새 회의는 어디에서 만드는가?
- 새 응답이나 확정 같은 변화는 어디에서 확인하는가?

회의 내부 화면만 연결하는 것은 데모 플로우일 뿐, 반복 사용 가능한 계정형 서비스 구조가 아니다.

## 2. 결정

제품의 기본 진입점을 계정 홈으로 변경한다.

```text
홈
├─ 내가 답할 요청 → 참여자 응답 과업
├─ 내가 만든 회의 → 주최자 상태·결과
└─ 새 회의 → 생성 흐름

내 회의 → 내가 만든 회의 목록과 재진입
받은 요청 → 내 계정에 할당된 응답 과업
알림 → 상태 변화 이벤트와 해당 과업의 깊은 링크
```

전역 내비게이션의 항목은 `홈`, `내 회의`, `받은 요청`으로 제한한다. `알림`은 별도 아이콘으로 제공하고, `새 회의`는 탐색 항목이 아니라 전역 생성 행동으로 둔다.

## 3. 정보 우선순위

홈은 기능 카탈로그나 통계 대시보드가 아니다. 사용자가 지금 처리할 수 있는 항목을 우선순위대로 보여준다.

1. 응답이 필요한 받은 요청
2. 새 응답이 들어왔거나 결정이 필요한 내가 만든 회의
3. 확정된 회의와 낮은 우선순위 기록

각 행은 상태, 회의 제목, 최소 맥락, 하나의 다음 행동만 가진다. 같은 회의의 상세 근거나 모든 참석자 상태를 홈에서 반복하지 않는다.

## 4. 알림과 목록의 역할 구분

- 받은 요청: 현재 내게 할당된 응답 과업의 기준 목록
- 내 회의: 내가 주최한 회의의 기준 목록
- 알림: 새 요청, 새 응답, 확정처럼 무엇이 바뀌었는지 알려주는 이벤트 피드

알림은 목록을 대체하지 않는다. 알림을 읽은 뒤에도 과업은 받은 요청이나 내 회의에서 다시 찾을 수 있어야 한다.

## 5. 집중 화면과 전역 내비게이션

회의 생성과 참석자 응답은 입력 집중도가 높은 과업이므로 전역 메뉴 전체를 반복 노출하지 않는다.

- 생성 화면: 회의 컨텍스트 바의 브랜드를 누르면 홈으로 돌아간다.
- 참석자 응답: 간결한 헤더와 `받은 요청` 복귀 행동을 제공한다.
- 결과·대기·확정 화면: 회의 컨텍스트를 유지하고 브랜드를 누르면 홈으로 돌아간다.

따라서 계정 셸과 집중 과업 셸은 시각적으로 다르지만, 모든 화면에서 계정 수준으로 복귀할 수 있다.

## 6. 반응형 계약

- PC: 상단에 브랜드, `홈 / 내 회의 / 받은 요청`, 알림, `새 회의`를 둔다.
- 모바일: 상단에는 브랜드와 알림·새 회의만 두고, 하단 고정 내비게이션에 세 주요 목적지를 둔다.
- 모바일 하단 내비게이션은 생성·응답 집중 화면에는 중첩하지 않는다.
- 본문은 PC에서 읽기 폭을 제한하되, 상단 앱 바는 더 넓은 서비스 프레임을 사용한다.

### 전역 프레임 계약

- 계정 홈과 주최자 과업 화면은 같은 전역 상단 바를 사용한다. 라우트가 바뀌어도 브랜드, 메뉴, 알림, 새 회의의 높이와 정렬축은 움직이지 않는다.
- 상단 바 내부 프레임은 최대 `1180px`이다.
- 일반 본문의 실제 콘텐츠 폭은 최대 `960px`이며, 데스크톱 좌우 거터는 `24px`, 모바일 거터는 `18px`이다.
- 생성 화면의 회의 제목과 상태는 별도의 떠 있는 앱 바가 아니라 본문 컨텍스트 헤더로 표시한다.
- 결과 보드처럼 비교 폭이 필요한 화면만 최대 `1120px`의 wide 변형을 사용한다. 전역 상단 바와 배경은 그대로 유지한다.
- 생성·응답 같은 집중 과업에서 모바일 하단 내비게이션을 숨길 수 있지만, 동일한 상단 브랜드와 계정 복귀 경로는 유지한다.
- 짧은 화면과 긴 화면 사이에서 세로 스크롤바 유무로 중앙축이 움직이지 않도록 안정적인 스크롤바 거터를 예약한다.

## 7. 프로토타입 데이터 계약

실제 계정·서버 없이도 계정형 재진입 경험을 시연한다.

- 받은 요청의 대표 항목은 `token-p-sujin` 페르소나로 연결한다.
- 내가 만든 회의는 현재 대표 회의 fixture의 주최자 화면으로 연결한다.
- 알림은 같은 fixture의 요청·응답 변화 이벤트를 보여준다.
- 참여자 토큰은 화면에 노출하지 않고 DEV 라우팅에서만 사용한다.

## 8. 수용 기준

- 앱의 기본 URL은 계정 홈으로 진입한다.
- 홈에서 받은 요청 응답, 진행 중 회의 확인, 새 회의 생성을 시작할 수 있다.
- 내 회의와 받은 요청에서 각 과업으로 재진입할 수 있다.
- 알림에서 관련 요청이나 회의로 이동할 수 있다.
- 참여자 응답 화면에서 받은 요청 목록으로 돌아갈 수 있다.
- 주최자 내부 화면에서 계정 홈으로 돌아갈 수 있다.
- PC와 모바일에서 동일한 목적지를 제공하되 내비게이션 위치는 각 기기 사용성에 맞춘다.

## 9. 채용 과제용 대표 계정 시나리오

메뉴가 비어 보이는 문제를 기능 카드나 무관한 샘플 데이터로 채우지 않는다. 한 사용자의 하루 안에서 서로 다른 상태를 보여주는 다음 회의 세트를 사용한다.

| 회의                   | 사용자 역할 | 현재 상태                | 대표 행동      |
| ---------------------- | ----------- | ------------------------ | -------------- |
| 다음 주 제품 리뷰 회의 | 주최자      | 4/5명 응답, 새 응답 도착 | 결과 확인      |
| 온보딩 개선안 논의     | 참석자      | 오늘 마감, 응답 전       | 응답하기       |
| 3분기 목표 점검        | 주최자      | 시간 확정                | 확정 내용 확인 |
| 디자인 QA 기준 정리    | 참석자      | 응답 제출 완료           | 응답 확인·수정 |

표현 원칙:

- 홈은 위 시나리오 중 지금 행동 가능한 두 건과 가장 가까운 확정 회의만 보여준다.
- 내 회의는 `진행 중 / 확정`, 받은 요청은 `응답 필요 / 응답 완료` 상태를 전환할 수 있다.
- 알림은 같은 시나리오의 요청 도착, 새 응답, 확정, 저장, 마감 임박 이벤트를 시간순으로 보여준다.
- 데이터 밀도를 높여도 한 행에는 상태, 제목, 최소 맥락, 하나의 다음 행동만 둔다.
- 대표 항목은 실제 상세 흐름으로 연결한다. 화면 수를 채우기 위한 동작하지 않는 카드는 만들지 않는다.
