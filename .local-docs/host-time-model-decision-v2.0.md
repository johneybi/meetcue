# 주최자 역할과 시간 모델 결정 기록 v2.0

- 상태: 채택
- 결정일: 2026-07-10
- 적용 범위: 확정보드 P0
- 실행 명세: `UX-ARCHITECTURE.md`
- 제품 명세: `.local-docs/prd-v2.0.md`

> 후속 결정: `.local-docs/availability-window-model-decision-v2.3.md`가 주최자의 정확한 후보 직접 선택과 참석자의 후보별 응답 계약을 대체한다. 주최자가 항상 참석하고 최종 결정을 내린다는 역할 계약은 유지된다.

## 1. 결정 배경

회의 생성 흐름을 설계하면서 다음 개념이 섞여 있었다.

- 요청을 만드는 사람
- 회의에 참석하는 사람
- 후보 시간을 제안하는 사람
- 응답을 해석하고 최종 시간을 결정하는 사람
- 다른 사람을 대신해 일정만 조율하는 사람

이 역할을 모두 `주최자`로 부르면서 후보 시간 선택의 의미도 흔들렸다. 주최자가 회의에 참석하지 않을 수 있다면 후보 선택은 주최자의 가능 시간이 아니다. 반대로 주최자가 최종 시간을 결정하고 회의를 주관한다면 참석 여부를 다시 묻는 것도 부자연스럽다.

시간 단계에서도 다음 값이 구분되지 않았다.

- 회의를 실시할 수 있는 기간
- 회의가 차지하는 길이
- 주최자가 참석자에게 제안할 구체적인 후보
- 참석자가 답해야 하는 마감

## 2. 검토한 대안

### A. 주최자는 항상 참석한다

- 주최자가 고른 후보를 주최자가 참석 가능한 시간으로 해석할 수 있다.
- 후보 제안, 결과 판단, 최종 결정의 책임이 한 역할에 모인다.
- P0의 상태와 판정 규칙이 단순해진다.

### B. 주최자가 참석 여부를 선택한다

- `나도 참석해요`를 기본 선택으로 두고 해제할 수 있다.
- 대리 조율을 일부 지원할 수 있다.
- 요청자, 실제 회의 주최자, 최종 결정자가 누구인지 다시 정의해야 한다.

### C. 일정 조율 담당자를 별도 역할로 둔다

- 실제 업무의 대리 조율을 가장 정확하게 표현한다.
- 회의 주최자 지정, 권한 이전, 참석자 집계, 최종 결정 권한이 추가로 필요하다.
- P0의 핵심인 응답 해석과 조율 회복 검증을 흐린다.

## 3. 결정

P0에서는 A를 선택한다.

> 주최자는 회의에 참석하며, 후보 시간을 제안하고, 참석자의 응답을 확인한 뒤 최종 회의 시간을 결정한다.

대리 일정 조율은 별도 역할이 필요한 후속 범위로 둔다. P0에서 주최자의 참석 여부를 묻지 않는다.

## 4. 역할 계약

- 주최자는 항상 회의 참석자다.
- 주최자는 전체 참석 인원과 회의 성립 인원에 포함한다.
- 초대할 사람 수에는 주최자를 포함하지 않는다. 이 값은 링크를 받을 사람 수다.
- 주최자가 고른 후보는 모두 주최자가 참석 가능한 시간이다.
- 주최자에게 응답 링크를 보내지 않는다.
- 주최자는 후보별 미응답자로 계산하지 않는다.
- 꼭 필요한 사람을 고를 때 주최자를 다시 선택하게 하지 않는다.
- `나도 참석해요` 선택은 P0에서 제공하지 않는다.

주최자가 참석한다는 사실을 가짜 `Response` 레코드로 저장하지 않는다. 후보 선택과 주최자 역할에서 파생되는 암묵적 가능 상태로 계산한다.

## 5. 시간 계약

### 회의 가능 기간

회의를 실제로 실시할 수 있는 시작일과 종료일이다. 주최자의 개인 가용시간을 뜻하지 않는다.

### 회의 길이

> 이 절의 `회의 길이` 의미는 `.local-docs/meeting-time-reservation-decision-v2.1.md`에서 `일정 확보 시간`으로 구체화되었다. 실제 회의의 정확한 종료 시간을 뜻하지 않는다. 입력 가능 단위와 프리셋은 `.local-docs/time-quantum-decision-v2.2.md`의 30분 단위 결정으로 대체되었다.

후보 하나가 차지하는 시간이다. P0은 자주 쓰는 길이를 빠르게 고를 수 있도록 30분, 45분, 60분, 90분 프리셋을 제공한다. 프리셋에 맞지 않는 회의는 15~240분 범위에서 5분 단위로 직접 입력할 수 있다.

프리셋은 입력을 빠르게 만드는 장치이지 허용 가능한 회의 길이의 목록이 아니다. 초안에는 기본값을 넣지 않으며, 주최자가 회의 길이를 명시적으로 한 번 결정해야 후보 선택으로 진행할 수 있다. 모든 후보는 같은 회의 길이를 사용한다.

### 후보 시간

회의 가능 기간과 회의 길이 안에서 주최자가 참석자에게 확인할 구체적인 시작 시간이다. 주최자가 선택했으므로 주최자는 해당 후보에 참석 가능한 것으로 본다.

### 응답 마감

참석자가 후보에 답해야 하는 시점이다. 회의 가능 기간이나 회의 실시 기한과 같은 값이 아니다.

## 6. 생성 흐름

상위 단계는 유지한다.

```text
회의 안내 → 참석자 → 시간 → 확인
```

시간 단계 내부는 점진적으로 공개한다.

```text
회의 가능 기간과 길이 결정
→ 조건을 짧게 접음
→ 조건 안에서 후보 시간 선택
→ 응답 마감 결정
```

미래 하위 과업의 내용을 한 화면에 모두 펼치지 않는다. 현재 판단이 끝나면 그 결과를 한 줄로 접고 다음 판단을 연다.

## 7. 후보 개수 정책

- 제품 입력에서 5개를 하드 제한하지 않는다.
- 첫 요청은 3~5개를 권장한다.
- 6개부터 참여자의 응답 부담이 커질 수 있음을 비차단 방식으로 알린다.
- 프로토타입의 후보 5개는 결과 판단과 참여자 응답 시간을 검증하기 위한 대표 테스트 세트다.
- 실제 검증 전에는 5개를 보편적인 최댓값으로 주장하지 않는다.

## 8. 참여자 계약

- 참여자는 주최자가 만든 정확한 후보에만 응답한다.
- 후보마다 `참석 가능`, `조정하면 가능`, `참석 어려움` 중 하나를 고른다.
- 다른 참여자의 응답과 주최자의 내부 판정 기준은 보지 않는다.
- 새 후보가 추가되면 기존 응답은 유지하고 새 후보만 응답한다.

## 9. 조율 회복 계약

- 새 후보는 기존 회의 가능 기간 안에서 먼저 추가한다.
- 기간 안에서 적절한 후보를 만들 수 없으면 주최자가 회의 가능 기간을 확장하는 별도 판단을 한다.
- 기간을 바꿔도 기존 링크, 참석자, 기존 후보 응답은 유지한다.
- 기간 변경과 응답 마감 변경은 서로 독립적으로 다룬다.

## 10. 의도적으로 제외한 범위

- 비참석 일정 조율 담당자
- 대리 일정 생성과 최종 결정 권한 이전
- 외부 캘린더 연동
- 전체 가능 시간 자동 수집
- AI 후보 생성
- 자동 회의 확정

## 11. 기존 결정과의 관계

다음 문서의 일부 결정을 대체한다.

- `create-attendee-flow-design-v1.6.md`: `나도 참석해요`를 기본 선택으로 제공하는 방향
- `create-attendee-selection-redesign-v1.7.md`: 요청자와 참석 여부를 토글로 분리하는 방향
- `prd-v1.4.md`: 후보 시간 최대 5개 입력을 P0 하드 제한으로 둔 방향
- 기존 `UX-ARCHITECTURE.md`: 후보 최대 5개 계약

과거 문서는 사고 이력을 위해 보존한다. 실행 시에는 `prd-v2.0.md`와 갱신된 `UX-ARCHITECTURE.md`를 따른다.

## 12. 아직 확정하지 않은 문제

다음 문제는 이 결정에 포함하지 않는다.

- 최소 참석 인원을 응답 전에 받을지, 응답을 본 뒤 판단하게 할지
- 회의 가능 기간의 권장 길이와 기술적 상한
- 회의 길이 프리셋 구성이 실제 사용 빈도와 맞는지
- 6개 이상 후보에 대한 실제 참여자 응답 부담

이 항목은 별도 검증 또는 후속 결정 기록으로 다룬다.

## 13. 검증 가설

- 사용자는 주최자가 참석자라는 전제를 별도 설명 없이 이해한다.
- 사용자는 회의 가능 기간과 주최자의 개인 가능 시간을 구분한다.
- 사용자는 회의 가능 기간과 응답 마감을 혼동하지 않는다.
- 주최자는 기간과 길이를 먼저 정한 뒤 후보를 더 빠르게 고른다.
- 3~5개 권장이 하드 제한 없이도 과도한 후보 생성을 줄인다.
