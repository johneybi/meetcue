## 1. 결정 요약

기존 P0은 주최자가 회의 가능 기간과 일정 확보 시간을 정한 뒤 `09:00~10:30` 같은 정확한 후보를 3~5개 골라 참석자에게 보내는 후보 투표형이었다.

v2.3에서는 기본 모델을 다음과 같이 바꾼다.

> 주최자는 정확한 후보 시작 시각을 나열하지 않고, 자신이 회의를 진행할 수 있는 조율 가능 시간대를 선택한다. 참석자도 같은 범위에서 자신의 가능 시간을 표시한다. 제품은 일정 확보 시간과 연속 가용성을 결합해 정확한 후보를 계산하고, 기존 회의 성립 기준으로 판단한다.

따라서 생성 단계의 핵심 객체는 고정 길이 `후보 블록`이 아니라 길이를 늘리고 줄일 수 있는 `조율 가능 시간대`다. `후보 시간`은 입력값이 아니라 응답 이후 계산되는 결과다.

이 결정은 전체 가용시간 자동 수집이나 AI 추천을 의미하지 않는다. P0에서는 사용자가 30분 단위로 직접 표시한 시간대와 명시적 판정 규칙만 사용한다.

## 2. 결정이 다시 열린 계기

### 2.1 확보 시간과 후보 블록의 관계

90분을 확보하도록 정했을 때 시작 칸 하나가 아니라 연속 3칸을 차지하는 블록이 보여야 한다는 요구에서 출발했다. 이 변경은 후보가 단일 시점이 아니라 유한한 시간 구간이라는 사실을 올바르게 드러냈다.

그러나 전체 블록을 보이게 하자 다음 문제가 연쇄적으로 나타났다.

- 클릭한 칸이 블록의 시작점인지 사전에 알기 어렵다.
- 블록을 옮기거나 삭제하는 방법을 추론하기 어렵다.
- 선택 가능 범위의 끝과 블록 종료 범위가 섞여 보인다.
- 한 후보가 차지하는 구간 때문에 인접한 다른 시작점이 비활성화된다.

### 2.2 상호작용 개선만으로 해결되지 않은 문제

다음 개선을 순서대로 검토하고 구현했다.

- 전체 확보 시간을 보여주는 고스트 미리보기
- 시각적 셀 간격과 실제 호버 면적 분리
- 선택 블록의 명시적 삭제와 실행 취소
- 블록 관리 행동을 오른쪽 선택 목록으로 이동
- 선택 블록 호버 시 `×` 노출

이 개선들은 조작 가능성과 피드백을 높였지만, 후보 투표형 자체의 인지 문제는 남았다.

### 2.3 10시 후보가 9시와 9시 30분을 막는 문제

90분 후보가 10시에 시작하면 9시와 9시 30분 시작 후보도 기존 블록과 일부 겹친다. 중복 후보를 허용하지 않는 로직에서는 두 칸을 선택할 수 없게 만드는 것이 계산상 맞다.

하지만 사용자는 회색 칸을 다음처럼 해석했다.

> 9시와 9시 30분은 원래 선택할 수 없는 시간이다.

실제 의미는 다음과 같았다.

> 이 시각에 90분 후보를 새로 만들면 이미 선택한 10시 후보와 일부 겹친다.

시스템 상태와 사용자 해석이 달랐다.

### 2.4 실패한 `겹침` 표현

회색 비활성화와 구분하기 위해 인접 칸에 파란 점선과 `겹침` 문구를 표시했다. 이 시도는 채택하지 않는다.

실패한 이유는 다음과 같다.

- 시간표는 공간 관계로 이해하는 UI인데 각 칸을 읽어야 하는 텍스트 UI가 되었다.
- 무엇과 무엇이 겹치는지 문구만으로 알 수 없었다.
- 9시 칸의 `겹침`은 9시 자체가 이미 점유되었다는 잘못된 의미로 읽혔다.
- 같은 단어가 반복되어 실제 선택 블록보다 시끄러워졌다.
- 사용자의 정상적인 선택을 오류나 경고처럼 보이게 했다.
- 구조 문제를 해결하지 않고 설명 책임을 사용자에게 넘겼다.

이 실패를 통해 문제를 `겹침 상태를 어떻게 표시할까`가 아니라 `주최자가 애초에 정확한 시작 후보를 나열해야 하는가`로 다시 정의했다.

## 3. 조사 범위와 근거

조사 질문은 다음과 같았다.

- 일반적인 일정 조율 서비스는 정확한 후보 투표와 가능 시간대 수집 중 무엇을 사용하는가?
- 후보끼리 겹치는 제안은 실제 제품에서 허용되는가?
- 회의 길이와 가능 시간대를 분리해 후보를 계산하는 모델이 관례적으로 존재하는가?

공개된 공식 문서를 기준으로 확인했다. 각 서비스가 기능을 제공한다는 사실은 확인할 수 있었지만, 주최자가 실제로 어떤 방식을 더 자주 사용하는지에 대한 사용 빈도 자료는 찾지 못했다. 따라서 기능 존재와 일반 사용 빈도를 동일하게 해석하지 않는다.

### 3.1 정확한 후보 투표형

- [Doodle Group Poll](https://help.doodle.com/en/articles/9457353-how-do-i-create-a-group-poll)은 주최자가 먼저 회의 길이를 정하고 캘린더에서 구체적인 시작 시각을 추가한 뒤, 참석자가 각 후보에 투표하는 구조다.
- [Calendly Meeting Poll](https://calendly.com/help/how-to-set-up-a-meeting-poll)은 최대 40개의 구체적인 시각을 제안하며 공식적으로 서로 겹치는 후보도 허용한다.

따라서 고정 길이 후보와 겹치는 후보 자체는 비정상적인 모델이 아니다. 다만 이 모델의 핵심 과업은 `몇 개의 제안 중 하나를 고르는 투표`다.

### 3.2 가능 시간대 수집형

- [When2meet](https://www.when2meet.com/)은 주최자가 가능한 날짜와 시간 범위를 정하고 참석자가 그 안에서 자신의 가용시간을 칠하는 구조다.
- [Microsoft findMeetingTimes](https://learn.microsoft.com/en-us/graph/findmeetingtimes-example)는 검색 기간, 회의 길이, 필수·선택 참석자, 업무 시간, free/busy를 별도 조건으로 받은 뒤 가능한 회의 시각을 계산한다.
- [Google Calendar Appointment Schedule](https://support.google.com/calendar/answer/10729749?hl=en)은 예약 길이와 일반 가용시간을 별도로 설정하고 그 조합으로 예약 가능한 시각을 만든다.
- [Google Calendar Find a time](https://support.google.com/calendar/answer/6294878?co=GENIE.Platform%3DDesktop&hl=en)은 참석자 캘린더의 가용성을 비교해 열린 시간을 찾게 한다.

이 모델의 핵심 과업은 `정확한 시작 시각에 투표`하는 것이 아니라 `어느 연속 구간을 사용할 수 있는지 수집하고 조건에 맞는 시작 시각을 계산`하는 것이다.

## 4. 제품 해석

### 4.1 주최자의 초기 정보는 보통 시간대다

정량 사용 데이터가 아니라 제품 구조를 바탕으로 한 추론이다.

회의를 처음 조율할 때 주최자가 알고 있는 정보는 대체로 다음 형태에 가깝다.

```text
화요일 오전은 가능하다.
수요일 2시부터 5시까지는 가능하다.
```

다음처럼 모든 시작 시각을 먼저 열거하는 것은 시스템 계산을 사용자에게 넘긴다.

```text
90분 회의이므로 09:00, 09:30, 10:00, 10:30 시작을 각각 후보로 만든다.
```

09:00~12:00이라는 한 가지 가능 사실이 네 개의 후보로 부풀고, 참석자는 사실상 같은 가용성을 반복해서 답하게 된다.

### 4.2 확정보드는 단순 후보 투표보다 가용성형에 가깝다

확정보드의 핵심은 가장 많은 표를 받은 후보를 고르는 것이 아니다.

- 필수 참석자가 참석 가능한가
- 최소 참석 인원을 충족하는가
- 조정이 필요한 사람이 적은가
- 미응답 불확실성이 얼마나 남았는가
- 지금 확정해도 되는가

이 판단에는 후보 선호도보다 각 참석자의 연속적인 가능 시간 정보가 더 직접적인 원재료다.

정확한 후보 투표형을 유지하면 제품은 입력 단계에서 시간을 인위적으로 잘게 나누고, 응답 이후 다시 그 조각을 해석해야 한다. 가능 시간대 수집형은 사용자가 알고 있는 정보와 제품이 필요한 정보 사이의 변환을 제품이 담당한다.

## 5. 검토한 대안

### A. 후보 중복을 금지하고 비활성화 이유를 더 잘 설명한다

- 구현 변화가 가장 작다.
- `겹침` 문구, 범례, 툴팁 같은 설명이 계속 필요하다.
- 주최자가 정확한 시작 후보를 반복해서 만드는 부담은 남는다.
- 채택하지 않는다.

### B. 정확한 후보를 유지하되 겹치는 후보를 허용한다

- Calendly처럼 유효한 후보 투표 모델이 된다.
- 가까운 시작 시각을 비교할 수 있다.
- 좁은 주간 열에서 겹치는 블록을 나란히 배치해야 한다.
- 참석자가 비슷한 후보에 반복 응답해야 하는 부담은 남는다.
- 기본 모델로 채택하지 않는다. 후속 `특정 후보만 제안` 모드로 검토할 수 있다.

### C. 주최자의 시간대를 먼저 받은 뒤 초대 전에 시스템이 후보로 잘라준다

- 주최자의 입력 부담은 줄어든다.
- 참석자는 여전히 잘게 나뉜 후보에 반복 응답한다.
- 후보를 몇 개까지 어떤 기준으로 보낼지 주최자가 다시 결정해야 한다.
- 근본적인 응답 모델은 바뀌지 않으므로 채택하지 않는다.

### D. 주최자와 참석자의 가능 시간대를 수집하고 정확한 후보를 응답 이후 계산한다

- 사용자 입력이 실제로 알고 있는 시간 범위와 맞는다.
- 회의 길이와 연속 가용성을 제품이 계산할 수 있다.
- 후보 중복을 입력 화면에서 다룰 필요가 없다.
- 기존 후보 응답 데이터 구조와 참석자 화면을 크게 바꿔야 한다.
- 확정보드의 판단 중심 제품 원칙과 가장 일치하므로 채택한다.

## 6. 채택한 경험 계약

### 6.1 주최자 생성

시간 단계 내부 순서는 다음과 같다.

```text
회의 가능 기간과 일정 확보 시간
→ 주최자가 가능한 조율 시간대
→ 응답 마감
```

- 회의 가능 기간은 날짜 범위다.
- 일정 확보 시간은 후보 하나가 성립하려면 연속으로 비워야 하는 시간이다.
- 조율 가능 시간대는 주최자가 해당 회의를 시작해 전체 확보 시간을 비울 수 있는 범위다.
- 주최자는 30분 단위 그리드에서 연속 범위를 칠한다.
- 선택 범위는 확보 시간보다 짧을 수 없다.
- 한 날짜에 여러 범위를 선택할 수 있다.
- 선택 범위끼리 맞닿으면 하나의 연속 범위로 합칠 수 있다.
- 정확한 후보 개수와 시작 시각은 생성 단계에서 요구하지 않는다.

### 6.2 참석자 응답

참석자는 주최자가 연 조율 범위 안에서 30분 단위로 자신의 상태를 표시한다.

- `가능해요`: 별도 조정 없이 비울 수 있다.
- `일정 조정하면 가능해요`: 앞뒤 일정이나 개인 계획을 조정하면 비울 수 있다.
- `어려워요`: 해당 시간을 명시적으로 비울 수 없다.
- `미응답`: 아직 저장하지 않았거나 새로 열린 시간대에 답하지 않은 상태다. `어려워요`와 합치지 않는다.

다른 참석자의 응답은 보지 않는다. 기본 입력은 빠르게 연속 범위를 칠할 수 있어야 하며 클릭·탭 대안을 제공한다. 참석자가 기존 조율 범위의 응답을 저장하면 표시하지 않은 칸은 명시적 `어려워요`로 확정할 수 있다. 이후 새 시간대가 열리면 새 칸만 `미응답`으로 시작한다.

### 6.3 정확한 후보 계산

제품은 각 조율 시간대 안에서 30분 간격으로 가능한 시작 시각을 만든다. 각 시작 시각은 `일정 확보 시간`만큼 연속된 칸을 차지한다.

예를 들어 09:00~12:00 범위와 90분 확보 시간이 있으면 다음 시작 시각을 계산할 수 있다.

```text
09:00~10:30
09:30~11:00
10:00~11:30
10:30~12:00
```

이 후보들은 계산 결과이므로 서로 겹칠 수 있다. 생성 입력 화면에서 블록 충돌로 취급하지 않는다.

### 6.4 참석자별 후보 응답 파생

후보가 차지하는 모든 30분 칸을 기준으로 참석자 상태를 계산한다.

```text
하나라도 어려워요
→ 후보 응답은 어려워요

어려운 칸은 없지만 하나라도 미응답
→ 해당 참석자는 후보에 미응답

모든 칸이 가능해요
→ 후보 응답은 가능해요

모든 칸이 가능 또는 조정 가능이고, 하나 이상이 조정 가능
→ 후보 응답은 일정 조정하면 가능해요

```

이렇게 파생한 후보별 상태를 기존 `evaluateCandidates`의 필수 참석자, 최소 참석 인원, 조정 필요, 미응답 판정에 연결한다.

### 6.5 주최자 결정

- 정확한 후보는 Host Decision에서 처음 핵심 반복 객체가 된다.
- 제품은 파생 후보를 회의 성립 상태와 리스크로 정렬한다.
- 자동 확정하지 않고 주최자가 최종 후보를 선택한다.
- 연속된 시작 시각이 같은 판정 결과를 만들더라도 각각 실제로 확정 가능한 시각이므로 시간 정보는 유지한다.
- 후보가 과도하게 많아질 때의 묶음·축약 방식은 프로토타입에서 별도 검증한다.

## 7. 조율 회복 영향

기존 회복 원칙은 유지한다.

- 기존 링크를 유지한다.
- 기존 참석자와 응답을 유지한다.
- 새로 필요한 응답만 요청한다.

다만 회복의 기본 행동은 `새 후보 추가`가 아니라 다음 중 하나가 된다.

- 기존 조율 범위 안에서 필요한 사람에게 응답 또는 수정 요청
- 조율 가능 시간대 확장
- 회의 가능 날짜 범위 확장
- 일정 확보 시간 재검토

새 시간대를 추가하면 기존 칸 응답은 유지하고 새로 열린 칸만 미응답 상태로 둔다.

## 8. 데이터 모델 영향

현재 구현은 `meeting.candidates`와 후보별 `Response`를 생성·응답·판정의 단일 원본으로 사용한다. v2.3을 구현하려면 최소한 다음 개념을 분리해야 한다.

```ts
type AvailabilityState =
  | 'AVAILABLE'
  | 'ADJUSTABLE'
  | 'UNAVAILABLE'
  | 'UNANSWERED'

type AvailabilityWindow = {
  id: string
  meetingId: string
  ownerId: string
  startAt: string
  endAt: string
  state: AvailabilityState
}

type DerivedCandidate = {
  id: string
  meetingId: string
  startAt: string
  endAt: string
}
```

- 주최자의 조율 가능 시간대도 가용시간 데이터로 저장한다.
- 참석자 응답은 후보 ID가 아니라 시간 구간 또는 30분 셀에 연결한다.
- `DerivedCandidate`는 입력값이 아니라 가용시간과 확보 시간에서 계산되는 안정적인 뷰 모델이다.
- 후보 ID는 `meetingId + startAt + endAt`에서 안정적으로 만들 수 있다.
- 최종 확정 시각은 파생 후보 중 하나를 선택한 결과로 저장한다.

정확한 저장 형태를 셀 배열로 할지 병합된 구간으로 할지는 구현 결정으로 남긴다. 사용자 경험과 판정 의미는 두 방식에서 동일해야 한다.

## 9. 언어와 컴포넌트 영향

기본 생성 화면에서 다음 표현을 바꾼다.

| 기존 | v2.3 |
| --- | --- |
| 후보 시간을 골라주세요 | 회의를 잡아도 되는 시간대를 알려주세요 |
| 후보 시간 고르기 | 가능한 시간대 고르기 |
| N개 후보로 계속 | 시간대 선택 완료 |

| `TimeCandidatePicker` | `AvailabilityWindowPicker` |
| `SelectedCandidateList` | `SelectedAvailabilitySummary` |
| 참석자 후보 카드 | 참석자 가용시간 그리드 |

`후보`라는 단어는 응답 이후 주최자 판단 화면에서만 사용한다.

## 10. 기존 결정과의 관계

### 유지

- P0에서 주최자는 항상 참석한다.
- 실제 회의 종료와 일정 확보 시간을 구분한다.
- 일정 확보 시간과 시간 최소 단위는 30분의 양의 배수다.
- 주최자가 최종 결정을 내린다.
- 필수 참석자와 최소 참석 인원 판정을 유지한다.
- `가능`, `조정하면 가능`, `어려움`의 3상태 의미를 유지한다.
- 기존 링크와 응답을 보존하는 회복 원칙을 유지한다.

### 대체

- 주최자가 생성 단계에서 정확한 후보 3~5개를 직접 고른다.
- 참석자가 정확한 후보마다 3상태로 응답한다.
- 주최자가 선택한 후보만 주최자의 암묵적 가능 상태다.
- 회복 시 새 후보만 추가하고 새 후보만 미응답으로 연다.

### 새로 열린 문제

- 파생 후보가 많을 때 주최자 결정 화면에서 묶거나 줄이는 방식
- 편집 중 표시하지 않은 칸이 저장 시 `어려움`으로 확정된다는 사실을 어떻게 알릴지
- 참석자가 부분 응답 후 저장할 수 있는지
- 시간대 확장 시 누구에게 새 칸 응답을 다시 요청할지
- 기존 후보 응답형 프로토타입 데이터를 v2.3 모델로 마이그레이션할지 초기화할지

## 11. 의도적으로 제외한 범위

- Google·Outlook 캘린더 자동 연동
- 캘린더 free/busy 자동 수집
- AI 기반 후보 생성
- 참석자 대신 자동 응답
- 주최자가 참석하지 않는 대리 조율
- 후보별 서로 다른 확보 시간
- 30분보다 작은 시간 단위

## 12. 검증 가설

- 주최자는 정확한 시작 시각 여러 개보다 가능한 연속 시간대를 더 빠르게 입력한다.
- 90분 확보 시간과 09:00~12:00 가능 시간대를 서로 다른 값으로 이해한다.
- 참석자는 비슷한 고정 후보에 반복 응답하는 것보다 한 번의 가용시간 입력을 더 가볍게 느낀다.
- `일정 조정하면 가능`을 시간대에 칠하는 동작을 후보별 응답과 같은 의미로 이해한다.
- 제품이 계산한 정확한 후보를 주최자가 자신의 입력을 벗어난 추천으로 오해하지 않는다.
- 조율 범위를 확장할 때 기존 응답이 유지된다는 사실을 이해한다.

## 13. 프로토타입 검증 질문

- `회의를 잡아도 되는 시간대`를 주최자 개인의 가용시간으로 이해하는가?
- 09:00~12:00을 선택할 때 90분 회의가 그 안의 어느 시작 시각에도 놓일 수 있다고 이해하는가?
- 참석자는 가능한 칸을 모두 칠하는가, 가장 선호하는 칸만 칠하는가?
- 조정 가능 시간을 연속 범위로 표시할 때 실제 약속의 강도를 어떻게 해석하는가?
- 제품이 계산한 겹치는 시작 후보들을 결정 화면에서 비교할 수 있는가?
- 파생 후보가 많을 때 몇 개까지 한눈에 판단할 수 있는가?

## 14. 구현 전 조건

v2.3은 구조 변경이므로 현재 후보 블록 UI를 부분 수정하는 방식으로 구현하지 않는다.

구현 전에 다음을 먼저 정한다.

1. 주최자 시간대 선택의 클릭·드래그·수정·삭제 모델
2. 참석자 3상태 가용시간 입력 모델
3. 파생 후보 생성과 기존 판정 함수 연결 규칙
4. 파생 후보가 많은 Host Decision 대표 화면
5. 기존 P0 데이터의 전환 범위

이 조건을 검증한 뒤 코드의 `TimeCandidatePicker`, 후보별 응답, 회복 화면을 같은 상태 모델로 함께 바꾼다.
