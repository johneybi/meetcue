# 조직 계정 기반 응답 요청 결정 기록 v2.8

- 상태: 채택 / 프로토타입 반영 / 실제 연동 제외
- 결정일: 2026-07-12
- 적용 범위: MeetCue P0의 최초 요청, 재알림, 확정 알림, 참여자 식별
- 대체 결정: `personal-invite-identity-decision-v2.7.md`
- 제품 명세: `prd-v2.2.1.md`
- 실행 명세: `../UX-ARCHITECTURE.md`

## 1. 문제

v2.7은 공용 링크에서 참여자가 자기 이름을 고르는 문제를 개인 토큰 링크로 해결했다. 그러나 MeetCue는 참석자를 사내 디렉터리에서 검색하고, 최근 함께한 사람과 개인 캘린더를 사용하는 조직용 제품을 전제한다. 이때 참석자마다 링크를 복사해 외부로 전달하는 흐름은 이미 알고 있는 계정 식별을 다시 수동 전달 문제로 바꾼다.

## 2. 결정

> 주최자는 사람을 선택한 뒤 응답 요청을 보낸다. 제품은 각 조직 계정에 과업을 할당하고 알림을 전달한다. 참석자는 로그인된 자신의 계정에서 바로 응답한다.

- 생성 마지막 CTA는 `응답 요청 보내기`다.
- 요청 뒤에는 링크 복사 화면이 아니라 발송 대상과 응답 상태를 보여준다.
- 응답 전인 사람에게는 `다시 알리기`를 제공한다.
- PENDING 후보의 추가 응답도 선택한 계정에 시스템 요청으로 보낸다.
- 회의 확정 뒤에는 참석자 계정으로 확정 알림을 보낸다.
- 실제 알림, 이메일, 계정 인증, 서버 저장은 구현하지 않고 fixture 상태 변화로 시연한다.

## 3. 참여자 식별

- 실제 제품에서는 로그인 세션과 회의 요청의 수신자 ID로 참여자를 식별한다.
- 참여자는 자기 이름을 고르거나 개인 링크를 관리하지 않는다.
- 프로토타입의 `#/invite/:token`은 계정 세션을 대신하는 개발용 페르소나 라우트다.
- 이 토큰은 주최자 화면에 복사·공유 기능으로 노출하지 않는다.
- DEV 화면 바로가기에서 참여자별 신규·수정·완료 상태를 전환한다.

## 4. 검토한 대안

| 대안                    | 장점                              | 문제                                | 결정                 |
| ----------------------- | --------------------------------- | ----------------------------------- | -------------------- |
| 개인 링크 복사          | 백엔드 없이 전달 과정을 쉽게 시연 | 수동 전달, 오발송, 계정 모델과 중복 | 주 흐름에서 제거     |
| 공용 링크에서 이름 선택 | 구현이 단순                       | 오응답과 개인정보 노출              | 기각 유지            |
| 조직 계정에 과업 할당   | 실제 사용 맥락과 가장 일관됨      | 인증·알림 인프라가 필요             | 채택, 연동은 fixture |
| 외부 게스트용 보안 링크 | 계정 없는 사람도 참여 가능        | 별도 권한·만료·회수 설계 필요       | P0 이후 fallback     |

## 5. 화면 계약

### 요청 완료

- 요청받은 사람 수와 각 사람의 역할·응답 상태를 보여준다.
- 상태는 `요청됨`, `응답 완료`처럼 과업 상태로 표현한다.
- 링크 복사와 참여자 미리보기는 제공하지 않는다.
- 주요 행동은 `응답 현황 보기`다.

### 응답 대기

- 미응답자별로 `다시 알리기`를 제공한다.
- 다른 참여자 계정으로 전환하는 기능은 제품 화면에 두지 않는다.

### 참여자

- 자신의 계정에 배정된 회의 요청과 자기 응답만 본다.
- 다른 참석자의 이름, 응답 상태, 결정 기준은 보지 않는다.

## 6. 트레이드오프

- 계정 기반 모델은 조직 밖 참석자를 바로 지원하지 못한다.
- 프로토타입은 알림 성공·실패, 권한 만료, 계정 비활성화, 중복 발송을 검증하지 않는다.
- 채용 과제에서는 연동 완성도가 아니라 제품이 계정 기반으로 어떻게 동작할지를 시각적 상태와 더미 데이터로 증명한다.

## 7. 수용 기준

- 회의 생성 후 개인 링크 복사를 거치지 않고 요청 완료 화면으로 이동한다.
- 요청 완료 화면에서 대상과 상태를 확인하고 응답 현황으로 이동할 수 있다.
- 미응답자에게 다시 알리기를 실행할 수 있다.
- 참여자 전환은 DEV 도구에서만 가능하다.
- 주최자 제품 화면에 참여자 토큰이나 개인 링크가 노출되지 않는다.
- PENDING 재요청과 확정 안내도 문구 복사가 아니라 계정 알림으로 표현된다.
