# MeetCue 디자인 일관성 개선 기록

## 목적

원본 `main`을 변경하지 않는 별도 worktree에서 핵심 시연 흐름의 디자인 일관성을 검증하고 개선한다. 단순히 화면을 보기 좋게 만드는 것이 아니라, 사용자가 현재 상태와 다음 행동을 설명 없이 이해하도록 상태·단계·행동의 배치 규칙을 통일하는 것이 목표다.

## 작업 환경과 범위

- 브랜치: `codex/message-preview`
- worktree: `C:\tmp\meetcue-message-preview`
- 시안 서버: `http://127.0.0.1:5174/meetcue/toss`
- 원본 보존: `E:\06Toss`의 소스 코드는 수정하지 않는다.
- 디자인 기준: 현재 MeetCue의 토큰, 공용 UI 컴포넌트, 중립 표면과 파란 액센트를 유지한다.
- 제외: Figma, 백엔드, 실제 캘린더·알림 연동, 핵심 시연과 무관한 신규 기능

## 개선 원칙

1. 상태는 하나의 원천에서 파생하고 화면 안에서 서로 모순되지 않게 한다.
2. 단계가 바뀌면 새 과업의 제목이 보이고 포커스가 이동해야 한다.
3. 주 행동은 현재 작업 표면에 결합하고 화면마다 위치 규칙을 바꾸지 않는다.
4. 이미 시각적으로 표현된 상태를 설명 문구로 반복하지 않는다.
5. 데스크톱은 비교와 스캔을 돕고, 모바일은 한 번에 한 결정에 집중시킨다.
6. 데모 전용 UI는 제품의 핵심 조작과 콘텐츠를 가리지 않는다.
7. 색상만으로 상태를 전달하지 않고 텍스트·아이콘·선택 상태를 함께 사용한다.

## 완료 기준

- 단계 이동 직후 진행 상태와 새 제목이 첫 화면 안에 보인다.
- 참석 기준의 선택값, 요약, 결과 판정이 동일한 데이터에서 파생된다.
- 데모 가이드의 각 단계가 이름과 일치하는 fixture 상태를 연다.
- PC와 모바일에서 데모 가이드가 CTA 또는 시간표를 가리지 않는다.
- 생성 흐름의 주 CTA가 현재 카드와 시각적으로 결합된다.
- 결과 보드에서 선택 시간은 한 번만 강하게 강조되고 판정 근거가 다음 위계가 된다.
- 비교표는 날짜·시간 헤더를 한눈에 구분할 수 있고 선택 열과 참석자 열을 놓치지 않는다.
- 모바일 참석자 입력 첫 화면에서 안내보다 실제 시간 입력이 우선한다.
- `npm run build`와 핵심 시연 흐름의 PC·모바일 확인을 통과한다.

## 결정 기록

### D-001: 최종 확인 화면의 정보 위계

- 관찰: 회의명과 날짜·시간의 크기가 연속해서 커져 위계가 의도보다 장식적으로 보였다.
- 결정: 날짜·시간을 `확정할 일정`의 1차 정보로 두고, 회의명은 그 아래 맥락 정보로 낮춘다. 알림 대상과 CTA는 같은 카드 안에서 이어진다.
- 트레이드오프: 날짜가 먼저 보여 회의명이 상대적으로 약해지지만, 직전 결과 화면에서 이미 회의를 선택한 흐름이므로 최종 확인에서는 확정 시각이 더 중요한 정보다.
- 적용: `MessageScreen.tsx`, `MessageScreen.css`
- 검증: 데스크톱·모바일에서 확인 전/후 상태를 점검했고 빌드가 통과했다.

### D-002: 참석 기준은 preset과 역할에서 파생한다

- 관찰: `core_attendees`가 선택된 상태에서도 이전 최소 인원 값이 남아 `필수 3명 · 최소 4명`처럼 모순된 요약이 보였다.
- 결정: `all_hands`는 전체 인원, `core_attendees`는 필수 역할 수, `quorum`은 필수 역할 수 이상의 직접 설정값으로 정규화한다. 필수 역할을 수정할 때도 같은 규칙을 적용한다.
- 트레이드오프: 이전에 별도로 입력했던 최소 인원은 `필수 참석자만` 모드로 전환하는 순간 폐기된다. 하지만 이 모드에서는 그 값이 의미를 잃으므로 숨겨서 보존하는 것보다 상태를 명확히 만드는 편이 안전하다.
- 적용: `MeetingCriteriaReviewScreen.tsx`
- 검증: 빌드와 기존 도메인 테스트 32개가 통과했다.

### D-003: 화면과 생성 외부 단계 이동은 맥락의 시작점부터 다시 보여준다

- 관찰: 새 단계의 본문을 `scrollIntoView` 대상으로 사용해 상단 진행 상태가 화면 밖으로 밀렸다.
- 결정: hash 라우트가 바뀌면 페이지를 최상단으로 초기화한다. 생성의 외부 4단계가 바뀔 때는 전체 workflow를 스크롤 대상으로 사용하고, 새 단계 제목에 포커스를 둔다. 고정 헤더만큼 `scroll-margin-top`을 예약한다.
- 트레이드오프: 내부 시간 편집 위치보다 전체 단계 맥락을 먼저 보여준다. 외부 단계 전환에서는 사용자의 재지향이 우선이다.
- 적용: `useMeetCueController.ts`, `useCreateFlowController.ts`, `CreateScreen.css`
- 검증: 빌드 통과. 실제 PC·모바일 단계 이동 검증은 마지막 QA에서 수행한다.

### D-004: 데모 가이드 단계는 독립적으로 재현 가능한 fixture를 연다

- 관찰: ‘바뀐 결과 확인’을 직접 열면 수진의 제출 상태가 보장되지 않아 이전 PENDING 결과가 그대로 나타났다.
- 결정: 3단계는 미응답 수진 fixture, 4단계는 같은 후보에 수진의 가능 응답이 제출된 `responded` fixture를 사용한다.
- 트레이드오프: 3단계에서 사용자가 입력한 세부 상태를 4단계가 그대로 이어받지는 않는다. 대신 평가자가 어떤 단계부터 시작해도 의도한 시연 장면을 확실히 볼 수 있다.
- 적용: `DemoGuide.tsx`, `DevScreenSwitcher.tsx`, `prototypeState.ts`, `useMeetCueController.ts`
- 검증: 빌드와 기존 재판정 도메인 테스트가 통과했다.

### D-005: 생성 CTA와 참석자 하위 흐름은 한 작업 표면으로 묶는다

- 관찰: PC CTA가 카드 바깥 오른쪽에 놓여 현재 입력과 분리돼 보였다. 참석자 선택 완료 후 참석 기준 화면은 같은 2단계의 후속 질문보다 별개의 화면처럼 느껴졌다.
- 결정: CTA를 공용 `MainCardFooter` 안으로 옮기고 카드 너비를 사용한다. 참석자 선택과 기준 질문은 별도의 숫자 단계 없이 연결선과 상태 노드로 관계를 보여준다. 완료된 사람 선택은 한 줄 요약으로 접고 현재 질문만 펼친다.
- 트레이드오프: PC에서도 CTA의 시각적 비중이 커진다. 하지만 생성 흐름은 한 화면에 하나의 주 행동만 존재하므로 명확성과 모바일 일관성을 우선한다.
- 적용: `CreateFlowFrame.tsx`, `CreateScreen.css`, `CreateAttendeeFlow.css`, `useAttendeeSelection.ts`
- 검증: 데스크톱에서 사람 선택 전·후를 확인했다. 진행 상태, 완료 요약, 현재 질문, 카드 푸터 CTA가 한 화면에 유지된다.

### D-006: 결과 보드는 선택 시간보다 판정 근거를 이어서 읽게 한다

- 관찰: 후보 목록, `선택한 후보`, 큰 시간 제목, 판정 카드가 같은 시간을 반복해 판정 이유보다 선택 사실이 더 강하게 보였다. 비교표는 긴 날짜·시간이 여러 줄로 꺾여 후보 간 차이를 스캔하기 어려웠다.
- 결정: 선택 시간과 판정 상태를 하나의 상세 카드 헤더로 통합한다. 비교표 헤더는 짧은 날짜와 24시간 시간 범위로 분리하고, 참석자 열은 고정하며 선택된 후보 열에 연속된 파란 경계를 둔다.
- 트레이드오프: 시간 표현이 기존 자연어보다 압축되지만, 비교 화면에서는 문장형 표현보다 열 간 식별 속도를 우선한다.
- 적용: `HostCandidateDetail.tsx`, `HostCandidateDetail.css`, `HostDecisionMatrix.tsx`, `HostDecisionMatrix.css`
- 검증: 빌드 통과. 브라우저 제어가 차단되어 최종 시각 검증은 사용자 확인 대기 상태다.

### D-007: 모바일에서는 실제 입력과 비교를 개발 안내보다 우선한다

- 관찰: 데모 가이드가 고정 CTA와 시간표를 덮었고, 캘린더 불러오기 결과·반복 안내·범례가 날짜 탭과 실제 입력보다 먼저 많은 높이를 차지했다. 결과 후보 목록의 다음 카드 노출도 의도적인 탐색 단서보다 잘린 레이아웃처럼 보였다.
- 결정: 680px 이하에서는 데모 가이드를 숨긴다. 참석자 입력에서는 캘린더 불러오기 결과를 한 줄로 압축하고, 조작 설명은 숨기되 세 상태 범례는 유지한다. 결과 후보 카드는 화면의 78% 이내 폭과 오른쪽 스크롤 여백을 사용해 다음 카드의 일부가 일관되게 보이도록 한다.
- 트레이드오프: 모바일에서 개발용 단계 바로가기를 쓸 수 없다. 대신 제출용 제품 화면과 핵심 CTA가 개발 도구에 가려지지 않으며, 데모 탐색은 데스크톱 가이드와 직접 URL로 유지된다.
- 적용: `DemoGuide.css`, `ParticipantAvailabilityPanel.css`, `HostCandidateShortlist.css`
- 검증: 빌드 통과 후 실제 기기 폭의 사용자 확인이 필요하다.

### D-008: 하위 단계 표현은 실제 하위 단계가 둘 이상일 때만 사용한다

- 관찰: 참석자 선택 첫 화면에는 아직 하나의 질문만 보이는데도 노드와 32px 들여쓰기가 적용돼 제목·입력·목록이 카드 기준선에서 이탈했다. 카드 헤더에는 기존 하단 margin과 새 카드 gap이 동시에 적용돼 구분선 아래가 약 46px 벌어졌다.
- 결정: 카드 내부 헤더의 기존 하단 margin을 제거하고 카드 gap만 사용한다. 참석 기준 질문이 아직 나타나지 않은 첫 화면에서는 노드와 들여쓰기를 숨기며, 사람 선택 완료 후 두 하위 단계가 함께 존재할 때만 연결 구조를 활성화한다.
- 트레이드오프: 첫 질문에서는 앞으로 이어질 하위 질문을 미리 암시하지 않는다. 아직 조작할 수 없는 구조를 보여주는 것보다 현재 과업의 정렬과 집중도를 우선한다.
- 적용: `CreateScreen.css`, `CreateAttendeeFlow.css`
- 검증: 사용자 캡처에서 발견한 회귀를 기준으로 교정했으며 빌드 후 재확인이 필요하다.

### D-009: 레거시 reset과 충돌하는 입력 여백은 컴포넌트 소유 범위에서 명시한다

- 관찰: 참석자 검색 필드의 Tailwind `px-[18px]`가 레거시 입력 reset에 덮여 placeholder가 왼쪽 border에 붙었다.
- 결정: 참석자 검색 컴포넌트가 field 크기 Input의 18px 내부 여백을 명시적으로 소유한다.
- 트레이드오프: 공용 Input의 스타일 계층 문제를 이번 범위에서 전면 정리하지는 않는다. 다른 입력에서 같은 현상이 발견되면 공용 primitive의 로드 순서와 specificity를 별도 리팩터링해야 한다.
- 적용: `AttendeePeopleStep.css`
- 검증: 사용자 화면에서 재확인이 필요하다.

### D-010: 참석자 하위 흐름은 타임라인이 아니라 완료 요약과 현재 질문으로 구분한다

- 관찰: 상위 제목 `누구와 회의하나요?`와 내부 질문 `누구에게 시간을 물어볼까요?`가 거의 같은 크기와 무게로 경쟁했다. 사람 선택을 완료한 뒤에만 노드와 연결선이 갑자기 나타나 이전 화면과의 공간 모델도 달라졌다. 노드 크기와 선의 정렬까지 상태별로 달라 시각적 완성도를 낮췄다.
- 결정: 노드와 연결선을 전부 제거한다. 첫 화면의 내부 제목은 작은 섹션 레이블 `참석자`로 낮춘다. 사람 선택을 완료하면 같은 영역을 참석자 요약으로 접고, 구분선 아래에 참석 기준 질문을 현재 과업의 제목 크기로 펼친다.
- 트레이드오프: 하위 순서를 도식으로 직접 설명하지 않는다. 대신 완료된 정보와 현재 과업의 크기·접힘 상태·구분선만으로 진행 관계를 일관되게 전달한다.
- 적용: `AttendeePeopleStep.tsx`, `CreateAttendeeFlow.css`
- 검증: 사용자 화면에서 선택 전과 선택 후 상태를 각각 재확인해야 한다. 이 결정은 노드 사용을 전제로 한 D-005와 D-008의 표현 방식을 대체한다.

### D-011: 완료된 참석자 정보는 선으로 나누지 않고 한 행으로 요약한다

- 관찰: `참석자`, `3명`, 참석자 이름, `수정`이 두 행에 흩어졌고 요약 위·아래와 참석 기준 시작점에 divider가 연속으로 놓였다. 카드 헤더와 푸터 선까지 더해져 정보 위계보다 구획선이 먼저 보였다.
- 결정: 완료된 참석자 정보를 `참석자 · 3명 이름 · 수정`의 단일 수평 행으로 합치고 옅은 중립 배경만 사용한다. 참석 기준 시작점의 divider도 제거하고 28px 여백으로만 두 영역을 구분한다.
- 트레이드오프: 참석자 요약의 독립적인 섹션성이 약해지지만, 이미 완료된 정보이므로 현재 질문보다 조용하게 보여야 한다. 좁은 모바일에서는 이름 목록을 숨기고 인원수만 유지한다.
- 적용: `AttendeePeopleStep.css`, `CreateAttendeeFlow.css`
- 검증: 사용자 화면에서 데스크톱·모바일 정렬을 재확인해야 한다.

### D-012: 모바일 시간 범위 편집은 도구 모드 없이 칸의 포함 상태를 직접 바꾼다

- 관찰: 모바일의 `제외할 시간/추가할 시간` 모드는 사용자가 도구를 먼저 고르고 그 상태를 기억한 채 시간 칸을 조작하게 했다. 결과 상태보다 편집 모드가 앞서며 잘못된 모드로 칠할 가능성도 있었다.
- 결정: 모바일 시간 칸을 `포함/제외`의 binary 상태로 취급한다. 탭하면 현재 상태가 즉시 반전되고 칸 안에 상태 텍스트를 함께 표시한다. 편집 모드와 중복 범례는 숨기고 `기본값 복원`만 유지한다.
- 트레이드오프: 모바일에서는 여러 칸을 한 번에 추가·제외하는 도구 모드를 사용하지 않는다. 기본 범위가 이미 채워져 있고 주최자가 예외만 수정하는 흐름이므로 직접 조작의 명확성을 우선한다. 데스크톱 드래그 편집은 유지한다.
- 적용: `AvailabilityWindowPicker.tsx`, `AvailabilityWindowPicker.css`
- 검증: 포함된 칸과 제외된 칸을 각각 탭했을 때 상태·색상·접근성 레이블이 함께 반전되는지 모바일에서 확인해야 한다.

### D-013: 동일한 수준의 뒤로가기는 quiet 텍스트 패턴으로 통일한다

- 관찰: create의 `이전`은 테두리와 흰 배경을 가진 작은 컨트롤이었지만 결과에서 기준 수정 화면의 `결과로 돌아가기`는 화살표와 텍스트만 있는 quiet 탐색이었다. 두 행동의 중요도와 역할은 같은데 시각적 규칙이 달랐다.
- 결정: create도 18px `ArrowLeft`와 텍스트로 구성된 quiet 버튼을 사용하고 테두리·독립 배경을 제거한다. 콘텐츠 왼쪽 기준선에는 결과 화면과 같은 방식으로 정렬한다.
- 트레이드오프: 버튼의 외곽 경계가 없어 발견성은 조금 낮아지지만, 진행 상태와 주 CTA보다 뒤로가기가 강해지는 문제를 피하고 화면 간 탐색 규칙을 통일한다.
- 적용: `CreateFlowFrame.tsx`, `CreateScreen.css`
- 검증: create 2~4단계와 기준 수정 화면을 나란히 비교해야 한다.

### D-014: 시간표 단계의 외곽 폭 확장 시도는 폐기한다

- 관찰: 시간표 단계의 어색함을 공간 부족으로 해석해 surface를 820px에서 1040px로 넓혔지만, 내부 정보 구성과 밀도는 그대로여서 빈 공간만 증가했다. 사용자가 지적한 문제는 카드 크기가 아니라 내부에 애매하게 남는 여백과 정렬 관계였다.
- 결정: 단계별 1040px 확장을 되돌리고 공통 820px 카드 폭을 유지한다. 후속 교정은 외곽 폭이 아니라 카드 내부 패딩, 시간표의 실제 점유율, 상태·도구 영역의 정렬을 대상으로 한다.
- 트레이드오프: 7일 시간표의 각 열은 넓어지지 않는다. 그러나 현재 문제를 해결하지 않는 외곽 확장보다 기존 레이아웃 안정성을 우선한다.
- 적용: `CreateScreen.css`의 단계별 폭 override 제거
- 검증: 되돌린 상태를 기준으로 내부 여백이 발생하는 정확한 영역을 다시 특정해야 한다.

### D-015 — 시간 범위 편집 화면은 카드가 아니라 내부 레이아웃을 확장한다

- 상태: 적용
- 대상: 생성 3단계 `가능한 시간대 고르기`
- 문제: 카드 내부 가용 폭은 770px이지만 시간 편집 본문은 660px에서 끝나 오른쪽에 110px의 무의미한 여백이 남았다. 선택 범위 밖 날짜의 셀도 거의 보이지 않아 시간표가 더 짧고 미완성된 것처럼 보였다.
- 결정: 공통 카드 폭과 패딩은 유지한다. 시간 후보 편집 단계만 카드 내부 폭을 모두 사용하고, 범위 밖 날짜는 옅은 비활성 셀로 표시한다. 주간 탐색은 시간 축을 제외한 날짜 열 중앙에 정렬한다.
- 근거: 이 화면의 문제는 페이지 폭이 아니라 카드와 시간표가 서로 다른 내부 기준선을 사용한 데 있다. 카드 확장은 다른 생성 단계와의 일관성을 깨뜨리며 문제를 해결하지 못한다.
- CSS 소유권: 단계 폭의 최종 계약은 `CreateScreen.css`가 소유하고, 시간표의 로컬 상태와 탐색 정렬은 `AvailabilityWindowPicker.css`가 소유한다.

### D-016 — 요청 완료와 응답 현황은 하나의 상태 흐름으로 보이게 한다

- 상태: 적용
- 대상: `요청을 보냈어요`에서 `응답 현황 보기`로 이동하는 전환
- 문제: 요청 완료 화면은 760px·무그림자·14px 카드이고 응답 대기 화면은 820px·그림자·16px 카드라, 같은 회의의 상태가 바뀌는 대신 별도 제품 화면으로 이동한 인상을 주었다.
- 결정: 두 화면의 외곽 프레임과 카드 표면을 공통 규칙으로 통일한다. 응답 현황 화면에서는 별도 회의 헤더를 한 줄 메타 정보로 압축하고, 그라디언트 현황 패널을 중립 표면으로 바꾸며, 카드 밖 데모 CTA를 카드 푸터로 결합한다. 참석자 목록과 응답 진행 정보처럼 상태별로 필요한 내부 구조는 유지한다.
- 근거: 화면의 내용 밀도는 달라질 수 있지만 같은 객체를 추적하는 연속 흐름에서는 바깥 골격과 표면 규칙이 먼저 유지되어야 한다.

### D-017 — 응답 대기 화면은 제품 상태만 보여준다

- 상태: 적용
- 대상: `/host`의 `HOST_WAITING_EMPTY` 상태
- 문제: 제품 카드 안의 `데모 시연 / 다음 단계 보기`가 가장 강한 CTA였고, 내부 계산량인 후보 시간 개수와 동일한 응답 진행 수치가 반복됐다. 모바일에서는 참석자마다 전체 너비 재알림 버튼이 생겨 목록이 과도하게 길어졌다.
- 결정: 현황 패널에는 완료 인원, 진행 막대, 응답 마감만 남긴다. 참석자 목록의 대기·완료 합계는 제거하고, 정렬 안내는 상태가 섞였을 때만 표시한다. 모바일 재알림은 한 줄 아이콘 액션으로 축약한다. 데모 전환은 제거하지 않고 카드 하단의 낮은 위계 푸터로 분리해 `응답이 온 결과 보기`를 제공한다.
- 근거: 제품 상태와 시연 장치는 시각적으로 구분되어야 하지만, 독립 실행형 프로토타입에서는 핵심 시연 흐름의 연속성도 보장해야 한다. 데모 행동을 주 CTA보다 낮게 두면 두 요구를 함께 만족한다.

### D-018 — 생성 단계에서 6명 회의 조건을 실제로 구성할 수 있게 한다

- 상태: 적용
- 대상: `/create` 참석자 선택
- 문제: 최근 함께한 사람이 4명뿐이라 모두 선택해도 주최자 포함 5명으로, 과제의 6명 회의 조건을 생성 흐름에서 재현할 수 없었다. 로컬 저장소에 기존 4명 목록이 있으면 디렉터리를 늘려도 화면에는 반영되지 않았다.
- 결정: 선택 가능한 기본 인물 디렉터리를 6명으로 확장하고, 저장된 최근 목록과 기본 디렉터리를 병합한다. 사용자는 주최자 외 5명을 선택해 총 6명 회의를 만들 수 있다.
- 근거: 생성 입력과 결과 fixture가 같은 과제 조건을 공유해야 시연 중 인원 기준과 판정 결과가 모순되지 않는다.

### D-019 — 결과 보드는 후보 수가 늘어나도 레이아웃과 비교 의미를 유지한다

- 상태: 적용
- 대상: `/host` 응답 결과
- 문제: 생성 흐름에서 만들어진 75개 후보가 비교표의 75개 열로 그대로 노출돼 표 너비가 약 8,856px까지 늘어났다. 왼쪽 추천 목록의 높이 때문에 오른쪽 상세 카드도 626px로 늘어나 내용 아래가 큰 빈 공간이 됐다.
- 결정: 상세 카드는 내용 높이로 고정한다. 추천 목록은 날짜 다양성을 유지한 5개로 제한한다. 전체 비교표는 참석자별 판정 상태가 동일하며 30분 간격으로 연속되는 후보를 하나의 시간 범위로 묶고, 묶인 후보 수를 함께 표시한다.
- 근거: 내부 계산 후보는 유지하되 화면은 서로 다른 판단 결과를 비교하는 데 집중해야 한다. 동일 결과의 연속 후보를 묶으면 정보 손실 없이 가로 폭과 반복을 줄일 수 있다.

### D-020 — 결과 보드의 좌우 기준선과 비교표 여백을 맞춘다

- 상태: 적용
- 대상: `/host` 응답 결과
- 문제: 선택한 후보 상세 카드가 왼쪽의 `추천 후보` 제목 행과 정렬돼 실제 첫 후보 카드보다 위에서 시작했다. 전체 후보 비교 헤더는 상하 여백이 작아 넓은 결과 보드 안에서 압축돼 보였다.
- 결정: 데스크톱 상세 카드는 추천 목록 제목 높이와 간격만큼 내려 첫 후보 카드의 상단 기준선과 맞춘다. 모바일은 상세 카드가 먼저 나오는 구조이므로 오프셋을 제거한다. 비교표 summary의 최소 높이와 상하 패딩을 늘린다.
- 근거: 서로 직접 비교되는 좌우 표면은 같은 시작선을 공유해야 하며, 큰 비교 영역의 진입 헤더에는 본문 행보다 충분한 수직 여백이 필요하다.

### D-021 — 선택한 후보 상세 카드의 간격은 의미 그룹이 소유한다

- 상태: 적용
- 대상: `/host` 선택한 후보 상세 카드
- 문제: 상태 레이블, 제목, 설명, 근거, 인원 칩, CTA가 한 컨테이너의 형제였고 부모 gap 위에 음수·추가 margin이 중첩돼 실제 간격이 제각각이었다. 판단 근거의 위아래 divider도 카드를 필요 이상으로 분절했다.
- 결정: 카드 내부를 시간 헤더, 판정 요약, 판단 근거, 행동 영역으로 묶는다. 판정 요약 안에 상태와 인원 현황을 함께 배치하고, 판단 근거는 상단 divider 하나만 사용한다. CTA는 별도 행동 영역이 소유하며 개별 margin을 제거한다.
- 근거: 관련 정보는 가까이, 다른 의미의 정보는 일정하게 멀리 배치해야 사용자가 판정과 근거, 행동을 순서대로 읽을 수 있다.

### D-022 — 참석자는 이름 검색과 조직 탐색 중 상황에 맞는 방법으로 찾는다

- 상태: 적용
- 대상: `/create` 2단계 참석자 선택
- 문제: 이름 검색과 최근 함께한 사람만으로는 이름을 정확히 기억하지 못하거나 다른 팀 구성원을 역할·소속으로 찾기 어렵다.
- 결정: 검색 옆에 `팀에서 찾기`를 보조 진입점으로 제공한다. 프로토타입용 조직·팀·직무 데이터를 탐색하고 여러 명을 임시 선택한 뒤 완료 시 기존 참석자 목록에 반영한다. 데스크톱은 좌측 팀·우측 구성원 구조, 모바일은 가로 팀 탭과 구성원 목록을 사용한다. 오버레이는 `document.body`에 포털로 렌더링해 상위 작업 카드의 클리핑과 무관하게 전체 화면을 dim 처리한다.
- 근거: 검색은 아는 사람을 빠르게 찾는 방식이고 조직도는 모르는 사람을 소속 맥락으로 탐색하는 방식이므로 서로 대체 관계가 아니다.

### D-023 — 데모 가이드는 화면 목록이 아니라 판단 변화의 인과를 재현한다

- 상태: 적용
- 대상: 데모 가이드와 개발용 `핵심 시연` 1~5단계
- 문제: 기존 2단계는 `응답이 필요한 상황`이라는 상태 설명에 그쳐 주최자의 행동이 드러나지 않았다. 3단계는 진입 전 상태에 영향을 받을 수 있었고, 5단계는 이미 확정된 fixture를 열어 사용자가 최종 확인과 확정을 직접 수행하지 못했다.
- 결정: 2단계를 `추가 응답 요청하기`로 정의하고, 2·3단계가 동일한 PENDING 상황을 독립적으로 재현하도록 전용 `pending` fixture를 둔다. 4단계는 수진의 응답이 반영된 재판정 상태를 열고, 5단계는 확정 완료가 아니라 응답 반영 상태의 최종 확인 화면을 열어 사용자가 직접 확정한다.
- 근거: 핵심 시연은 `기준 설정 → 부족한 응답 식별·요청 → 참석자 응답 → 재판정 → 확정`의 원인과 결과를 보여줘야 한다. 각 번호로 직접 진입해도 같은 장면이 재현되어야 데모 가이드가 상태에 의존하는 화면 바로가기로 퇴행하지 않는다.

### D-024 — 전체 후보 비교표의 첫 열과 하단 범례는 같은 외곽 여백을 공유한다

- 상태: 적용
- 대상: `/host` 전체 후보 비교표
- 문제: 첫 열 헤더와 참석자 행이 서로 다른 작은 좌우 패딩을 사용해 카드 외곽과의 간격이 부족했고, 이름 바로 뒤에 붙는 `필수` 배지는 이름 길이에 따라 위치가 흔들렸다. 마지막 행 아래 범례도 10px 수준의 상하 패딩만 있어 카드 바닥에 붙어 보였다.
- 결정: 첫 열을 180px 고정 열로 두고 헤더·본문의 좌우 패딩을 20px로 통일한다. 이름 영역을 남은 폭만큼 확장해 `필수` 배지를 첫 열 우측 기준선에 정렬한다. 행 높이를 52px로 조정하고 하단 범례도 같은 20px 좌우 기준선과 더 넉넉한 하단 여백을 사용한다.
- 근거: 표에서는 반복되는 셀의 기준선이 배지 자체의 강조보다 우선한다. 역할 표시는 유지하되 고정된 끝선에 배치하면 이름 길이와 무관하게 스캔할 수 있고, 외곽 여백을 통일하면 표가 카드 안에 안정적으로 놓인다.
