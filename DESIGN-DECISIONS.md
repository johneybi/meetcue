# MeetCue Design Decisions

상태: **현행 디자인 방향의 근거 기록**  
목적: 디자인 도구와 실험 결과가 바뀌어도 채택된 판단과 폐기된 판단을 구분한다.

## 1. 역할

`DESIGN.md`가 현재 지켜야 할 규칙을 정의한다면, 이 문서는 그 규칙이 어디에서 왔고 무엇을 의도적으로 채택하지 않았는지 기록한다. 구현자는 이 문서를 새 요구사항 목록으로 사용하지 않는다.

## 2. 소스별 판정

| 소스 | 현재 지위 | 채택 | 채택하지 않음 |
| --- | --- | --- | --- |
| TDS·Toss 제품 UI | 원칙 참고 | 명확한 회색 위계, 절제된 파란색, 직접적인 한국어 문장, 안정적인 조작 크기, 낮은 그림자 의존도 | 공식 TDS 사용 또는 완전한 복제 주장, Toss 브랜드 자산 |
| ASTRYX | 기존 구현 기반 | reset과 접근성 구현에서 얻은 학습 | MeetCue의 시각 기준, 새 컴포넌트의 필수 의존성 |
| OMD | 과거 리뷰·선호 기록 | 카드 남용 억제, 평평한 목록, 읽을 수 있는 한국어 타입, 역할 기반 버튼 크기 | OMD 파이프라인 자체를 디자인 권위로 취급, 당시 리뷰를 현재 화면 요구사항으로 재사용 |
| Creative Production 무드보드 | 시각 탐색 자료 | 차분한 저채도 배경, 낮은 elevation, 파란색을 결정 상태에 집중 | 반투명 frosted shell, bento를 기본 레이아웃으로 사용, glow와 장식적 3D 컨트롤 |
| Figma·Pencil 산출물 | 시점별 화면 참고 | 현재 제품 결정과 일치하며 별도 승인된 구성 | 파일 전체를 source of truth로 지정, 오래된 정적 데이터와 흐름 |
| 현재 코드와 화면 | 리팩터링 기준선 | 검증된 흐름, 정보 위계, 상호작용, 승인된 시각적 관계 | CSS cascade, 중복 override, 우연한 폭·간격·radius, 깨진 반응형 결과 |

## 3. 확정된 시각 방향

- MeetCue는 회의 일정 관리 대시보드가 아니라 주최자의 결정을 돕는 과업형 제품이다.
- 기본 인상은 차분하고 직접적이며 신뢰할 수 있어야 한다.
- 파란색은 선택과 주요 행동에 집중한다.
- 초록·노랑·빨강은 후보의 상태를 설명할 때만 사용한다.
- 추천 후보와 선택 후보는 같은 개념이 아니다. 추천은 근거가 있는 시스템 상태이고, 선택은 사용자의 현재 조작 상태다.
- 한 화면에는 하나의 주 과업 표면만 둔다. 반복 객체가 아닌 페이지 섹션을 카드로 만들지 않는다.
- PC는 모바일을 가운데 확대해 놓은 화면이 아니다. 같은 우선순위를 더 넓고 비교하기 쉬운 구조로 바꾼다.
- 현재 화면의 미감을 보존한다는 것은 픽셀을 복제하는 것이 아니라 승인된 위계와 관계를 보존한다는 뜻이다.

## 4. 명시적으로 폐기한 방향

- TDS를 사용했다고 표현하거나 Toss 화면을 그대로 복제하는 방향
- OMD가 생성한 규칙을 검토 없이 자동 적용하는 방향
- 모든 정보를 bento card로 나누는 대시보드 구성
- 반투명 표면, blur, glow를 제품의 기본 조형으로 사용하는 방향
- 화면의 빈 공간을 요약 카드나 설명 문장으로 채우는 방식
- 현재 CSS 값을 그대로 Tailwind utility로 번역하는 방식

## 5. 변경 절차

새 시각 방향을 도입할 때는 다음 순서로 판단한다.

1. 사용자 과업과 정보 위계를 개선하는가?
2. 기존 승인 결정과 충돌하는가?
3. 공통 토큰 또는 컴포넌트 규칙으로 설명할 수 있는가?
4. 모바일과 PC에서 같은 의미를 유지하는가?
5. 채택한다면 `DESIGN.md`를 수정하고, 배경과 트레이드오프는 이 문서에 남긴다.

도구를 바꾸었다는 이유만으로 디자인 방향을 변경하지 않는다.

## 6. 리팩터링 기준값 결정

- 새 토큰의 초기값보다 현재 승인된 화면의 실제 값이 우선한다.
- 첫 이전 대상인 `HostDecisionMatrix`는 구조를 분리한 뒤 기존 픽셀을 동일한 `--mc-*` alias로 치환한다.
- 9px·11px radius와 2px·5px·6px·10px 간격은 새 규칙이 아니라 시각 동등성을 위한 과도기 값이다.
- 공통 8px·12px radius와 4px 간격 스케일로 정상화하는 작업은 별도 시각 변경으로 검수한다.
