# UX Architecture

상태: **실행 기준 / 조율 가능 시간대 계약 v2.3 반영**
기준일: 2026-07-11
적용 범위: 확정보드 P0 프로토타입의 정보 구조, 화면 템플릿, 상태 전이, 반응형 동작

## 1. 문서 역할과 우선순위

이 문서는 화면을 구현하고 검수할 때 사용하는 단일 UX 실행 명세다. 제품의 이유와 범위를 다시 설명하는 PRD가 아니며, 색상과 타이포그래피를 정의하는 디자인 시스템 문서도 아니다.

충돌 시 우선순위는 다음과 같다.

1. `.local-docs/prd-v1.5.md`: 제품 목표, 사용자, P0 범위
2. `.local-docs/availability-window-model-decision-v2.3.md`: P0 시간 입력, 참석자 응답, 후보 생성의 후속 계약
3. `UX-ARCHITECTURE.md`: IA, 화면 구조, 상태, 내비게이션, 반응형 동작
4. `DESIGN.md`: TDS 기반 시각 언어, 컴포넌트 표현, UX 라이팅
5. 코드: 위 문서를 구현한 결과

`.local-docs/host-time-model-decision-v2.0.md`는 주최자 역할과 초기 시간 구조의 결정 기록이다. `.local-docs/meeting-time-reservation-decision-v2.1.md`는 실제 종료와 일정 확보 시간을 구분하고, `.local-docs/time-quantum-decision-v2.2.md`는 P0의 30분 시간 최소 단위를 정한다. `.local-docs/availability-window-model-decision-v2.3.md`는 정확한 후보 직접 선택을 조율 가능 시간대 수집과 파생 후보 계산으로 대체한다. `.local-docs/ia-v1.md`, `create-*-v1.5~v1.9.md`, `meeting-invite-context-model.md`는 과거 의사결정 근거다. 현재 구현 규칙을 결정할 때 직접 조합하지 않는다. 이 문서에 채택된 결론만 실행 기준으로 사용한다.

## 현재 실행 상태

- 문서 기준 통합: 조율 가능 시간대 계약 v2.3까지 완료
- 역할·상태·라우트 골격: 코드에 존재, 후보 입력형에서 가용시간 수집형으로 전환 필요
- 생성 Focused Task 골격: 반영됨
- 모바일 생성 Bottom Action Bar: 반영됨
- 대표 화면 A `주최자 생성 / 참석자 조건`: 기존 구현 존재, 주최자 역할 계약 재반영 필요
- 대표 화면 A `주최자 생성 / 시간`: 확보 시간 입력은 유지, 현재 후보 블록 선택은 v2.3과 불일치하며 조율 가능 시간대 선택으로 재설계 필요
- 대표 화면 B `참석자 응답`: 후보별 응답을 3상태 가용시간 입력으로 재설계 필요
- 대표 화면 C `주최자 결정`: B 완료 후 진행

다음 작업은 v2.3의 주최자 시간대 선택, 참석자 가용시간 입력, 파생 후보 계산을 하나의 상태 모델로 설계하는 것이다. 현재 `겹침` 표시와 후보 블록 입력 화면은 사고 과정의 프로토타입일 뿐 실행 완성본으로 간주하지 않는다.

## 2. 구조 동결 규칙

아래 네 가지는 현재 구조 기준으로 동결한다.

1. 역할과 사용자 여정
2. 조율 상태와 상태 전이
3. 화면 템플릿과 정보 노출 순서
4. PC·모바일 반응형 계약

구조를 다시 여는 조건은 다음 중 하나다.

- PRD와 명백히 충돌한다.
- 사용자가 핵심 과업을 완료하지 못한다.
- 데이터나 상태 전이가 성립하지 않는다.
- 접근성 또는 기기 제약으로 조작할 수 없다.

다음 문제는 구조를 다시 여는 근거가 아니다.

- 여백이 어색하다.
- 카드가 덜 세련돼 보인다.
- 색상이나 radius가 마음에 들지 않는다.
- 카피가 길거나 짧다.
- 특정 화면만 시각적으로 비어 보인다.

이 문제들은 `DESIGN.md`, 컴포넌트, 토큰, 카피 단계에서 해결한다.

## 3. 제품 경험 모델

제품 경험의 기본 단위는 메뉴나 화면 영역이 아니라 다음 흐름이다.

```text
진입 맥락
→ 사용자 역할
→ 조율 상태
→ 지금 답해야 하는 질문
→ 필요한 정보
→ 하나의 우선 행동
→ 다음 상태
```

제품 원칙:

> 응답은 가볍게, 해석은 제품이, 결정은 주최자가.

역할:

- 주최자: 회의에 참석하며, 요청과 조율 가능 시간대를 만들고, 파생 후보의 응답을 판단해 시간을 확정한다.
- 참석자: 회의 맥락과 조율 범위를 확인하고 자신의 가능 시간대를 빠르게 응답·수정한다.

P0에서 주최자는 항상 참석자다. 주최자가 연 조율 가능 시간대는 주최자가 전체 일정 확보 시간을 비울 수 있는 범위이며, 주최자는 응답 대상이나 미응답자로 계산하지 않는다. 대리 일정 조율 담당자는 별도 역할이 필요한 후속 범위다.

제품은 모든 기능을 둘러보는 대시보드가 아니다. 각 사용자는 자신의 상태에 필요한 다음 행동을 위해 들어온다.

## 4. 상태 모델

### 4.1 진입

| 상태    | 핵심 질문         | 우선 행동        |
| ------- | ----------------- | ---------------- |
| `ENTRY` | 무엇을 할 수 있나 | 회의 시간 정하기 |

### 4.2 주최자

| 상태                     | 핵심 질문                 | 화면 템플릿   | 우선 행동                        |
| ------------------------ | ------------------------- | ------------- | -------------------------------- |
| `HOST_DRAFT`             | 어떤 요청을 만들고 있나   | Focused Task  | 다음 생성 단계                   |
| `HOST_SHARE_READY`       | 무엇을 보내면 되나        | Share         | 링크 복사                        |
| `HOST_WAITING_EMPTY`     | 아직 응답이 없는가        | Host Status   | 요청 문구 복사                   |
| `HOST_WAITING_PARTIAL`   | 누구의 응답이 더 필요한가 | Host Status   | 필요한 사람에게 요청             |
| `HOST_DECISION_READY`    | 지금 정할 수 있는가       | Host Decision | 이 시간으로 정하기               |
| `HOST_REVIEW_NEEDED`     | 확인 후 정할 수 있는가    | Host Decision | 근거 확인 후 확정 또는 보류      |
| `HOST_RECOVERY_REQUIRED` | 무엇을 다시 물어야 하는가 | Recovery      | 새 시간 추가 또는 요청 문구 복사 |
| `HOST_CONFIRMED`         | 무엇을 공유하면 되는가    | Confirmation  | 확정 문구 복사                   |

### 4.3 참석자

| 상태                     | 핵심 질문                         | 화면 템플릿      | 우선 행동    |
| ------------------------ | --------------------------------- | ---------------- | ------------ |
| `PARTICIPANT_NEW`        | 언제 회의 시간을 비울 수 있는가   | Participant Task | 응답 저장    |
| `PARTICIPANT_EDITING`    | 기존 응답을 어떻게 바꿀 것인가    | Participant Task | 수정 저장    |
| `PARTICIPANT_ADDED_ONLY` | 새로 열린 시간대 중 언제 가능한가 | Participant Task | 새 응답 저장 |
| `PARTICIPANT_DONE`       | 응답이 저장되었는가               | Participant Done | 응답 수정    |

### 4.4 후보 상태와 화면 상태

후보 상태는 카드 단위이고 `HOST_*`는 화면 단위다.

| 후보 상태              | 사용자 문구                     | 기본 행동                   |
| ---------------------- | ------------------------------- | --------------------------- |
| `READY_TO_CONFIRM`     | 바로 정할 수 있어요             | 확정                        |
| `HIGH_ADJUSTMENT_RISK` | 정하기 전에 한 번 확인해 주세요 | 근거 확인                   |
| `REQUIRED_PENDING`     | 아직 정하기 어려워요            | 필요한 응답 요청            |
| `BLOCKED`              | 이 시간은 어려워요              | 다른 후보 또는 새 시간 검토 |

화면 상태 우선순위:

1. 확정 가능한 후보가 하나라도 있으면 `HOST_DECISION_READY`다.
2. 확정 가능은 없지만 조정 후 가능한 후보가 있으면 `HOST_REVIEW_NEEDED`다.
3. 필수 응답 부족이 병목이면 `HOST_WAITING_PARTIAL`이다.
4. 모든 후보가 막혔거나 마감 후 성립 기준을 못 맞추면 `HOST_RECOVERY_REQUIRED`다.

주최자는 자신이 연 모든 조율 가능 시간대에서 암묵적으로 참석 가능하다. 별도 응답 레코드를 요구하지 않으며 `missingRequired`, `missingOptional`, 미응답 수에 포함하지 않는다. 파생 후보의 참석 가능 인원과 전체 참석 예정 인원에는 포함한다.

## 5. 라우트 계약

프로토타입에서는 회의 ID를 생략하지만, 라우트는 아래 의미를 유지한다.

| 현재 라우트      | 역할   | 의미                            |
| ---------------- | ------ | ------------------------------- |
| `#/`             | 공통   | Entry                           |
| `#/create`       | 주최자 | Host Create                     |
| `#/share`        | 주최자 | Share Link                      |
| `#/host`         | 주최자 | 상태 기반 Waiting 또는 Decision |
| `#/recover`      | 주최자 | Host Recover                    |
| `#/message`      | 주최자 | Confirmation Message            |
| `#/invite`       | 참석자 | Participant New                 |
| `#/invite/edit`  | 참석자 | Participant Editing             |
| `#/invite/added` | 참석자 | Participant Added Only          |
| `#/invite/done`  | 참석자 | Participant Done                |

라우트는 상태를 여는 주소다. 라우트 목록을 사용자가 보는 전역 메뉴로 만들지 않는다.

## 6. 셸과 내비게이션

### Entry Shell

- 제품의 용도를 한 문장으로 설명한다.
- 회의 만들기 CTA 하나를 우선한다.
- 초대 링크는 받은 링크에서 바로 응답한다는 보조 안내만 둔다.
- 후보, 결과, 응답 현황을 미리 보여주지 않는다.

### Host Shell

- 제품 정체성, 회의 맥락, 현재 상태에 필요한 보조 행동만 제공한다.
- `HOST_DRAFT`에서는 기능 내비게이션을 노출하지 않는다.
- 링크 생성 뒤에만 `응답 링크 보기`, `응답 상태로 돌아가기` 같은 상태 기반 행동을 제공한다.
- `후보 추가`, `결과 보기`처럼 기능명을 나열하지 않는다.
- 화면의 우선 CTA는 셸이 아니라 본문이 소유한다.

### Participant Shell

- Host Shell과 완전히 분리한다.
- 회의 맥락과 자신의 응답만 제공한다.
- 주최자 상태, 다른 참석자 응답, 후보 판정 근거로 이동할 수 없다.

### 뒤로 이동

- 첫 단계에는 이전 행동을 렌더링하지 않는다.
- 생성 2단계부터 상단 뒤로 이동을 제공한다.
- 입력 상태는 뒤로 이동해도 유지한다.
- 브라우저 뒤로가기와 화면 내 뒤로가기가 서로 다른 결과를 만들지 않는다.

## 7. 화면 템플릿

### 7.1 Focused Task

용도: 생성처럼 여러 판단을 순서대로 완료하는 흐름.

구조:

```text
진행 상태
현재 단계 제목
한 문장 안내
단일 과업 표면
현재 입력
현재 우선 CTA
```

규칙:

- 한 화면에 현재 상위 단계 하나만 노출한다.
- 미래 단계의 제목이나 내용을 본문 아래에 나열하지 않는다.
- 진행 표시는 위치만 알려주며 내비게이션을 대신하지 않는다.
- 제목, 입력, CTA는 하나의 읽기 축을 사용한다.
- 전체 요청 요약은 최종 확인 전까지 상시 노출하지 않는다.

### 7.2 Participant Task

용도: 로그인 없이 조율 범위 안의 가능 시간대를 남기거나 수정하는 흐름.

구조:

```text
최소 회의 맥락
응답 마감과 수정 가능 안내
조율 가능 시간대 그리드
시간대별 3가지 가용 상태
저장 CTA
```

규칙:

- 참석자의 핵심 과업보다 회의 설명이 더 길어지지 않는다.
- 다른 참석자의 응답을 보여주지 않는다.
- 시간대 확장 상태에서는 기존 응답을 읽기 상태로 낮추고 새로 열린 시간대만 활성화한다.

### 7.3 Host Status

용도: 응답이 충분하지 않은 주최자에게 현재 상태와 다음 행동을 알려준다.

구조:

```text
현재 상태 결론
응답 현황
기다리는 이유 또는 필요한 사람
하나의 우선 행동
```

규칙:

- 후보 비교 카드를 기본 구조로 사용하지 않는다.
- 진단만 하고 끝내지 않는다.
- 자동 발송처럼 보이지 않도록 복사와 수동 공유를 분명히 한다.

### 7.4 Host Decision

용도: 응답을 해석해 확정 가능한 후보와 다음 행동을 보여준다.

구조:

```text
결론과 가장 중요한 다음 행동
후보 상태 카드 목록
선택한 후보의 판단 근거
확정 또는 회복 행동
```

규칙:

- 원본 응답표보다 제품의 해석을 먼저 보여준다.
- 후보 카드는 이 템플릿에서만 핵심 반복 객체다.
- 상세 근거는 별도 전역 화면이 아니라 선택한 후보의 disclosure다.
- 제품이 자동 확정하지 않는다.

### 7.5 Share / Confirmation

용도: 주최자가 링크 또는 확정 메시지를 외부로 전달한다.

규칙:

- 복사할 대상과 복사 CTA가 첫 시야에 있어야 한다.
- 자동 전송 완료처럼 말하지 않는다.
- 정보 수정과 상태 확인은 보조 행동으로 둔다.

### 7.6 Recovery

용도: 첫 조율 범위의 응답으로 결정하지 못했을 때 기존 조율을 유지하며 다시 묻는다.

규칙:

- 기존 링크와 응답이 유지된다는 사실을 먼저 확인시킨다.
- 상황별 요청 문구 복사 또는 새 시간 추가를 제공한다.
- 새로 열린 시간대만 미응답 상태로 연다.
- 새 회의를 다시 만들게 하지 않는다.

## 8. 생성 흐름 계약

상위 단계는 네 개로 고정한다.

```text
회의 안내 → 참석자 → 시간 → 확인
```

### 8.1 회의 안내

목표: 참석자가 응답 전에 이해할 최소 회의 맥락을 만든다.

필수:

- 회의 이름
- 이 회의에서 확인하거나 결정할 내용 한 문장
- 주최자 또는 팀

선택:

- 관련 자료 이름 또는 링크

결정:

- 업무 맥락을 추상적인 칩 분류로 먼저 묻지 않는다.
- 어떤 일과 관련된지는 회의 이름과 한 문장 목적에서 구체적으로 드러나게 한다.
- 관련 자료는 기본으로 접어두며, 응답 전 필수 과제로 보이지 않게 한다.
- 플레이스홀더와 도움말이 같은 내용을 반복하지 않는다.

### 8.2 참석자

내부 순서는 세 판단으로 고정한다.

1. 누구에게 시간을 물을지 정한다.
2. 이 회의는 모두 참석해야 하는지 정한다.
3. 일부가 빠져도 된다면 꼭 필요한 사람과 최소 참석 인원을 정한다.

규칙:

- 주최자의 참석 여부를 묻지 않는다. P0에서 주최자는 항상 참석한다.
- 초대할 사람 수는 선택한 사람의 합계다. 전체 참석 예정 인원은 주최자를 포함해 `초대할 사람 수 + 1`이다.
- 주최자를 검색 결과나 선택한 사람 목록에 다시 넣지 않는다.
- `나도 참석해요` 토글을 제공하지 않는다.
- 첫 판단과 두 번째 판단을 동시에 펼치지 않는다.
- 사람 선택 완료 후 첫 판단은 짧은 요약으로 접고 참석 조건을 연다.
- 내부 판단에 별도 단계 번호, 진행 노드, `진행 중` 캡션을 사용하지 않는다.
- 검색 전 보조 목록은 `최근 함께한 사람`으로 표시한다.
- 프로토타입에서 추천 알고리즘이 있는 것처럼 말하지 않는다.
- 선택 결과는 검색 결과와 분리하고, 인원 수·체크·선택 목록으로 명확히 보여준다.

참석 조건 질문:

```text
이 회의는 모두 참석해야 하나요?
```

- 모두 참석해야 해요
- 몇 명은 빠져도 진행할 수 있어요

여기서 `모두`는 주최자와 초대할 사람 전체를 뜻한다. 주최자는 항상 참석 가능하므로 별도 응답이나 재선택을 요구하지 않는다.

`몇 명은 빠져도 진행`을 선택한 경우에만 꼭 참석해야 하는 사람과 최소 참석 인원을 순서대로 정한다. 최소 참석 인원은 주최자와 꼭 필요한 사람 수보다 작을 수 없고, 전체 참석 예정 인원보다 클 수 없다. 이 기준이 있어야 제품이 후보를 인기순으로 보여주는 대신 회의 성립 여부로 판정할 수 있다.

### 8.3 시간

내부 순서는 세 판단으로 고정한다.

```text
회의 가능 기간과 확보할 시간
→ 조율 가능 시간대
→ 응답 마감
```

#### 회의 가능 기간과 확보할 시간

- 회의 가능 기간은 회의를 실제로 실시할 수 있는 시작일과 종료일이다.
- 확보할 시간은 계산된 후보 전체에 참여하기 위해 일정에서 연속으로 비워둘 유한한 시간 블록이다. 실제 회의가 정확히 그 시각에 끝난다는 뜻이 아니다.
- P0의 시간 최소 단위는 30분이다.
- 확보 시간은 30분의 양의 배수이며, 30분·60분·90분·120분은 각각 1~4칸으로 표현한다.
- 실제 예상 시간이 단위 사이에 있으면 다음 30분 경계로 올려 확보한다.
- 초기에는 값을 선택하지 않고 30분·60분·90분·120분을 빠른 선택으로 제공한다.
- 직접 입력을 선택하면 빠른 선택은 접고 30분 단위 스테퍼와 숫자 입력만 펼친다.
- 직접 입력 컨테이너는 선택 카드처럼 강조하지 않고, 현재 값과 증감 버튼에만 조작 위계를 준다.
- 스테퍼 위에는 낮은 위계의 `직접 입력` 라벨과 닫기 아이콘을 둔다.
- 닫기 아이콘과 `Esc`는 직접 입력값을 취소하고 시간 칩을 다시 표시하며, 포커스를 `직접 입력` 버튼으로 돌려보낸다.
- 모든 파생 후보는 같은 확보 시간을 사용한다.
- 이 값은 주최자의 개인 가능 시간을 입력하는 화면이 아니다.
- 두 값을 정하기 전에는 조율 가능 시간대 그리드를 펼치지 않는다.
- 완료하면 `7월 13일~17일 · 1시간 확보`처럼 한 줄로 접고 시간대 선택을 연다.

#### 조율 가능 시간대

- 주최자는 회의 가능 기간 안에서 자신이 전체 확보 시간을 비울 수 있는 연속 범위를 선택한다.
- PC와 모바일의 그리드 한 칸은 30분 단위다.
- 선택 범위는 확보 시간보다 짧을 수 없다.
- 한 날짜에 여러 범위를 만들 수 있고, 맞닿은 범위는 하나로 합칠 수 있다.
- PC는 주간 그리드에서 클릭과 키보드로 범위를 만들고, 드래그로 길이를 조정할 수 있다.
- 모바일은 날짜를 먼저 고른 뒤 시작과 끝을 선택하거나 연속 칸을 탭한다.
- 드래그는 보조 입력으로만 허용하며 모든 동작에 클릭·탭 대안을 제공한다.
- 선택된 시간대 수와 범위 목록을 즉시 보여주고 목록에서도 수정·삭제할 수 있다.
- 가능한 시간대가 0개면 다음 판단으로 갈 수 없다.
- 생성 단계에서는 정확한 후보 개수나 시작 시각을 요구하지 않는다.
- 현재 구현의 고정 길이 후보 블록, 중복 금지, `겹침` 표시는 이 계약에 포함하지 않는다.

#### 참석자 가용시간과 파생 후보

- 참석자는 주최자가 연 시간대 안에서 각 30분 칸을 `가능`, `조정하면 가능`, `어려움`으로 표시한다.
- 제품은 조율 시간대 안에서 30분 간격으로 시작 시각을 이동하며 확보 시간 전체가 들어오는 후보를 계산한다.
- 후보 칸 중 하나라도 명시적 `어려움`이 있으면 해당 참석자의 후보 상태는 `어려움`이다.
- 명시적 `어려움`은 없지만 하나라도 미응답이면 해당 참석자는 그 후보에 미응답이다.
- 후보의 모든 칸이 가능하면 참석자 상태는 `가능`이다.
- 모든 칸이 가능 또는 조정 가능이고 하나 이상이 조정 가능이면 `조정하면 가능`이다.
- 파생 후보는 서로 겹칠 수 있으며 입력 오류로 취급하지 않는다.
- 후보별 파생 상태를 기존 필수 참석자, 최소 참석 인원, 조정 필요, 미응답 판정에 사용한다.

#### 응답 마감

- 응답 마감은 참여자가 후보에 답해야 하는 시점이다.
- 회의 가능 기간 또는 회의 실시 기한과 같은 값으로 취급하지 않는다.
- 첫 조율 가능 시간대보다 앞선 시점만 선택할 수 있다.
- 시스템이 임의의 마감을 확정하지 않는다. 권장값은 제안할 수 있지만 주최자가 확인해야 한다.

시간 단계는 조작 면적이 필요하므로 PC에서 다른 생성 단계보다 넓어질 수 있다. 하위 판단은 동시에 펼치지 않고 완료된 결과만 짧게 접어 유지한다.

### 8.4 확인

- 참석자에게 보일 정보 순서로 최종 요청을 보여준다.
- 회의 맥락, 주최자를 포함한 전체 참석 예정 인원, 참석 조건, 회의 가능 기간, 확보 시간, 조율 가능 시간대, 응답 마감을 확인한다.
- 각 영역은 수정 진입점을 제공할 수 있다.
- 내부 프리셋명이나 판정 로직을 보여주지 않는다.
- 우선 CTA는 응답 링크 만들기다.

## 10. 반응형 계약

PC와 모바일은 같은 상태와 데이터를 사용하지만 같은 화면을 축소하지 않는다.

### PC

- 앱 셸 최대 폭은 1440px다.
- 집중형 입력 단계의 과업 폭은 720~760px를 기준으로 한다.
- 확인 단계는 약 820px까지 확장할 수 있다.
- 시간 그리드는 960~1040px까지 확장할 수 있다.
- 한 상태 안에서 `1040 → 820 → 680`처럼 여러 가시 컨테이너를 중첩하지 않는다.
- 현재 제목, 과업 표면, 입력, CTA는 같은 정렬축을 공유한다.
- 생성 CTA는 본문 아래 왼쪽에 둔다.
- 빈 공간을 반복 요약이나 장식으로 채우지 않는다.

### 모바일

- 760px 이하에서 단일 열을 사용한다.
- 페이지 가로 여백은 16px을 기준으로 한다.
- 현재 상위 단계 하나만 보여준다.
- 제품 단계 CTA는 하단 고정 액션 바에서 제공한다.
- CTA는 현재 하위 과업에 따라 바뀐다.
  - 사람 선택 중: `N명 선택 완료`
  - 참석 조건 선택 후: `회의 시간 정하기`
  - 시간 조건 입력 후: `가능한 시간대 고르기`
  - 시간대 선택 중: `시간대 선택 완료`
  - 응답 마감 입력 후: `요청 내용 확인하기`
- 하단 액션 바는 safe area를 포함하며 본문을 가리지 않는다.
- 사람 검색은 전체 화면 레이어 또는 바텀시트로 제공한다.
- 사람 검색 레이어가 열리면 배경의 하단 CTA를 숨긴다.
- 시간 그리드를 축소하거나 가로 스크롤시키지 않고 날짜 스트립과 날짜별 슬롯으로 전환한다.
- 날짜 직접 선택은 접근 가능한 캘린더 바텀시트로 제공한다.

### 공통 접근성

- 터치 대상은 최소 44px이다.
- 선택 상태는 색상만으로 표현하지 않는다.
- 관련 입력은 `fieldset`과 `legend` 또는 동등한 의미 구조로 묶는다.
- 단계 이동 후 새 제목에 포커스를 이동한다.
- `prefers-reduced-motion`에서는 자동 스크롤과 전환 애니메이션을 줄인다.

## 12. 대표 품질 기준 화면

전체 화면을 동시에 고도화하지 않는다. 다음 세 상태를 제품의 품질 기준으로 먼저 완성한다.

### A. 주최자 생성 / 참석자 조건

경로: `#/create`의 참석자 단계  
검증 범위: 과업 표면, 사람 검색, 선택 상태, 점진적 공개, PC·모바일 CTA

완료 기준:

- 누구를 선택했는지 즉시 알 수 있다.
- 사람 선택과 참석 조건이 동시에 경쟁하지 않는다.
- 모바일에서 하단 CTA와 바텀시트 CTA가 겹치지 않는다.
- PC와 모바일의 관계가 같은 상태 모델로 설명된다.

### B. 참석자 응답

경로: `#/invite`  
검증 범위: 최소 회의 맥락, 시간대 스캔, 3상태 가용시간 입력, 저장과 수정 가능 안내

완료 기준:

- 5초 안에 어떤 회의인지 이해할 수 있다.
- 다른 참석자의 상태 없이 자신의 응답만 빠르게 남길 수 있다.
- `일정 조정하면 가능해요`의 의미가 오해되지 않는다.
- 모바일에서 한 손으로 주요 응답과 저장을 수행할 수 있다.

### C. 주최자 결정

경로: `#/host`의 결정 가능 상태  
검증 범위: 결론 우선 위계, 후보 카드, 판단 근거, 확정 CTA, 회복 행동

완료 기준:

- 첫 시야에서 지금 확정할 수 있는지 알 수 있다.
- 원본 표를 계산하지 않고 다음 행동을 결정할 수 있다.
- 후보 상태와 화면 전체 상태를 혼동하지 않는다.
- 확정과 회복 행동의 우선순위가 명확하다.

이 세 화면의 컴포넌트와 시각 규칙을 확정한 뒤 Share, Waiting, Recovery, Message, Edit, Added, Done에 확장한다.

## 13. 실행 순서

### Gate 1. 구조 정합성

1. 현재 라우트를 상태 모델에 연결한다.
2. 각 화면이 정확히 하나의 템플릿을 사용하게 한다.
3. 역할별 셸을 분리한다.
4. 모든 핵심 CTA와 상태 전이를 연결한다.

통과 조건: Entry부터 생성, 응답, 판단, 회복, 확정 공유까지 링크가 끊기지 않는다.

### Gate 2. 대표 화면 완성

1. 생성 / 참석자 조건
2. 참석자 응답
3. 주최자 결정

각 화면은 구조, 카피, 상태, 반응형, 접근성을 함께 완료한다. 세 화면을 번갈아 조금씩 고치지 않는다.

### Gate 3. 시스템화

1. 대표 화면에서 반복된 컴포넌트를 공통화한다.
2. `DESIGN.md` 토큰과 상태 표현을 적용한다.
3. 나머지 화면에 같은 문법을 확장한다.

### Gate 4. 전체 플로우 QA

1. 새 사용자 생성
2. 참석자 첫 응답
3. 참석자 응답 수정
4. 주최자 대기와 판단
5. 회복과 새로 열린 시간대 응답
6. 확정 메시지 복사

## 14. Definition of Done

화면은 다음을 모두 만족해야 완료다.

### 제품

- 사용자의 현재 질문이 하나로 설명된다.
- 첫 시야에 하나의 우선 CTA가 있다.
- 행동 후 다음 상태가 실제로 연결된다.
- 실패·대기 상태에도 수동 다음 행동이 있다.

### 정보 구조

- 역할에 필요하지 않은 정보가 노출되지 않는다.
- 내부 용어와 기능 목록이 내비게이션이나 카피에 나타나지 않는다.
- 동일 정보를 좌우 또는 상하에서 반복하지 않는다.

### 시각

- 제목, 본문, 입력, CTA의 정렬축이 설명 가능하다.
- 카드 안에 페이지 섹션용 카드를 중첩하지 않는다.
- 간격, radius, 타입 위계가 `DESIGN.md`와 일치한다.
- 선택, 대기, 오류, 완료 상태가 구분된다.

### 반응형

- 모바일은 PC를 단순 축소하지 않는다.
- 텍스트와 버튼이 잘리거나 겹치지 않는다.
- 모바일 고정 CTA가 본문·키보드·바텀시트와 충돌하지 않는다.
- 시간 선택이 마우스와 터치에서 모두 가능하다.

### 기술

- 핵심 링크와 버튼이 동작한다.
- 키보드 조작과 포커스 순서가 자연스럽다.
- `npm run lint`와 `npm run build`가 통과한다.

## 15. 보류 항목

아래 항목은 현재 구조 확정을 막지 않는다.

- 조직도 또는 실제 사내 디렉터리 연동
- 비참석 일정 조율 담당자와 대리 일정 생성
- 캘린더 연동
- 자동 알림과 자동 메시지 발송
- AI 기반 후보 추천
- 반복 회의
- 회의실 예약
- AI 기반 참석 가능성 예측
- 세밀한 분석 대시보드

보류 항목을 프로토타입 화면의 설명용 UI로 미리 만들지 않는다.
