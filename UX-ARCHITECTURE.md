# UX Architecture

상태: **실행 기준 / P0 제출 범위 잠금 v2.10 반영**
기준일: 2026-07-12
적용 범위: MeetCue P0 프로토타입의 정보 구조, 화면 템플릿, 상태 전이, 반응형 동작

## 1. 문서 역할과 우선순위

이 문서는 화면을 구현하고 검수할 때 사용하는 단일 UX 실행 명세다. 제품의 이유와 범위를 다시 설명하는 PRD가 아니며, 색상과 타이포그래피를 정의하는 디자인 시스템 문서도 아니다.

충돌 시 우선순위는 다음과 같다.

1. `.local-docs/p0-submission-scope-lock-v2.10.md`: 제출 범위, 보류 범위, 대표 시연 순서
2. `.local-docs/sprint-decision-coherence-v2.2.1.md`: 스프린트 목표와 완료 게이트
3. `.local-docs/prd-v2.2.1.md`: 제품 목표, 사용자, P0 범위, 검증 계약
4. `.local-docs/decision-model-v2.2.md`: 후보 판정, 정렬, 설명, 다음 행동 규칙
5. `.local-docs/availability-window-model-decision-v2.3.md`: 참석자 응답과 후보 생성 계약
6. `.local-docs/host-search-scope-input-decision-v2.4.md`: 주최자 기본 탐색 범위와 예외 편집 계약
7. `.local-docs/participant-availability-input-decision-v2.6.md`: 참여자 더미 캘린더, 타임테이블과 예외 편집 계약
8. `UX-ARCHITECTURE.md`: IA, 화면 구조, 상태, 내비게이션, 반응형 동작
9. `DESIGN.md`: MeetCue 시각 언어, 컴포넌트 표현, UX 라이팅
10. `DESIGN-DECISIONS.md`: 과거 디자인 소스에서 채택·폐기한 판단
11. `DESIGN-REFACTOR-BASELINE.md`: 리팩터링 중 보존·정상화·격리 기준
10. 코드: 위 문서를 구현한 결과

`.local-docs/host-time-model-decision-v2.0.md`는 주최자 역할과 초기 시간 구조의 결정 기록이다. `.local-docs/meeting-time-reservation-decision-v2.1.md`는 실제 종료와 일정 확보 시간을 구분하고, `.local-docs/time-quantum-decision-v2.2.md`는 P0의 30분 시간 최소 단위를 정한다. `.local-docs/availability-window-model-decision-v2.3.md`는 정확한 후보 직접 선택을 가용시간 수집과 파생 후보 계산으로 대체한다. `.local-docs/host-search-scope-input-decision-v2.4.md`는 주최자가 빈 그리드를 필수로 칠하는 구조를 제품이 제안한 탐색 범위의 예외 편집으로 대체한다. `.local-docs/participant-availability-input-decision-v2.6.md`는 더미 캘린더 자동 채움과 역할별 타임테이블 예외 편집을 채택한다. v2.7~v2.9의 개인 링크, 조직 계정 전달, 계정 내비게이션은 후속 제품 탐색 기록이며 P0 완료 게이트에 직접 조합하지 않는다. 현재 구현 규칙은 `.local-docs/p0-submission-scope-lock-v2.10.md`에 채택된 결론을 우선한다.

## 현재 실행 상태

- 문서 기준 통합: P0 제출 범위 잠금 v2.10 반영 완료
- 계정 셸과 콘텐츠: 후속 제품 탐색 코드로 보존하며 DEV 화면에서만 접근
- 전역 앱 셸: 동일한 프레임 토큰 위에서 제출용 주최자 화면은 집중 헤더, DEV 계정 화면은 계정 헤더를 사용하며 일반 콘텐츠 960px, wide 결과 1120px, PC 24px·모바일 18px 거터 계약을 반영함
- 역할·상태·라우트 골격: 가용시간 수집형 기준으로 반영됨
- 생성 Focused Task 골격: 반영됨
- 모바일 생성 Bottom Action Bar: 반영됨
- 대표 화면 A `주최자 생성 / 참석자 조건`: 반영됨
- 대표 화면 A `주최자 생성 / 시간`: 파생 후보 계산은 반영됨. 기본 탐색 범위와 예외 편집은 구현 대기
- 대표 화면 B `참석자 응답`: 더미 캘린더 반영, 반응형 타임테이블, 30분 단위 3상태 예외 편집과 저장 경계 반영됨
- 대표 화면 C `주최자 결정`: READY·PENDING·IMPOSSIBLE을 같은 결과 보드에 통합
- 이어서 조율: 한 사람의 응답 추가·수정과 재판정만 P0에 포함
- 새 시간 범위 추가: P0 제외

다음 작업은 실제 사용성 검증으로 첫 결정 시간·올바른 후속 행동·불필요한 대기 감소를 확인하는 것이다.

## 2. 구조 동결 규칙

아래 네 가지는 현재 구조 기준으로 동결한다.

1. 역할과 사용자 여정
2. 조율 상태와 상태 전이
3. 화면 템플릿과 정보 노출 순서
4. PC·모바일 반응형 계약

구조를 다시 여는 조건은 다음 중 하나다.

- PRD와 명백히 충돌한다.
- 사용자가 핵심 과업을 완료하지 못한다.
- 데이터나 상태 전이가 성립하지 않는다.
- 접근성 또는 기기 제약으로 조작할 수 없다.

다음 문제는 구조를 다시 여는 근거가 아니다.

- 여백이 어색하다.
- 카드가 덜 세련돼 보인다.
- 색상이나 radius가 마음에 들지 않는다.
- 카피가 길거나 짧다.
- 특정 화면만 시각적으로 비어 보인다.

이 문제들은 `DESIGN.md`, 컴포넌트, 토큰, 카피 단계에서 해결한다.

## 3. 제품 경험 모델

제품 경험의 기본 단위는 메뉴나 화면 영역이 아니라 다음 흐름이다.

```text
진입 맥락
→ 사용자 역할
→ 조율 상태
→ 지금 답해야 하는 질문
→ 필요한 정보
→ 하나의 우선 행동
→ 다음 상태
```

제품 원칙:

> 응답은 가볍게, 해석은 제품이, 결정은 주최자가.

역할:

- 주최자: 회의에 참석하며, 제품이 제안한 탐색 범위를 확인·교정하고, 파생 후보의 응답을 판단해 시간을 확정한다.
- 참석자: 회의 맥락과 조율 범위를 확인하고 자신의 가능 시간대를 빠르게 응답·수정한다.

P0에서 주최자는 항상 참석자다. 제품이 제안한 기본 탐색 범위를 확인하고 계속한 뒤 남은 조율 가능 시간대는 주최자가 전체 일정 확보 시간을 비울 수 있는 범위이며, 주최자는 응답 대상이나 미응답자로 계산하지 않는다. 대리 일정 조율 담당자는 별도 역할이 필요한 후속 범위다.

제품은 모든 기능을 둘러보는 대시보드가 아니다. 각 사용자는 자신의 상태에 필요한 다음 행동을 위해 들어온다.

## 4. 상태 모델

### 4.1 P0 진입

| 상태         | 핵심 질문                      | 우선 행동           |
| ------------ | ------------------------------ | ------------------- |
| `HOST_DRAFT` | 어떤 회의 기준을 설정할 것인가 | 최소 생성 흐름 시작 |

기본 URL과 알 수 없는 URL은 `#/create`로 진입한다. 제출용 전역 헤더에는 계정 메뉴를 노출하지 않는다.

### 4.1.1 후속 제품 계정 진입 · DEV 전용

| 상태            | 핵심 질문                      | 우선 행동                          |
| --------------- | ------------------------------ | ---------------------------------- |
| `ACCOUNT_HOME`  | 지금 내가 처리할 일은 무엇인가 | 받은 요청 응답 또는 회의 결과 확인 |
| `MY_MEETINGS`   | 내가 만든 회의는 어떻게 됐나   | 진행 중 회의 다시 열기             |
| `MY_REQUESTS`   | 내가 답할 요청은 무엇인가      | 내 시간 응답하기                   |
| `NOTIFICATIONS` | 무엇이 새로 바뀌었나           | 관련 요청 또는 회의 열기           |

이 계정 구조는 후속 제품 방향을 검토하기 위한 DEV 전용 화면이다. P0 완료 게이트와 평가용 기본 경로에는 포함하지 않는다.

### 4.2 주최자

| 상태                 | 핵심 질문               | 화면 템플릿   | 우선 행동                |
| -------------------- | ----------------------- | ------------- | ------------------------ |
| `HOST_DRAFT`         | 어떤 요청을 만들고 있나 | Focused Task  | 다음 생성 단계           |
| `HOST_SHARE_READY`   | 누구에게 요청했나       | Request Sent  | 응답 현황 보기           |
| `HOST_WAITING_EMPTY` | 아직 응답이 없는가      | Host Status   | 미응답자 다시 알리기     |
| `HOST_DECISION`      | 지금 정할 수 있는가     | Host Decision | 확정·응답 요청·기준 검토 |
| `HOST_CONFIRMED`     | 무엇을 공유하면 되는가  | Confirmation  | 확정 알림 보내기         |

### 4.3 참석자

| 상태                    | 핵심 질문                       | 화면 템플릿      | 우선 행동      |
| ----------------------- | ------------------------------- | ---------------- | -------------- |
| `PARTICIPANT_NEW`       | 언제 회의 시간을 비울 수 있는가 | Participant Task | 응답 저장      |
| `PARTICIPANT_EDITING`   | 기존 응답을 어떻게 바꿀 것인가  | Participant Task | 수정 저장      |
| `PARTICIPANT_DONE`      | 응답이 저장되었는가             | Participant Done | 응답 수정      |
| `PARTICIPANT_CONFIRMED` | 언제 회의가 열리는가            | Confirmation     | 확정 시간 확인 |

### 4.4 후보 상태와 화면 상태

후보 상태는 카드 단위이고 `HOST_*`는 화면 단위다.

| 후보 상태    | 사용자 문구             | 기본 행동                     |
| ------------ | ----------------------- | ----------------------------- |
| `READY`      | 지금 정할 수 있어요     | 확정                          |
| `PENDING`    | 응답이 더 필요해요      | 요청 대상 선택                |
| `IMPOSSIBLE` | 다른 시간을 찾아야 해요 | 다른 후보 또는 회의 기준 검토 |

화면 상태 우선순위:

1. 응답이 한 건도 없으면 `HOST_WAITING_EMPTY`다.
2. 응답이 한 건 이상 있으면 후보 상태 구성과 무관하게 `HOST_DECISION`이다.
3. 결과 보드는 READY, PENDING, IMPOSSIBLE 후보군을 같은 구조로 보여준다.
4. 마감 경과는 화면 상태를 바꾸지 않고 결과 보드의 보조 상태로만 표시한다.

주최자는 자신이 확인한 모든 조율 가능 시간대에서 암묵적으로 참석 가능하다. 별도 응답 레코드를 요구하지 않으며 `missingRequired`, `missingOptional`, 미응답 수에 포함하지 않는다. 파생 후보의 참석 가능 인원과 전체 참석 예정 인원에는 포함한다.

## 5. 라우트 계약

프로토타입에서는 회의 ID를 생략하지만, 라우트는 아래 의미를 유지한다.

| 현재 라우트            | 역할   | 의미                        |
| ---------------------- | ------ | --------------------------- |
| `#/`                   | 주최자 | `#/create`로 정규화         |
| `#/home`               | DEV    | Post-P0 Account Home        |
| `#/meetings`           | DEV    | Post-P0 My Meetings         |
| `#/requests`           | DEV    | Post-P0 Received Requests   |
| `#/notifications`      | DEV    | Post-P0 Notification Events |
| `#/create`             | 주최자 | Host Create                 |
| `#/share`              | 주최자 | Request Sent                |
| `#/host`               | 주최자 | Waiting 또는 통합 Decision  |
| `#/message`            | 주최자 | Confirmation Message        |
| `#/invite`             | 참석자 | Invalid Invite              |
| `#/invite/:token`      | 참석자 | Participant New             |
| `#/invite/:token/edit` | 참석자 | Participant Editing         |
| `#/invite/:token/done` | 참석자 | Participant Done            |

계정 라우트는 개발 환경에서만 접근한다. 생성·응답·결정 라우트가 P0의 제출용 경험이며, `current` DEV 전환은 회의와 선택 후보를 보존해 참여자 제출 뒤 같은 결과 보드를 다시 계산한다.

## 6. 셸과 내비게이션

### Entry Shell

- 제품의 용도를 한 문장으로 설명한다.
- 회의 만들기 CTA 하나를 우선한다.
- 요청받은 사람은 앱 알림에서 바로 응답한다는 보조 안내만 둔다.
- 후보, 결과, 응답 현황을 미리 보여주지 않는다.

### Host Shell

- 제품 정체성, 회의 맥락, 현재 상태에 필요한 보조 행동만 제공한다.
- `HOST_DRAFT`에서는 기능 내비게이션을 노출하지 않는다.
- 요청 전송 뒤에만 `요청 현황 보기`, `응답 상태로 돌아가기` 같은 상태 기반 행동을 제공한다.
- `후보 추가`, `결과 보기`처럼 기능명을 나열하지 않는다.
- 화면의 우선 CTA는 셸이 아니라 본문이 소유한다.

### Participant Shell

- Host Shell과 완전히 분리한다.
- 실제 제품은 로그인된 조직 계정과 배정된 요청으로 참석자를 식별한다.
- 프로토타입의 개인 토큰은 계정 세션을 대신하는 DEV 전용 페르소나 라우트다.
- 참석자에게 자기 이름을 선택하게 하거나 다른 참석자 목록을 노출하지 않는다.
- 토큰이 없거나 일치하지 않으면 잘못된 초대 링크 상태만 보여준다.
- 회의 맥락과 자신의 응답만 제공한다.
- 주최자 상태, 다른 참석자 응답, 후보 판정 근거로 이동할 수 없다.
- 프로토타입의 페르소나 전환은 제품 화면이 아니라 개발용 화면 전환 도구가 소유한다.

### 뒤로 이동

- 첫 단계에는 이전 행동을 렌더링하지 않는다.
- 생성 2단계부터 상단 뒤로 이동을 제공한다.
- 입력 상태는 뒤로 이동해도 유지한다.
- 브라우저 뒤로가기와 화면 내 뒤로가기가 서로 다른 결과를 만들지 않는다.

## 7. 화면 템플릿

### 7.1 Focused Task

용도: 생성처럼 여러 판단을 순서대로 완료하는 흐름.

구조:

```text
진행 상태
현재 단계 제목
한 문장 안내
단일 과업 표면
현재 입력
현재 우선 CTA
```

규칙:

- 한 화면에 현재 상위 단계 하나만 노출한다.
- 미래 단계의 제목이나 내용을 본문 아래에 나열하지 않는다.
- 진행 표시는 위치만 알려주며 내비게이션을 대신하지 않는다.
- 제목, 입력, CTA는 하나의 읽기 축을 사용한다.
- 전체 요청 요약은 최종 확인 전까지 상시 노출하지 않는다.

### 7.2 Participant Task

용도: 로그인 없이 조율 범위 안의 가능 시간대를 남기거나 수정하는 흐름.

구조:

```text
최소 회의 맥락
응답 마감과 수정 가능 안내
조율 가능 시간대 그리드
시간대별 3가지 가용 상태
저장 CTA
```

규칙:

- 참석자의 핵심 과업보다 회의 설명이 더 길어지지 않는다.
- 다른 참석자의 응답을 보여주지 않는다.
- 첫 입력부터 제출 전까지 사람 상태는 초안이며 주최자 판정에는 미응답으로 보인다.
- 이미 제출한 응답을 수정 중이면 마지막 제출본은 새 초안을 저장할 때까지 주최자 판정에 유지한다.
- 제출 뒤에는 기존 칸을 수정하고 같은 범위 안에서 다시 저장할 수 있다.

### 7.3 Host Status

용도: 아직 제출된 응답이 한 건도 없는 주최자에게 현재 상태와 다음 행동을 알려준다.

구조:

```text
현재 상태 결론
응답 현황
기다리는 이유 또는 필요한 사람
하나의 우선 행동
```

규칙:

- 후보 비교 카드를 기본 구조로 사용하지 않는다.
- 진단만 하고 끝내지 않는다.
- 자동 발송처럼 보이지 않도록 복사와 수동 공유를 분명히 한다.

### 7.4 Host Decision

용도: 응답을 해석해 확정 가능한 후보와 다음 행동을 보여준다.

구조:

```text
현재 결정 상태와 하나의 우선 행동
가장 행동 가능한 후보
상태별로 묶인 나머지 후보
선택한 후보의 이유와 영향 인원
필요할 때 사람별 상세 응답
```

규칙:

- 원본 응답표보다 제품의 해석을 먼저 보여준다.
- 후보 카드는 이 템플릿에서만 핵심 반복 객체다.
- 상세 근거는 별도 전역 화면이 아니라 선택한 후보의 disclosure다.
- 후보 카드의 기본 정보는 시각, 상태, 핵심 이유, 영향 인원, 다음 행동이다.
- 슬라이딩 윈도우로 계산한 전체 후보를 카드로 나열하지 않는다. 기본 목록은 날짜 다양성과 중첩 구간 축약을 적용한 대표 후보 최대 6개로 제한한다.
- 전체 후보는 접힌 시간 지도에서 날짜별로 탐색하며, 추천 후보 수와 전체 가능 시간 수를 서로 다른 값으로 표시한다.
- 추천은 READY 후보 중 일정 조정, 기피 표시, 미응답, 참석 가능 인원 순으로 비교한다. 이 조건이 같고 시간 순서만 다른 후보는 모두 같은 추천 우선순위로 표시한다.
- READY 후보 상세은 성립 이유 다음에 일정 조정 약속과 기피 표시를 사람 이름으로 보여주고, 그 아래에 응답 수와 확정 행동을 둔다. 부담이 없을 때도 `일정 변경 없이 참석`, `피하고 싶은 표시 없음`을 명시한다.
- 단순 응답 수와 선호 조건은 결론보다 낮은 상세 정보로 둔다.
- `응답이 더 필요해요`는 모든 미응답자가 아니라 결정에 영향을 주는 사람을 가리킨다.
- 제품이 자동 확정하지 않는다.

### 7.5 Request Sent / Confirmation

용도: 주최자가 조직 계정에 전달된 최초 요청과 확정 알림을 확인한다.

규칙:

- 최초 요청 뒤 참석자별 이름, 역할, `요청됨`·`응답 완료` 상태를 보여준다.
- 링크 복사와 참여자 페르소나 미리보기를 주최자 제품 화면에 두지 않는다.
- 주요 행동은 `응답 현황 보기`다.
- 미응답자 재요청과 회의 확정은 계정 알림으로 표현한다.
- 실제 네트워크 발송은 fixture 상태 변화로 시연하되 제품 카피에 구현 설명을 노출하지 않는다.

### 7.6 생성 확인

- 확인 단계는 파생 후보를 검토하는 화면이 아니라 참석자에게 물어볼 회의 맥락과 시간 범위를 검토하는 화면이다.
- 참석자 응답 전에 후보 시작 시각이나 후보 개수를 나열하지 않는다.
- 회의 안내, 참석자와 진행 기준, 시간을 섹션으로 나누고 각 정보는 왼쪽 라벨과 오른쪽 값의 가로형 행으로 정렬한다.
- 참석자 이름은 장식적인 태그로 반복하지 않고 참석자 행의 텍스트로 보여준다.
- 날짜 범위 자체를 시간 섹션의 핵심 값으로 보여주고, 기간의 날짜 개수는 강조하지 않는다.
- 시간도 별도 강조 카드로 만들지 않고 다른 정보와 같은 가로형 행으로 정렬한다.
- 페인트형 입력의 의미가 유지되도록 기간, 회의 길이, 기본 범위, 제외한 시간, 추가한 시간을 구분해 확인한다.
- 제외·추가 예외가 없으면 `없음`으로 표시하고 내부 `AvailabilityWindow` 목록은 노출하지 않는다.
- 응답 마감은 같은 시간 섹션의 마지막 행에 둔다.
- 기본 탐색 범위를 그대로 사용했다면 내부 `AvailabilityWindow` 개수를 사용자 정보로 노출하지 않는다.
- CTA는 선택한 계정에 요청을 할당하는 `응답 요청 보내기`다.

## 8. 생성 흐름 계약

상위 단계는 네 개로 고정한다.

```text
회의 안내 → 참석자 → 시간 → 확인
```

### 8.1 회의 안내

목표: 참석자가 응답 전에 이해할 최소 회의 맥락을 만든다.

필수:

- 회의 이름
- 이 회의에서 확인하거나 결정할 내용 한 문장
- 주최자 또는 팀

선택:

- 관련 자료 이름 또는 링크

결정:

- 업무 맥락을 추상적인 칩 분류로 먼저 묻지 않는다.
- 어떤 일과 관련된지는 회의 이름과 한 문장 목적에서 구체적으로 드러나게 한다.
- 관련 자료는 기본으로 접어두며, 응답 전 필수 과제로 보이지 않게 한다.
- 플레이스홀더와 도움말이 같은 내용을 반복하지 않는다.

### 8.2 참석자

내부 순서는 세 판단으로 고정한다.

1. 누구에게 시간을 물을지 정한다.
2. 이 회의는 모두 참석해야 하는지 정한다.
3. 일부가 빠져도 된다면 꼭 필요한 사람과 최소 참석 인원을 정한다.

규칙:

- 주최자의 참석 여부를 묻지 않는다. P0에서 주최자는 항상 참석한다.
- 초대할 사람 수는 선택한 사람의 합계다. 전체 참석 예정 인원은 주최자를 포함해 `초대할 사람 수 + 1`이다.
- 주최자를 검색 결과나 선택한 사람 목록에 다시 넣지 않는다.
- `나도 참석해요` 토글을 제공하지 않는다.
- 첫 판단과 두 번째 판단을 동시에 펼치지 않는다.
- 사람 선택 완료 후 첫 판단은 짧은 요약으로 접고 참석 조건을 연다.
- 내부 판단에 별도 단계 번호, 진행 노드, `진행 중` 캡션을 사용하지 않는다.
- 검색 전 보조 목록은 `최근 함께한 사람`으로 표시한다.
- 프로토타입에서 추천 알고리즘이 있는 것처럼 말하지 않는다.
- 선택 결과는 검색 결과와 분리하고, 인원 수·체크·선택 목록으로 명확히 보여준다.

참석 조건 질문:

```text
이 회의는 모두 참석해야 하나요?
```

- 모두 참석해야 해요
- 몇 명은 빠져도 진행할 수 있어요

여기서 `모두`는 주최자와 초대할 사람 전체를 뜻한다. 주최자는 항상 참석 가능하므로 별도 응답이나 재선택을 요구하지 않는다.

`몇 명은 빠져도 진행`을 선택한 경우에만 꼭 참석해야 하는 사람과 최소 참석 인원을 순서대로 정한다. 최소 참석 인원은 주최자와 꼭 필요한 사람 수보다 작을 수 없고, 전체 참석 예정 인원보다 클 수 없다. 이 기준이 있어야 제품이 후보를 인기순으로 보여주는 대신 회의 성립 여부로 판정할 수 있다.

### 8.3 시간

내부 순서는 세 판단으로 고정한다.

```text
회의 가능 기간과 확보할 시간
→ 제품이 제안한 탐색 범위 확인과 예외 편집
→ 응답 마감
```

#### 회의 가능 기간과 확보할 시간

- 회의 가능 기간은 회의를 실제로 실시할 수 있는 시작일과 종료일이다.
- 확보할 시간은 계산된 후보 전체에 참여하기 위해 일정에서 연속으로 비워둘 유한한 시간 블록이다. 실제 회의가 정확히 그 시각에 끝난다는 뜻이 아니다.
- P0의 시간 최소 단위는 30분이다.
- P0 확보 시간의 기본값은 1시간이며 주최자는 30분 단위로 변경할 수 있다.
- 모든 파생 후보는 같은 확보 시간을 사용한다.
- 이 값은 주최자의 개인 가능 시간을 입력하는 화면이 아니다.
- 두 값을 정하면 제품이 기본 탐색 범위를 생성한다.
- 완료하면 `7월 13일~17일 · 1시간 확보`처럼 한 줄로 접고 탐색 범위 확인을 연다.

#### 기본 탐색 범위와 조율 가능 시간대

- 제품은 회의 가능 기간 안에 기본 탐색 범위를 먼저 제안한다. P0 fixture는 평일 09:00~18:00을 사용한다.
- 주최자는 가능한 범위를 빈 그리드에서 모두 만들지 않고, 안 되는 시간을 제외하거나 필요한 시간을 추가한다.
- 그리드를 조작하지 않고 계속하면 현재 남은 범위를 자신이 전체 확보 시간을 비울 수 있는 조율 가능 시간대로 확인한 것으로 본다.
- PC와 모바일의 그리드 한 칸은 30분 단위다.
- 한 칸만 추가하거나 제외할 수 있다. 확보 시간보다 짧은 연속 범위는 입력 오류가 아니라 후보가 파생되지 않는 범위다.
- 한 날짜에 여러 범위를 만들 수 있고, 맞닿은 범위는 하나로 합칠 수 있다.
- PC는 주간 그리드에서 클릭·키보드로 한 칸을 칠하고, 드래그로 여러 칸을 연속해서 칠한다.
- 모바일은 날짜를 먼저 고른 뒤 한 칸씩 탭하거나 터치 드래그로 연속 칸을 칠한다.
- 드래그는 보조 입력으로만 허용하며 모든 동작에 클릭·탭 대안을 제공한다.
- 주최자 시간 입력에는 블록 선택, 호버 리사이즈 핸들, 블록별 이동·삭제 UI를 사용하지 않는다.
- 확보 시간은 칠하기 단위가 아니라, 칠해진 연속 범위 안에서 파생 후보를 계산하는 길이 제약이다.
- 현재 적용 범위와 예외를 즉시 요약하고 편집 취소와 기본값 복원을 제공한다.
- 남은 시간대가 0개면 다음 판단으로 갈 수 없다.
- 생성 단계에서는 정확한 후보 개수나 시작 시각을 요구하지 않는다.
- 현재 구현의 고정 길이 후보 블록, 중복 금지, `겹침` 표시는 이 계약에 포함하지 않는다.

#### 참석자 가용시간과 파생 후보

- 신규 참석자는 더미 캘린더에서 비어 있음과 일정 있음 상태를 불러온다. 실제 API와 백엔드는 구현하지 않는다.
- 비어 있는 칸은 `가능`, 일정이 있는 칸은 `어려움` 초안으로 채우고 참여자가 제출 전에 확인한다.
- 캘린더를 사용하지 않으면 `대체로 가능` 또는 `가능한 시간이 적음`을 직접 입력 시작점으로 선택한다.
- 참석자는 상단의 `가능`, `조정하면 가능`, `어려움` 브러시를 고르고 예외 시간만 수정한다.
- PC는 주간 타임테이블, 모바일은 날짜 스트립과 하루 타임테이블에서 상태를 칠한다.
- 날짜 전체 적용으로 현재 브러시를 해당 날짜의 모든 열린 칸에 적용할 수 있다.
- 제품은 조율 시간대 안에서 30분 간격으로 시작 시각을 이동하며 확보 시간 전체가 들어오는 후보를 계산한다.
- 후보 칸 중 하나라도 명시적 `어려움`이 있으면 해당 참석자의 후보 상태는 `어려움`이다.
- 명시적 `어려움`은 없지만 하나라도 미응답이면 해당 참석자는 그 후보에 미응답이다.
- 후보의 모든 칸이 가능하면 참석자 상태는 `가능`이다.
- 모든 칸이 가능 또는 조정 가능이고 하나 이상이 조정 가능이면 `조정하면 가능`이다.
- 파생 후보는 서로 겹칠 수 있으며 입력 오류로 취급하지 않는다.
- 후보별 파생 상태를 기존 필수 참석자, 최소 참석 인원, 조정 필요, 미응답 판정에 사용한다.

#### 응답 마감

- 응답 마감은 참여자가 답하기를 기대하는 운영 시점이다.
- 회의 가능 기간 또는 회의 실시 기한과 같은 값으로 취급하지 않는다.
- 첫 조율 가능 시간대보다 앞선 시점만 선택할 수 있다.
- 마감 이후에도 응답과 수정은 허용하며 후보 상태는 동일한 규칙으로 다시 계산한다.
- 시스템이 임의의 마감을 확정하지 않는다. 권장값은 제안할 수 있지만 주최자가 확인해야 한다.

시간 단계는 조작 면적이 필요하므로 PC에서 다른 생성 단계보다 넓어질 수 있다. 하위 판단은 동시에 펼치지 않고 완료된 결과만 짧게 접어 유지한다.

### 8.4 확인

- 참석자에게 보일 정보 순서로 최종 요청을 보여준다.
- 회의 맥락, 주최자를 포함한 전체 참석 예정 인원, 참석 조건, 회의 가능 기간, 확보 시간, 조율 가능 시간대, 응답 마감을 확인한다.
- 각 영역은 수정 진입점을 제공할 수 있다.
- 내부 프리셋명이나 판정 로직을 보여주지 않는다.
- 우선 CTA는 응답 링크 만들기다.

## 10. 반응형 계약

PC와 모바일은 같은 상태와 데이터를 사용하지만 같은 화면을 축소하지 않는다.

### PC

- 앱 셸 최대 폭은 1440px다.
- 집중형 입력 단계의 과업 폭은 720~760px를 기준으로 한다.
- 확인 단계는 약 820px까지 확장할 수 있다.
- 시간 그리드는 960~1040px까지 확장할 수 있다.
- 한 상태 안에서 `1040 → 820 → 680`처럼 여러 가시 컨테이너를 중첩하지 않는다.
- 현재 제목, 과업 표면, 입력, CTA는 같은 정렬축을 공유한다.
- 생성 CTA는 본문 아래 왼쪽에 둔다.
- 빈 공간을 반복 요약이나 장식으로 채우지 않는다.

### 모바일

- 760px 이하에서 단일 열을 사용한다.
- 페이지 가로 여백은 16px을 기준으로 한다.
- 현재 상위 단계 하나만 보여준다.
- 제품 단계 CTA는 하단 고정 액션 바에서 제공한다.
- CTA는 현재 하위 과업에 따라 바뀐다.
  - 사람 선택 중: `N명 선택 완료`
  - 참석 조건 선택 후: `회의 시간 정하기`
  - 시간 조건 입력 후: `가능한 시간대 고르기`
  - 시간대 선택 중: `시간대 선택 완료`
  - 응답 마감 입력 후: `요청 내용 확인하기`
- 하단 액션 바는 safe area를 포함하며 본문을 가리지 않는다.
- 사람 검색은 전체 화면 레이어 또는 바텀시트로 제공한다.
- 사람 검색 레이어가 열리면 배경의 하단 CTA를 숨긴다.
- 시간 그리드를 축소하거나 가로 스크롤시키지 않고 날짜 스트립과 날짜별 슬롯으로 전환한다.
- 날짜 직접 선택은 접근 가능한 캘린더 바텀시트로 제공한다.

### 공통 접근성

- 터치 대상은 최소 44px이다.
- 선택 상태는 색상만으로 표현하지 않는다.
- 관련 입력은 `fieldset`과 `legend` 또는 동등한 의미 구조로 묶는다.
- 단계 이동 후 새 제목에 포커스를 이동한다.
- `prefers-reduced-motion`에서는 자동 스크롤과 전환 애니메이션을 줄인다.

컴포넌트를 추가하는 조건:

- 두 화면 이상에서 같은 역할과 상태를 공유한다.
- 접근성 동작을 한곳에서 보장할 필요가 있다.
- 의미 있는 중복을 제거한다.

단순히 CSS가 비슷하다는 이유로 추상화하지 않는다.

`ActionToast` 규칙:

- 화면 전환이 없는 가벼운 행동은 전역 토스트로 즉시 완료 여부를 알린다.
- 같은 행동의 토스트는 고정 ID로 교체해 반복 실행 시 쌓이지 않게 한다.
- 토스트는 레이아웃을 밀지 않으며, 실행한 행에는 해당 세션 동안 완료 상태를 함께 남긴다.
- 토스트만으로 중요한 상태 변경이나 실패 복구 방법을 전달하지 않는다.

프로토타입 시연 경로:

- 평가자는 `생성 → 요청 발송 → 응답 대기 → PENDING 결과 → 재요청 → 응답 변경 → READY → 확정`을 DEV 메뉴 없이 이어서 수행할 수 있어야 한다.
- 백엔드 없이 비동기 응답을 재현하는 전환은 `일부 응답이 도착했어요`라는 사용자 사건으로 표현하고, 개발 도구와 구분한다.
- 시연 전환은 생성한 회의의 제목과 목적을 유지한 채 검증된 판정 fixture를 결합한다.
- DEV 화면 바로가기는 임의 화면 접근과 QA를 위한 보조 수단이며 핵심 시연 경로를 대신하지 않는다.

## 12. 대표 품질 기준 화면

전체 화면을 동시에 고도화하지 않는다. 다음 세 상태를 제품의 품질 기준으로 먼저 완성한다.

### A. 주최자 생성 / 참석자 조건

경로: `#/create`의 참석자 단계  
검증 범위: 과업 표면, 사람 검색, 선택 상태, 점진적 공개, PC·모바일 CTA

완료 기준:

- 누구를 선택했는지 즉시 알 수 있다.
- 사람 선택과 참석 조건이 동시에 경쟁하지 않는다.
- 모바일에서 하단 CTA와 바텀시트 CTA가 겹치지 않는다.
- PC와 모바일의 관계가 같은 상태 모델로 설명된다.

### B. 참석자 응답

경로: `#/invite`  
검증 범위: 최소 회의 맥락, 시간대 스캔, 3상태 가용시간 입력, 저장과 수정 가능 안내

완료 기준:

- 5초 안에 어떤 회의인지 이해할 수 있다.
- 다른 참석자의 상태 없이 자신의 응답만 빠르게 남길 수 있다.
- `일정 조정하면 가능해요`의 의미가 오해되지 않는다.
- 모바일에서 한 손으로 주요 응답과 저장을 수행할 수 있다.

### C. 주최자 결정

경로: `#/host`의 결정 가능 상태  
검증 범위: 결론 우선 위계, 후보 카드, 판단 근거, 확정 CTA, 회복 행동

완료 기준:

- 첫 시야에서 지금 확정할 수 있는지 알 수 있다.
- 원본 표를 계산하지 않고 다음 행동을 결정할 수 있다.
- 가장 많은 사람이 가능한 후보와 가장 안전한 후보가 다를 때 후자를 선택할 수 있다.
- 기다릴 필요가 없는 미응답과 반드시 필요한 응답을 구별할 수 있다.
- 제출된 응답이 들어오면 READY 유무와 무관하게 통합 결과 보드로 이동한다.

이 세 화면의 컴포넌트와 시각 규칙을 확정한 뒤 Share, Waiting, Message, Edit, Done에 확장한다.

## 13. 실행 순서

### Gate 1. 구조 정합성

1. 현재 라우트를 상태 모델에 연결한다.
2. 각 화면이 정확히 하나의 템플릿을 사용하게 한다.
3. 역할별 셸을 분리한다.
4. 모든 핵심 CTA와 상태 전이를 연결한다.

통과 조건: Entry부터 생성, 응답, 판단, 응답 수정, 확정 공유까지 링크가 끊기지 않는다.

### Gate 2. 대표 화면 완성

1. 생성 / 참석자 조건
2. 참석자 응답
3. 주최자 결정

각 화면은 구조, 카피, 상태, 반응형, 접근성을 함께 완료한다. 세 화면을 번갈아 조금씩 고치지 않는다.

### Gate 3. 시스템화

1. 대표 화면에서 반복된 컴포넌트를 공통화한다.
2. `DESIGN.md` 토큰과 상태 표현을 적용한다.
3. 나머지 화면에 같은 문법을 확장한다.

### Gate 4. 전체 플로우 QA

1. 새 사용자 생성
2. 참석자 첫 응답
3. 참석자 응답 수정
4. 주최자 대기와 판단
5. PENDING 요청 대상 선택과 재판정
6. 모든 후보가 어려울 때 회의 기준 검토
7. 확정 메시지 복사

## 14. Definition of Done

화면은 다음을 모두 만족해야 완료다.

### 제품

- 사용자의 현재 질문이 하나로 설명된다.
- 첫 시야에 하나의 우선 CTA가 있다.
- 행동 후 다음 상태가 실제로 연결된다.
- 실패·대기 상태에도 수동 다음 행동이 있다.

### 정보 구조

- 역할에 필요하지 않은 정보가 노출되지 않는다.
- 내부 용어와 기능 목록이 내비게이션이나 카피에 나타나지 않는다.
- 동일 정보를 좌우 또는 상하에서 반복하지 않는다.

### 시각

- 제목, 본문, 입력, CTA의 정렬축이 설명 가능하다.
- 카드 안에 페이지 섹션용 카드를 중첩하지 않는다.
- 간격, radius, 타입 위계가 `DESIGN.md`와 일치한다.
- 선택, 대기, 오류, 완료 상태가 구분된다.

### 반응형

- 모바일은 PC를 단순 축소하지 않는다.
- 텍스트와 버튼이 잘리거나 겹치지 않는다.
- 모바일 고정 CTA가 본문·키보드·바텀시트와 충돌하지 않는다.
- 시간 선택이 마우스와 터치에서 모두 가능하다.

### 기술

- 핵심 링크와 버튼이 동작한다.
- 키보드 조작과 포커스 순서가 자연스럽다.
- `npm run lint`와 `npm run build`가 통과한다.

## 15. 보류 항목

아래 항목은 현재 구조 확정을 막지 않는다.

- 조직도 또는 실제 사내 디렉터리 연동
- 계정 홈, 내 회의, 받은 요청, 알림과 여러 회의 재진입
- 비참석 일정 조율 담당자와 대리 일정 생성
- 실제 계정 인증과 외부 캘린더 API 연동. 더미 데이터 기반 캘린더 반영 경험은 P0에 포함한다.
- 자동 알림과 자동 메시지 발송
- AI 기반 후보 추천
- 반복 회의
- 회의실 예약
- AI 기반 참석 가능성 예측
- 세밀한 분석 대시보드

보류 항목을 프로토타입 화면의 설명용 UI로 미리 만들지 않는다.
