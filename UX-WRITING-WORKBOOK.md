# MeetCue UX 라이팅 작업 문서

상태: **외부 교정 반영 완료**  
작성 기준: 2026-07-12 코드베이스  
적용 범위: `src/App.tsx`, `src/components`, 사용자에게 노출되는 상태·오류·알림 문구

## 적용된 결정

2026-07-12 외부 UX 라이팅 검토 결과와 실제 제품 동작을 대조해 다음 기준을 확정했어요.

| 항목 | 결정 |
| --- | --- |
| 제품명 | `MeetCue` |
| 시간 표기 | `오후 3:00` |
| 기본 말투 | 해요체 |
| 응답하지 않은 참석자 | 본문 `아직 응답하지 않은 사람`, 좁은 상태 라벨 `응답 전` |
| 선택하지 않은 시간 | 본문 `선택하지 않음`, 좁은 상태 라벨 `선택 전` |
| 필수 역할 | 설명 `꼭 참석해야 하는 사람`, 좁은 배지 `필수 참석자` |
| 회의 성립 조건 | `참석 기준` |
| 참석자 진입 | 계정에 도착한 요청에서 확인 |
| 응답 요청 | 제품 내부에서 요청을 보냄 |
| 확정 안내 | 제품 내부에서 알림을 보냄 |
| 응답 마감 | 마감 뒤에도 응답·수정 가능, 새 응답을 결과에 반영 |
| 미선택 시간 | 저장할 때 `참석하기 어려워요`로 처리 |
| 최소 참석 인원 | 주최자 포함 |
| 캘린더 | 일정 생성은 지원하지 않음 |

외부 문서에서 수동 링크·문구 공유를 전제로 한 변형은 사용하지 않았어요. `S-002`, `W-001`, `W-002`, `P-042`에는 계정 기반 변형을 적용했고, `F-002`, `Q-006`, `M-002`, `M-003`은 실제 내부 요청·알림 동작에 맞춰 수정했어요.

`PD-005`에 해당하는 `조정하면 가능해요` 응답은 현재처럼 추가 확인 없이 참석 가능한 응답으로 계산해요. 확정 화면에서는 필요한 경우 기존 일정을 조정해 달라고 안내해요.

## 이 문서의 목적

이 문서는 MeetCue에서 사용하는 명칭과 화면 문구를 한곳에서 검토하기 위한 작업지예요. 제품 코드를 직접 수정하지 않고도 용어, 말투, 제목, 버튼, 안내, 오류 문구를 먼저 다듬을 수 있도록 구성했어요.

외부에서 문구를 수정한 뒤 이 문서를 다시 저장소에 가져오면, `수정안` 열을 기준으로 실제 화면에 반영할 수 있어요. 의미나 기능까지 바꿔야 하는 제안은 문구 수정과 구분해 `제품 논의`에 남겨주세요.

## 작업 방법

1. 먼저 `용어 사전`에서 제품 전체에 사용할 명칭을 정해요.
2. `공통 문장 규칙`에서 말투와 표기 방식을 확정해요.
3. 화면별 표의 `수정안`을 작성해요.
4. 의미가 모호하면 추측해서 고치지 않고 `제품 논의`에 질문을 남겨요.
5. 삭제할 문구는 수정안에 `[삭제]`라고 적어요.
6. 원문을 유지하려면 수정안에 `[유지]`라고 적어요.

문구 상태는 다음 셋 중 하나로 표시해요.

- `검토 필요`: 아직 결정하지 않은 문구
- `확정`: 그대로 코드에 적용할 문구
- `제품 논의`: 기능이나 정책 확인이 먼저 필요한 문구

## 제품 이해

MeetCue는 여러 사람의 가능한 시간을 모아 회의 시간을 정하도록 돕는 제품이에요.

주요 사용자는 두 역할로 나뉘어요.

- 주최자: 회의를 만들고, 참석자를 정하고, 가능한 시간을 요청하고, 결과를 확인해 회의 시간을 확정해요.
- 참석자: 요청받은 회의를 확인하고, 가능한 시간을 표시하고, 응답을 저장하거나 수정해요.

제품이 직접 하는 일과 하지 않는 일을 구분해야 해요.

- 가능한 시간을 수집하고 회의가 성립하는 시간을 계산해요.
- 바로 정할 수 있는 시간과 응답이 더 필요한 시간을 구분해요.
- 주최자가 선택하기 전에는 회의 시간을 자동으로 확정하지 않아요.
- 복사한 문구나 링크를 외부 메신저로 자동 전송하지 않아요.
- 참석자의 캘린더 일정 제목은 공유하지 않아요.

## 보이스 원칙

MeetCue는 사용자를 대신해 판단하지 않아요. 현재 상태와 다음 행동을 차분하고 구체적으로 알려줘요.

- 기본 말투는 `-해요`체를 사용해요.
- 제목은 현재 결론이나 사용자가 답해야 할 질문으로 써요.
- 설명은 제목을 반복하지 않고 이유, 영향, 다음 변화를 알려줘요.
- 버튼은 누르면 생기는 결과를 동사로 써요.
- 시스템의 계산 용어보다 사용자가 이해하는 말을 먼저 써요.
- 완료, 저장, 복사, 전송, 확정은 실제로 일어난 범위까지만 말해요.
- 사용자를 탓하거나 재촉하지 않아요.
- 한 화면에서 `회의`, `요청`, `응답`, `시간`의 대상을 생략해 모호해지지 않게 해요.

## 표기 규칙

| 항목 | 기준 | 예시 |
| --- | --- | --- |
| 말투 | 해요체 | 확인해 주세요, 저장했어요 |
| 사용자 호칭 | 이름 뒤에 `님` | 민수님 |
| 인원 | 숫자와 `명`을 붙임 | 4명 |
| 시간 길이 | 숫자와 단위를 붙임 | 30분, 1시간 |
| 날짜 | 화면 맥락에 따라 연도 생략 | 7월 16일 목요일 |
| 시각 | 오전·오후 또는 24시간제 중 하나로 통일 | 오후 3:00 |
| 버튼 | 결과가 드러나는 동사형 | 응답 저장하기 |
| 토스트 | 처리한 대상 + 결과 | 응답 링크를 복사했어요 |
| 질문 | 한 문장에 한 가지 판단만 요청 | 모두 참석해야 하나요? |
| 괄호 | 부가 정보에만 제한적으로 사용 | 선택 사항 |

시각 표기 방식은 제품 전체에서 하나만 선택해 주세요.

- 제안 A: `오후 3:00` — 일반 사용자에게 익숙함
- 제안 B: `15:00` — 표와 시간 그리드에서 짧고 정확함
- 최종 선택: `[작성 필요]`

| E-010 | 날짜 역전 또는 범위 오류 | 올바른 입력 예시 |  | 확인 필요 |

접근성 이름도 사용자 문구의 일부로 검토해요.

- 닫기 버튼은 닫히는 대상을 포함해요: `응답 요청 창 닫기`
- 증가·감소 버튼은 변경하는 값을 포함해요: `최소 참석 인원 늘리기`
- 상태를 색상으로만 설명하지 않아요.
- 시간 셀은 날짜, 시각, 현재 상태, 선택 가능 여부를 함께 읽을 수 있어야 해요.

## 개발 전용 문구

다음 문구는 개발 화면에서만 사용하고 실제 사용자 화면에는 노출하지 않아요.

- 핵심 시연
- 생성
- PENDING 결과
- 재판정 결과
- fixture
- 참여자 fixture
- 후속 제품 · P0 보류
- READY, PENDING, IMPOSSIBLE
- HOST_DRAFT 같은 내부 상태명
- 대상군, 점수, 판정 근거 같은 계산 모델 명칭

개발 화면에서도 `참여자 fixture`처럼 제품 용어와 다른 명칭을 쓰면 검토 과정에서 혼란이 생길 수 있어요. 개발 전용임을 표시하되 사용자 명칭은 가능한 한 동일하게 유지해요.

## 제품 논의가 필요한 질문

문구를 확정하기 전에 아래 항목의 실제 동작을 확인해야 해요.

1. 참석자는 공용 링크에서 이름을 선택하나요, 개인 계정에서 요청을 확인하나요?
2. 주최자가 회의를 만들면 제품이 참석자에게 직접 요청을 보내나요, 공유할 문구나 링크만 만들나요?
3. 응답 마감 뒤에도 참석자가 응답하거나 수정할 수 있나요?
4. 선택하지 않은 시간은 자동으로 `참석하기 어려움`으로 저장되나요?
5. `조정하면 가능`으로 표시한 시간이 확정되면 참석자에게 별도의 확인이 필요한가요?
6. 최소 참석 인원에 주최자가 포함되나요?
7. `모두 참석`은 주최자를 포함한 전체 인원을 뜻하나요?
8. 확정 뒤 제품이 캘린더 일정을 만들거나 알림을 보내나요?
9. `응답 요청` 버튼은 실제 요청 전송인가요, 요청 문구 복사인가요?
10. `응답 마감`은 입력을 막는 시점인가요, 주최자에게 판단을 돕는 기준 시점인가요?

## 외부 작업 결과 작성 형식

전체 표를 직접 수정해도 되고, 아래 형식으로 확정 문구만 모아도 돼요. ID는 삭제하거나 바꾸지 마세요.

```yaml
copy_updates:
  - id: S-004
    status: 확정
    copy:
      title: 지금 정할 수 있는 시간을 확인해 보세요
      description: 바로 정할 수 있는 시간과 아직 응답이 필요한 시간을 나눠서 보여드려요.
    rationale: 판단, 가능 응답 조건 같은 시스템 용어를 사용자 언어로 바꿈

  - id: D-007
    status: 제품 논의
    question: 이 그룹에서 한 사람의 가능 응답만 오면 되는지 확인이 필요해요.
```

새 문구를 제안할 때는 다음 기준을 지켜주세요.

- 변수는 `{{participantName}}`, `{{count}}`, `{{dateTime}}`처럼 표시해요.
- 한 ID에 서로 다른 상황이 섞이면 ID 뒤에 `-A`, `-B`를 붙여 나눠요.
- 원문의 의미를 바꾸는 수정은 `rationale`에 이유를 적어요.
- 화면 구조 변경이 필요한 제안은 `layout_note`에 적어요.

## 코드 적용 규칙

이 문서가 돌아오면 다음 순서로 반영해요.

1. `확정` 상태의 ID만 구현 대상으로 삼아요.
2. `제품 논의` 항목은 동작을 확인하기 전까지 기존 문구를 유지해요.
3. 같은 개념의 문구는 공통 상수나 포맷 함수로 묶어요.
4. 사용자용 카피와 개발용 fixture 라벨을 분리해요.
5. 변수형 문구는 0명, 1명, 여러 명과 긴 이름으로 확인해요.
6. 모바일에서 제목, 버튼, 상태 라벨이 의도치 않게 잘리지 않는지 확인해요.
7. 저장, 복사, 확정, 오류 상태를 실제 동작으로 검증해요.
8. 적용 후 이 문서의 ID와 코드 위치를 매핑해 변경 이력을 남겨요.
